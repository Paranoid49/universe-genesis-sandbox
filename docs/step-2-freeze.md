# 步骤 2 冻结验收口径：持续演化的宇宙

状态：已冻结，实现进行中  
冻结日期：2026-07-14  
唯一上位产品事实：`docs/product-definition.md`  
实施顺序：`docs/milestones.md`  
旧模块去留：`docs/legacy-module-migration.md`

本文件是步骤 2 实现、测试、文档和独立评审的唯一验收基线。冻结后的实现与评审不得增加需求、提高完成标准或评价步骤 3 及后续能力。发现确实影响范围的歧义时，必须先由主执行 Agent 依据当前事实来源裁定；会显著改变产品范围时才请求用户裁决。

## 1. 工作单元目标

把步骤 1 的“可追因静态宇宙”迁移为可以暂停、推进、保存、恢复并继续确定性演化的运行中宇宙。

步骤 2 的产品闭环是：玩家从初始状态启动宇宙，推进时间，观察至少一个世界对象形成、变化或消亡，查看由真实状态差异产生的事件及因果链，保存运行状态，重新打开后从同一位置继续演化。

步骤 2 不负责规定玩家探索目标，也不要求生命、文明或自主实体出现。

## 2. 当前必须实现的要求

### R1：唯一运行事实来源

- 建立版本化 `UniverseState`，作为新运行时唯一事实来源。
- 规则判断、状态转换、因果记录、事件、指标、解释和新主界面只能消费 `UniverseState` 或其只读派生投影。
- `UniverseSummary` 只允许留在隔离的步骤 1／旧存档兼容读取路径，不得继续作为新运行时事实来源。
- 任何新运行事实不得只存在于事件、指标、解释、React 状态或存档元数据中。

### R2：确定性模拟时钟与转换协议

- 建立确定性模拟时钟，至少包含当前逻辑时间、步序号、运行状态和时间模型版本。
- 建立类型化状态转换协议，明确转换前状态、适用规则、有序显式输入、随机流状态、状态差异、因果节点和转换后状态。
- 支持暂停、单步推进、连续运行和速度调整。
- 连续运行只是重复提交确定性时间步，不得依赖浏览器帧率、墙上时间、计时器漂移或隐藏标签页调度决定模拟结果。
- 暂停后不得继续提交新的状态转换。

### R3：可继续的确定性随机流

- 保存 PRNG 算法版本、命名随机流身份、完整内部状态或等价可恢复状态、流游标、抽样序号和输入顺序。
- 恢复后的下一次随机决定必须与未中断运行完全一致。
- 随机流恢复必须继续满足步骤 1 的随机转录、结果绑定和因果可追溯要求。
- 不得通过重新使用 Seed 从头生成到目标时间来冒充状态恢复。

### R4：由状态变化产生历史

- 新运行时不得预生成完整未来时间线。
- 每个运行时事件必须由已经提交的真实状态差异生成，并进入因果网络。
- 事件是状态变化的只读可读投影，不得反向成为世界状态的事实来源。
- 尚未发生的事件不得提前作为确定事实显示、保存或参与后续规则判断。
- 已发生历史可以从事件投影中删除并由状态差异日志重新生成，不得因此丢失世界事实。

### R5：删除反向生成链

- 新运行时不得调用或复制 `timeline → timelineImpact → 空间／生命／文明` 的反向生成链。
- `timelineImpact` 不得进入 `UniverseState`、状态转换、运行存档或新主界面。
- 旧 `timelineImpact` 只能存在于明确隔离的旧存档只读解释器中。
- 架构门禁必须能够证明新运行时没有依赖该旧链路。

### R6：持久世界对象

- 当前传统空间兼容宇宙中的世界对象必须拥有稳定身份和跨时间步持久状态。
- 至少提供一个可自动验证的完整过程，使传统空间对象发生形成、字段变化或消亡，并产生后续影响。
- 对象变化必须由状态转换产生，不能通过替换整份一次性样本树冒充演化。
- 步骤 2 不要求增加新对象数量，也不要求形成生命、文明或自主实体。
- 旧生物圈和文明种子不得继续作为已经确定的未来路径进入新运行时；完整生命与自主实体过程由步骤 6 接管。

### R7：派生指标、事件与解释

- 固定全局指标降级为 `UniverseState` 或观察数据的派生投影，不得反向驱动世界事实。
- 随机年龄评分必须由模拟时钟和状态替换。
- 指标、事件和解释必须可以从状态、差异日志和因果数据重新计算。
- 派生投影删除后重新生成必须得到相同结果和值级因果入口。

### R8：运行状态存档

- 建立版本化运行状态存档信封，至少保存宇宙定义身份、状态版本、模拟位置、状态快照、有序输入日志、因果完整性信息、随机流状态和完整性校验信息。
- 保存操作必须形成一致检查点，不得把跨两个时间步的部分状态写入同一存档。
- 保存后重新打开必须恢复同一运行状态，并允许继续演化。
- 损坏、未知版本、身份错配、状态哈希错配、历史缺失或随机流不完整必须明确拒绝，不得静默回退成新宇宙。
- A1 只作为旧基线只读导入格式，不能冒充步骤 2 运行存档；转换为新运行宇宙时必须创建新身份并声明历史不等价。

### R9：浏览器存储边界

- 浏览器运行状态使用版本化 IndexedDB 存储适配器或具备同等容量、事务和版本迁移能力的浏览器存储适配器。
- `localStorage` 只允许保存轻量偏好和迁移标记，不得保存完整状态快照、输入日志、因果网络或随机流状态。
- 存储写入必须具备原子提交语义；写入失败时不得先提交页面成功状态。
- 测试必须能够注入内存适配器，不依赖真实用户数据。

### R10：运行控制与历史浏览分离

- 新主界面必须提供当前运行位置、暂停、单步推进、连续运行、速度调整、保存和继续入口。
- 历史浏览游标与模拟时钟必须是两个独立状态：浏览过去只改变视图，不回写当前宇宙状态。
- 从历史位置恢复查看不得修改当前运行分支；修改过去和创建分支属于步骤 4，本步骤不提供。
- `OverviewPage` 在步骤 2 只负责接入当前运行状态、最近变化和运行控制，不提前实现步骤 3 的认知边界产品形状。
- `TimelinePage` 和观察台只能显示已经发生的历史；旧未来滑块和 `timelineImpact` 展示退出新主流程。

### R11：迁移、兼容与主流程隔离

- 步骤 1 的因果查询、确定性不变量、旧 UGS070／I1 分享读取和 A1 只读兼容能力不得静默损坏。
- 旧静态宇宙、旧分享和 A1 可以在明确标识的只读兼容查看器中打开，但不得进入新运行事实来源。
- 新主导航和默认启动流程必须进入运行中宇宙，不得继续把一次性报告作为产品主流程。
- 被替换的旧生成、时间线、`timelineImpact` 和完整 `UniverseSummary` 事实路径不得通过隐藏入口继续被新页面调用。

### R12：分层精度与资源控制

- 长时间跨度使用明确的分层精度、批处理或等价计算预算，不做无上限微观全量模拟。
- 连续运行必须可以停止，并在组件卸载、页面切换、测试结束或错误恢复时释放计时器和待处理任务。
- 同一运行实例不得并发提交两个时间步；重复点击、定时器重入和恢复期间推进必须被串行化或明确拒绝。
- 任何性能降级策略不得改变确定性状态序列。

## 3. 明确不实现的内容

- 不实现步骤 3 的自由探索信息架构、认知边界、玩家推测、研究记录簿或动态观察通道。
- 不实现步骤 4 的分支身份、共同祖先比较、回到过去修改、界外实验和新分享包。
- 不实现步骤 5 的可组合多本体规则包、动态空间拓扑、动态指标集合或动态干预能力。
- 不实现步骤 6 的自主实体感知、记忆、信念、意图和行动，也不要求生命或文明出现。
- 不实现步骤 7 的长期创作档案、结构化编年史导出或正式三类文件包。
- 不增加新的固定事件池、纪元配额、对象配额、文明种子、奇迹类型或生成文案规模。
- 不把真实墙上时间、后台运行时长或浏览器随机数写入宇宙事实。
- 不允许覆盖式回滚、清除历史或用重新生成冒充恢复。

## 4. 可验证完成条件

### C1：运行闭环

玩家可以启动新宇宙，暂停，单步推进，连续运行，调整速度，观察至少一个对象状态变化，保存，重新打开并继续推进。

### C2：确定性状态序列

相同宇宙定义、初始状态、规则版本、随机流状态和有序输入产生逐步完全一致的状态哈希、差异日志、事件、因果节点 ID 和随机转录。

### C3：恢复后的下一步一致

在同一检查点分别执行不中断推进和“保存—关闭—恢复—推进”，两条路径的下一状态、事件、随机决定、因果节点 ID 和后续可继续结果完全一致。

### C4：事件来自状态差异

每个新运行时事件都能定位到对应状态差异、适用规则、前置状态、随机决定和根因；删除事件投影后能够从差异日志重新生成相同事件。

### C5：无预制未来与反向生成

自动化门禁证明新运行时不预生成未来事件，不读取 `timelineImpact`，也不存在通过事件摘要创造世界对象的调用路径。

### C6：唯一事实来源

删除指标、事件、解释和页面派生缓存后可以由 `UniverseState`、差异日志和因果数据重建；删除任一派生投影不会丢失世界事实。

### C7：持久对象过程

至少一个传统空间对象在多个时间步中保持稳定身份，并完成形成、变化或消亡及一个可观察后续影响；全过程可追因并可保存恢复。

### C8：存档拒绝与原子性

有效运行存档能够恢复；损坏版本、未知版本、身份错配、状态哈希错配、缺失输入、缺失随机流和部分写入均被明确拒绝；失败后原存档和当前内存状态保持不变。

### C9：浏览器存储隔离

架构测试证明完整运行数据只经版本化状态存储适配器访问，`localStorage` 没有完整状态、历史、因果或随机流写入路径。

### C10：时钟与历史游标分离

浏览已发生历史不会改变当前状态哈希、模拟位置、输入日志或随机流；暂停后等待真实时间不会产生新状态；恢复当前视图后仍从原运行位置继续。

### C11：兼容与隔离

步骤 1 因果功能和明确保留的旧只读兼容路径通过回归测试；旧静态生成器、旧时间线、旧 `timelineImpact` 和 A1 不再被新主流程当作运行事实来源。

### C12：纵向浏览器验收

真实浏览器至少覆盖一次“启动—推进—观察变化—追因—保存—刷新—恢复—继续推进”的完整路径，并覆盖键盘、窄屏、无障碍和错误提示。

### C13：迁移矩阵关闭

`docs/legacy-module-migration.md` 中归属于步骤 2 的生成、时间线、指标、空间、存档、页面和测试条目均具有实现、迁移、测试、文档和兼容证据。

### C14：评审与问题处置

完成实现与完整发布门禁后，由未参与步骤 2 实现的全新独立 Agent 依据本冻结基线执行首次完整评审。有效 P0、P1 必须关闭，P2 必须修复、以证据驳回、降级或登记技术债并明确处置，建议项不阻塞完成。

## 5. 接口、数据与兼容边界

### 5.1 新公共契约

- `UniverseDefinition`：宇宙定义身份、Seed、规则和初始输入。
- `UniverseState`：版本、逻辑时间、步序号、持久对象和规则运行所需事实。
- `SimulationClock`：逻辑时间、步长、运行状态、速度和时间模型版本。
- `RandomStreamState`：算法版本、流身份、内部状态、游标和抽样序号。
- `TransitionInput`：类型化、有序、可追因的外部输入。
- `StateTransition`：转换前状态身份、适用规则、输入、随机决定、差异、因果记录和转换后状态身份。
- `StateDiff`：可验证的新增、修改和删除操作，不保存无法核对的摘要文案。
- `RuntimeEvent`：从已提交 `StateDiff` 派生的只读历史投影。
- `RuntimeArchiveEnvelope`：运行状态、输入日志、随机流、因果完整性与版本信息。
- `RuntimeStorageAdapter`：事务写入、读取、列举、删除、迁移和失败恢复接口。

具体类型名可以在实现中调整，但上述职责不得合并回单一聚合对象。

### 5.2 版本边界

- 新运行状态、随机流、转换协议和存档信封必须拥有独立版本，不得借用 `ugs-ruleset@0.7.0` 或 `ugs-causality@13` 冒充全部兼容身份。
- 任何会改变状态序列、随机流恢复、差异语义、因果节点身份或存档解释的修改必须评估并更新相应版本。
- 旧 `UniverseSummary`、UGS070、I1 和 A1 只属于兼容身份，不得自动升级成等价运行历史。

### 5.3 步骤 1 不变量

- 所有状态变化和事件继续满足因果闭包、双向查询、合法根、规则来源和精确随机证据。
- 同果异因历史不得合并。
- 模拟核心继续禁止读取 React、浏览器状态、系统时间、网络和本地存储。
- 因果图、状态快照、随机流和已提交转换对外只读。

## 6. 质量要求

### 6.1 必须执行的门禁

- `npm run lint`。
- 应用与测试 TypeScript 类型检查。
- 步骤 1 永久不变量与兼容回归测试。
- 步骤 2 状态、时钟、转换、随机恢复、事件投影、存档和存储契约测试。
- UI 组件测试、真实浏览器 E2E、键盘和 axe。
- 生产构建、体积检查和独立非覆盖率性能测试。
- 最终完整入口仍为 `npm run check:release`，并新增步骤 2 专属门禁，不得绕过现有步骤 1 检查。

### 6.2 覆盖率

- 模拟核心全局语句、函数和行覆盖率不低于 90%，分支不低于 82%。
- UI 全局语句、函数和行覆盖率不低于 75%，分支不低于 70%。
- 状态转换、随机流恢复、存档校验和存储适配器必须设置独立文件阈值，不得只依赖全局平均值。
- 每个公开错误码和存档拒绝原因必须有直接负向测试。

### 6.3 性能与响应预算

在文档化的代表性状态规模、两次预热和至少十二个多模板样本下执行非覆盖率测试：

- 单步状态转换 P50 小于 50 毫秒，P90 小于 100 毫秒。
- 连续执行 100 个确定性时间步 P50 小于 1500 毫秒，P90 小于 2500 毫秒。
- 暂停命令在当前时间步提交完成后 100 毫秒内停止调度下一步。
- 运行状态快照序列化 P90 小于 500 毫秒，恢复并完成完整性校验 P90 小于 1000 毫秒。
- 覆盖率插桩环境可以采用单独记录的放宽预算，但不得替代上述非覆盖率门禁。
- 性能优化不得改变状态序列、随机决定、因果节点或存档内容。

### 6.4 可靠性与资源

- 状态转换要么完整提交，要么不产生任何可见变化。
- 存档写入要么完整提交，要么保留先前有效版本。
- 连续运行、恢复和存储任务发生异常时必须进入可恢复错误状态，不得继续静默推进。
- 测试结束必须关闭预览服务、连续运行计时器、工作线程和待处理任务，并删除 `coverage/`、`dist/`、`playwright-report/` 与 `test-results/`。
- 手动启动命令为 `npm run dev`，停止方式为同一终端按 `Ctrl + C`。

### 6.5 UI 运行时与体积

- 步骤 2 开始实现前复核 Preact React-compat 是否仍满足运行控制、IndexedDB 和错误边界需求。
- JavaScript gzip 硬上限保持 115 KiB、保护线保持 110 KiB；CSS gzip 硬上限保持 10 KiB、保护线保持 8 KiB。
- 若步骤 2 的必要能力突破保护线，必须通过正式架构决策说明原因、备选方案和新预算，不得直接放宽脚本。

## 7. 必要威胁模型

| 威胁 | 必须具备的防御与验证 |
| --- | --- |
| 墙上时间污染确定性 | 模拟结果只使用逻辑时钟；篡改系统时间、计时器频率和页面可见性不改变状态序列 |
| 随机流游标漂移 | 存档保存完整流状态；删除、交换或篡改任一流游标时恢复被拒绝或重放失败 |
| 部分状态提交 | 转换和存档使用原子提交；异常注入后前后状态只能是完整旧状态或完整新状态 |
| 重复与并发推进 | 双击、计时器重入、恢复期间推进和并发命令不会重复提交同一步 |
| 状态与事件分叉 | 事件只能从已提交差异生成；篡改事件不能改变状态，篡改差异会被完整性校验发现 |
| 派生投影冒充事实 | 指标、事件、解释和 UI 缓存不进入运行事实来源；删除后可以重建 |
| 旧链路回流 | 新运行时依赖扫描拒绝 `timelineImpact`、旧完整生成器和旧 A1 作为事实来源 |
| 历史浏览改写当前状态 | 历史游标为只读视图状态；浏览操作不改变状态哈希、输入日志和随机流 |
| 跨宇宙存档拼接 | 存档绑定宇宙定义身份、状态哈希、历史和随机流；任一身份错配明确拒绝 |
| 损坏或未知版本静默回退 | 未知版本只能明确拒绝或进入标识清楚的只读兼容路径，不能生成新宇宙冒充恢复 |
| 存储竞争覆盖新数据 | 使用事务和版本检查；过期异步写入不得覆盖较新的检查点 |
| 连续运行资源泄漏 | 暂停、卸载、错误和测试结束均取消调度；4173／5173 与项目 Node 进程无残留 |
| 长时间跨度失控 | 分层精度和批处理有明确预算；超预算时暂停并提示，不静默改变规则或跳过因果记录 |
