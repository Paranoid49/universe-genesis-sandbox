## 1. P1-01：拓扑模块没有形成运行时关系状态

- 结论：问题成立，按根因整改。
- 实现：`UniverseState` 增加宪法驱动的拓扑身份和实际关系记录；`runtime-topology.ts` 从拓扑模块、对象本体和初始关系声明建立确定性关系状态。
- 行为：物质预设形成层级拓扑，奥术预设形成关系网络，梦境预设形成非空间语义拓扑；观察页面能够查询实际关系，而不是只显示拓扑摘要。
- 验证：步骤 5 运行测试覆盖三种真实拓扑、非法关系引用、关系预算和保存恢复一致性。

## 2. P1-02：认知模块没有约束公开公理和隐藏字段

- 结论：问题成立，按根因整改。
- 实现：三个参考预设分别声明公开公理；宪法领域校验拒绝不存在的公开公理，并拒绝隐藏字段与观察可见字段冲突。
- 行为：观察访问端口和规律追踪观察都只公开宪法明确允许的规则和字段，不能通过动态观察绕过认知边界读取底层完整状态。
- 验证：宪法与运行测试覆盖不存在公理、隐藏字段冲突、合法公开公理、空公开公理集合和可信观察重算。

## 3. P1-03：边界预算未完整校验和执行

- 结论：问题成立，按根因整改。
- 实现：边界模块增加 `maximumRelations`，规则评估、效果、对象和关系预算均要求为正安全整数；初始状态和每次转换执行四类预算守卫。
- 原子性：规则执行器在消费随机流前进行保守预算预检，超限不会留下部分状态、随机游标、输入日志、执行记录或因果节点。
- 验证：宪法和运行测试覆盖非法预算、对象与关系超限、规则与效果超限以及失败后的状态与随机流不变。

## 4. P1-04：实验与干预受固定范围和首对象绑定

- 结论：问题成立，按根因整改。
- 实现：`createExperimentInput` 和 `createInterventionInput` 支持显式 `objectId`，干预同时支持显式 `capabilityId`；公共输入边界删除固定正负二十限制。
- 界面：实验页提供实验对象、干预对象和能力选择，实验范围由目标属性上下界计算，干预范围由宪法能力声明决定。
- 验证：模拟测试覆盖大于二十的合法动态干预和非首对象目标，UI 测试覆盖对象、能力、范围和提交载荷。

## 5. P1-05：多规则随机决定伪造额外规则原因

- 结论：问题成立，按根因整改。
- 实现：运行因果网络按照规则执行记录保存的随机决定身份建立边，只把随机节点连接到实际消费该决定的规则。
- 验证：运行测试构造同一转换包含多条规则但只有一条规则消费随机的场景，断言随机决定只有唯一合法规则归属。

## 6. P2-01：动态对象标签泄露内部类型 ID

- 结论：问题成立并已修复。
- 实现：对象标签使用本体模块声明的中文名称；未知类型显示“未知对象类型”，不回退为内部 ID。
- 验证：运行与 UI 测试覆盖三个预设的中文标签和未知类型回退。

## 10. 第一轮有限复查后的 P1-02 补充整改

第一轮有限复查确认 P1-01、P1-03、P1-04、P1-05、P2-01 和 P2-02 已关闭，但复现了 P1-02 的剩余路径：自定义认知模块把 `publicAxiomIds` 设为空时，公开公理列表为空，`rule-trace` 观察仍直接显示转换中的内部规则 ID。

根因是 `observation.ts` 的规律追踪直接输出 `StateTransition.ruleIds`，没有经过认知模块公开公理集合过滤。补充整改后，规律追踪只返回同时存在于实际转换与 `publicAxiomIds` 中的规则；没有可公开规则时返回“尚无已应用公开规则证据”，不暗示内部规则不存在，也不泄露其身份。

新增运行测试构造空公开公理认知模块，确认公开公理列表和规律追踪都不包含 `rule.material-condensation@1`，并保持知识状态为证据不足。步骤 3 的既有无规则观察断言同步采用更准确的公开规则文案。
