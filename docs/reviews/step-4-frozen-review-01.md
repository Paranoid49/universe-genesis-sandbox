## 一、总体结论喵

步骤 4 已经建立了正确的产品骨架，包括正式分支契约、界外实验与宇宙内干预的类型分离、确定性重放、分支存档、共同祖先比较、独立 IndexedDB 适配器、两类分享包、新主导航和研究记录分支隔离喵。

## 三、逐项问题喵

### P1-01：步骤 2 检查点恢复会在当前正式分支身份下原地替换历史喵

- 严重等级为 P1 喵。
- 涉及位置为 `src/ui/useRuntimeUniverseModel.ts:137` 至 `src/ui/useRuntimeUniverseModel.ts:151`、`src/components/RuntimeSession.tsx:37`、`src/ui/useBranchLaboratory.ts:30` 至 `src/ui/useBranchLaboratory.ts:31` 喵。
- 现实触发或复现方式为先在根分支推进并保存步骤 2 运行检查点，再创建实验子分支并继续推进，然后进入“存档”执行“恢复最近检查点”，此时 `restoreLatest` 直接把运行状态替换为旧状态，而分支协调层随后通过 `updateUniverseBranchState` 把该旧状态重新挂到当前 `branchId` 上喵。
- 实际影响是同一正式分支身份的已提交转换、输入日志、随机流和历史哈希可以被缩短或替换，后续保存或再次分叉会把被覆盖后的状态当作当前分支事实，研究记录仍使用原 `branchId`，从而产生历史身份与研究证据不一致喵。
- 判断依据是冻结 R3 明确禁止删除、覆盖或就地修改已提交历史，R10 要求步骤 2 检查点转换为正式分支记录时创建明确分支身份，C2 要求父分支逐字节不变，C9 要求任何回到过去的修改都创建新分支喵。
- 整改建议是禁止 `RuntimeSession` 直接把通用 `restoreLatest` 作为步骤 4 当前分支恢复入口，将检查点恢复交给分支协调层，校验检查点所属宇宙、分支和历史前缀后创建或切换到明确分支，并禁止 `updateUniverseBranchState` 接受不是当前历史合法后继的状态喵。

### P1-02：分支输入缺少冻结要求的追踪信封，规则拒绝也没有稳定结果喵

- 严重等级为 P1 喵。
- 涉及位置为 `src/sim/contracts/branching.ts:8` 至 `src/sim/contracts/branching.ts:16`、`src/sim/branch-inputs.ts:13` 至 `src/sim/branch-inputs.ts:43`、`src/sim/runtime-causality.ts:106` 至 `src/sim/runtime-causality.ts:139` 喵。
- 现实触发或复现方式为提交强度为 0、目标不存在或字段不受支持的宇宙内干预，当前实现只抛出普通 `Error`，输入日志、转换历史和运行因果网络均不产生拒绝记录，且错误对象没有稳定错误码和字段路径喵。
- 咪使用临时定向探针亲自验证了强度为 0 的 `InUniverseIntervention` 只抛出“非零整数”异常，原分支 `inputLog` 和 `transitions` 仍为空，该探针文件已经删除喵。
- 实际影响是玩家无法追溯一次正式干预为什么被拒绝，自动化调用方也只能依赖中文消息文本判断失败类型，接受的输入本身也没有记录发起分支、适用规则、前置状态和结果转换身份喵。
- 判断依据是冻结 R5 要求每个实验和干预记录输入 ID、类型、顺序、发起分支、目标、参数、适用规则、前置状态和结果转换，并要求拒绝输入形成稳定拒绝结果，C4 明确要求拒绝输入也有稳定结果，兼容边界还要求所有新增公开拒绝结果具有稳定错误码和字段路径喵。
- 整改建议是建立版本化分支输入信封和类型化结果契约，显式携带 `branchId`、目标、规则引用、`beforeStateId` 与结果引用，接受和拒绝都返回稳定 `code` 与 `fieldPath`，拒绝结果进入正式输入结果或因果节点，但不得伪造为已发生状态变化喵。

### P1-03：分支历史哈希和校验没有约束完整谱系元数据，可伪造共同祖先喵

- 严重等级为 P1 喵。
- 涉及位置为 `src/sim/branching.ts:63` 至 `src/sim/branching.ts:78`、`src/sim/branch-comparison.ts:9` 至 `src/sim/branch-comparison.ts:13` 喵。
- 现实触发或复现方式为保持 `branchId`、`parentBranchId`、`forkStateId` 和内部状态不变，只改写 `lineage`、`forkTick`、`commonTransitionCount` 与 `branchInputIds`，当前 `validateBranch` 仍会通过，因为 `branchHistoryHash` 不包含这些字段，比较器随后直接信任被改写的 `lineage` 喵。
- 咪使用临时定向探针亲自验证了把另一条兄弟分支插入谱系、把分叉时刻和共同历史范围改为 999、把分支输入索引改为伪造 ID 后，`validateBranch` 仍然成功，比较器还会把兄弟分支报告为共同祖先，该探针文件已经删除喵。
- 实际影响是历史包或内存分支可以携带与真实状态不一致的谱系和分叉范围，主要比较入口会给出错误共同祖先、错误共同历史和错误首个分歧位置，历史完整性校验无法发现这类篡改喵。
- 判断依据是冻结 R1 要求保存正式谱系与有序分支输入，R6 要求完整原因和输入顺序可验证，R7 禁止伪造共同祖先，C7 要求定位真实共同历史，C11 要求历史包身份和历史篡改明确拒绝，威胁模型明确列出“共同祖先伪造”和“输入顺序篡改”喵。
- 整改建议是把规范化谱系、父子关系、分叉时刻、分叉状态、共同历史范围、分支输入索引和共享来源全部纳入历史哈希与检查点身份，并在 `validateBranch` 中逐项验证父子链、唯一性、范围、历史前缀和输入归属，比较前必须使用经验证的谱系证明喵。

### P1-04：共同祖先比较没有提供冻结要求的可定位因果解释，失败还会被静默吞掉喵

- 严重等级为 P1 喵。
- 涉及位置为 `src/sim/contracts/branching.ts:42` 至 `src/sim/contracts/branching.ts:54`、`src/sim/branch-comparison.ts:22` 至 `src/sim/branch-comparison.ts:30`、`src/sim/branch-comparison.ts:55`、`src/ui/useBranchLaboratory.ts:47` 至 `src/ui/useBranchLaboratory.ts:53`、`src/components/pages/BranchesPage.tsx:26` 至 `src/components/pages/BranchesPage.tsx:33` 喵。
- 现实触发或复现方式为正常比较两条有差异的分支，结果只给出首个输入 ID、状态字段差异、左右独有转换 ID 和左右独有因果节点 ID，状态差异的 `ruleId` 被统一写成 `branch.compare@1`，没有实际规则、输入到差异的路径或仍然相同部分喵。
- 另一现实触发路径是导入不同宇宙的创世包后，旧宇宙分支仍在本地列表中，点击与当前分支比较会使 `compareUniverseBranches` 抛出跨宇宙错误，但 `useMemo` 的空 `catch` 只返回 `undefined`，页面不显示可访问错误喵。
- 实际影响是玩家只能看到差异数量，不能回答关键结果由哪个输入、规则或因果节点造成，比较失败时也无法判断操作是否执行，这使步骤 4 的主要产品价值“比较偶然与必然”无法达到冻结标准喵。
- 判断依据是冻结 R7 要求比较结果包含共同祖先、首个不同输入、状态差异、转换差异、因果路径差异和仍然相同部分，并要求每项关键差异定位到输入、规则、状态或因果节点，C7要求可定位共同历史和因果差异，R9 要求比较失败显示可访问错误喵。
- 整改建议是把比较结果改为结构化差异证据，包含实际输入引用、实际规则引用、前后状态、因果节点及路径、共同部分和转换详情，并把比较异常提交到协调层 `error` 状态，通过 `role="alert"` 展示并测试跨宇宙与损坏谱系失败喵。

### P1-05：活动分支切换和分享包导入没有进入 IndexedDB 原子事务喵

- 严重等级为 P1 喵。
- 涉及位置为 `src/sim/contracts/branching.ts:81` 至 `src/sim/contracts/branching.ts:88`、`src/ui/useBranchLaboratory.ts:83` 至 `src/ui/useBranchLaboratory.ts:116`、`src/ui/branchStorage.ts` 喵。
- 现实触发或复现方式为导入有效创世条件包或历史分支包，页面立即显示“分享内容已创建独立本地分支”，但 `importPackage` 没有调用任何 `storage.put`，刷新页面后导入分支会丢失喵。
- 另一触发路径是切换到已保存分支后刷新，存储契约中没有活动分支指针，`switchBranch` 也没有事务写入，页面会重新从内存根分支开始，现有 E2E 必须在刷新后手动寻找并再次切换才能继续喵。
- 实际影响是页面可以显示无法恢复的虚假成功，导入、活动分支身份与分支记录不能原子提交，数据库不可用、事务中止和并发竞争时也无法证明原活动分支保持不变喵。
- 判断依据是冻结 R10 明确要求分支写入、活动分支切换、检查点保存和导入先完成校验与事务再提交页面成功状态，C13 要求写入或事务失败时旧数据和当前活动分支保持不变且页面不显示虚假成功喵。
- 整改建议是扩展版本化存储契约，增加宇宙级活动分支记录和原子 `commitImport`、`switchActiveBranch` 或等价事务接口，协调层必须等待事务成功后再更新 React 状态与运行状态，并增加失败、事务中止、并发和刷新恢复测试喵。

### P1-06：历史包接收者在导入时立即分叉，且创世包身份错配没有完整校验喵

- 严重等级为 P1 喵。
- 涉及位置为 `src/sim/branch-packages.ts:17` 至 `src/sim/branch-packages.ts:37`、`src/sim/branching.ts:47` 至 `src/sim/branching.ts:60`、`src/ui/useBranchLaboratory.ts:106` 至 `src/ui/useBranchLaboratory.ts:115` 喵。
- 现实触发或复现方式为导入历史分支包，`continueHistoryBranchPackage` 在接收阶段立即调用 `continueSharedUniverseBranch` 创建子分支，页面只加入这个子分支，分享节点本身没有作为可恢复的接收节点保存，尚未发生第一次继续操作就已经改变了分支身份喵。
- 创世条件包解析只校验包类型、版本和外层校验和，没有验证 `UniverseDefinition` 版本、`universeDefinitionId` 与 Seed、规则版本、模板及初始输入的一致性，重新计算校验和后的身份错配定义会在导入时被 `createInitialUniverseState` 静默规范化为另一个身份，而不是按冻结要求拒绝喵。
- 实际影响是分享接收者无法先恢复并查看原分享节点，首次继续操作的语义无法验证，创世条件包的声明身份也可能与实际创建身份不同喵。
- 判断依据是冻结 R11 明确要求历史包先恢复分享节点，接收者第一次继续操作时才创建自己的子分支，并要求身份错配明确拒绝，C10 要求两类包在恢复和继续语义上明确区分，C11 要求历史内容篡改拒绝喵。
- 整改建议是把“接收历史节点”和“首次继续分叉”拆成两个操作，接收时保存只读共享节点，首次推进、实验或干预时才创建接收者子分支，并对创世定义执行规范重建后逐字段完全一致校验，未知版本、身份和初始输入错配返回稳定错误码喵。

### P2-01：历史分支页面没有完整显示冻结要求的分支身份信息喵

- 严重等级为 P2 喵。
- 涉及位置为 `src/components/pages/BranchesPage.tsx:10` 至 `src/components/pages/BranchesPage.tsx:16` 喵。
- 现实触发或复现方式为打开“历史分支”页面查看当前分支，页面显示分支 ID、父分支、分叉时刻、状态哈希、历史哈希和检查点，但没有显示 `commonTransitionCount` 对应的共同历史范围，也没有显示当前分支是否可继续运行及其原因喵。
- 实际影响是玩家无法仅凭分支身份卡确认共同历史边界和可继续状态，遇到接收包、损坏分支或未来受控只读分支时难以判断下一步操作喵。
- 判断依据是冻结 R9 明确要求历史分支页面显示当前分支、父分支、分叉点、共同历史范围、状态哈希、历史哈希、检查点和可继续运行状态喵。
- 整改建议是在当前分支卡和列表项中增加共同历史范围与结构化可继续状态，并为不可继续原因提供可访问说明和直接组件测试喵。

### 建议-01：评估把 32 位完整性指纹升级为更强的内容摘要喵

- 严重等级为建议项喵。
- 涉及位置为 `src/sim/runtime-integrity.ts:3` 至 `src/sim/runtime-integrity.ts:5` 以及步骤 4 分支、存档和分享包校验调用处喵。

- 整改建议是在不改变本轮冻结范围的前提下评估使用更长的确定性内容摘要，并保持版本化迁移与旧摘要明确拒绝或受控读取喵。
