## 4. P1 问题喵

### P1-01：观察强度投影把无关文明和神话登记为直接原因喵

证据如下喵

- 实际观察值只消费当前区域的文明集合，区域过滤发生在 `src/ui/observationProjection.ts:102-107`，星系、恒星系和行星的文明、魔法与神性公式位于 `src/ui/observationProjection.ts:104-131` 喵
- 因果投影却在 `src/ui/observationCausalProjection.ts:60-74` 使用整个宇宙的全部文明与全部神话作为任意节点文明、魔法和神性强度的直接原因，已经计算出的 `context.civilizations` 没有用于这些来源集合喵
- `tests/ui/causal-projection-integration.test.tsx:64-75` 进一步断言每一个宇宙文明和神话都必须成为当前行星投影的直接原因，因此测试正在保护错误的因果关系喵
- 这与 `docs/product-definition.md:49-61` 要求保留实际发生原因，以及 `docs/non-functional-requirements.md:70` 要求二维观察值引用实际原因的约束冲突喵

影响如下喵

玩家追溯一颗行星的文明、魔法或神性强度时，会看到位于其他星系且从未参与该公式的文明被列为直接原因喵

这不是证据冗余问题，而是把未参与计算的对象伪造为原因，会直接破坏产品最高优先级的因果真实性喵

可验证整改条件如下喵

1. 每个观察值的因果来源必须由实际计算操作数生成，文明、魔法和神性至少要使用当前 `SpatialContext` 的区域文明及对应神话，而不能使用整个宇宙集合喵
2. 应把观察值计算与来源提取收敛到同一份可执行投影定义，避免公式和因果映射分别维护后再次漂移喵
3. 新增负向测试，构造当前区域外的文明，并证明其不会出现在当前节点对应强度的直接原因中喵
4. 新增逐项重算测试，证明投影描述中的数值能够由登记的领域操作数和观察规则重新计算得到喵

### P1-02：跨宇宙法则差异结果在单侧图中声称了双侧结论，却没有双侧原因喵

证据如下喵

- `src/ui/lawComparisonCausalProjection.ts:34-57` 使用左右两个宇宙共同计算 `右值 - 左值`、最大绝对差和并列裁决结果喵
- `src/ui/lawComparisonCausalProjection.ts:72-89` 为每个投影节点只登记当前单侧宇宙的领域与法则节点，但同一个投影节点的描述仍展示另一侧数值、差值公式以及跨六领域选出的最大差异结论喵
- `tests/ui/causal-projection-integration.test.tsx:105-123` 只验证左右图彼此独立和每一侧包含六领域，没有验证承载双侧结论的节点是否同时具备双侧证据喵
- `docs/product-definition.md:49-61` 要求每个结果保留自己实际发生的原因，而 `docs/non-functional-requirements.md:70` 只禁止伪造跨图边，并没有授权把缺失的另一侧原因从结果中省略喵

影响如下喵

玩家在左侧或右侧图中看到的是一个完整双侧比较结论，但因果链最多只能证明其中一半输入喵

尤其是“最大差异领域”必须由两侧六个领域的十二个值共同决定，任何单侧图都不能独立证明该选择喵

可验证整改条件如下喵

1. 建立明确的跨图组合证据契约，至少记录左右宇宙身份、左右图身份、左右来源节点和比较规则，并允许 UI 从组合结果分别进入两侧图喵
2. 如果步骤 1 不引入组合图，则单侧投影节点只能陈述并追溯本侧评分，不得在该节点中宣称差值或最大差异选择已经被完整证明喵
3. 为领域差值、最大差异、并列裁决和左右宇宙身份错配增加负向测试，证明缺失任一侧来源时比较结果不能通过完整性校验喵
4. 保持两张随机历史互不合并，但必须让跨图结果的证据组合本身成为可校验的一等契约喵

### P1-03：因果闭包校验无法证明输入根是真实显式输入，公开构建器还允许自由拼接不相关数据喵

证据如下喵

- `src/sim/contracts/causality.ts:5-74` 的输入根只包含通用的 `kind`、`root`、`subjectId` 和说明字段，没有输入类型、顺序、规范值摘要或生成信封身份喵
- `src/sim/causality-query.ts:376-379` 对任意 `root === "input"` 的节点直接返回真实，不会核对该节点是否对应本次生成调用中的 Seed、模板、规则版本或有序干预输入喵
- `src/sim/causality.ts:28-55` 的 `buildUniverseCausalGraph` 分别接收宇宙结果、随机追踪和干预输入，并只对最终图执行结构闭包断言，没有验证三者来自同一次生成喵
- `src/sim/index.ts:4-13` 把该低层构建器作为公共 API 导出，因此这不是只能由受控内部代码触发的实现细节喵
- `tests/step-1-causality.test.ts:293-334` 的全部问题码负向测试没有“伪造输入根”“额外输入根”“输入顺序变化”或“宇宙与随机追踪不匹配”用例喵

影响如下喵

任意无原因事实都可以通过改成 `input` 根来绕过根可达性约束，互不相关的宇宙结果和随机追踪也可以被装配成结构上有效的图喵

因此，当前闭包校验能够证明图形结构闭合，却不能证明根因确实来自本次显式输入，不能承担产品文档宣称的因果真实性边界喵

可验证整改条件如下喵

1. 因果图或其生成信封必须包含规范化且有序的显式输入清单，并区分 Seed、规则版本、创世条件和每一项干预输入喵
2. 校验入口必须接收或携带可信生成上下文，并拒绝额外输入根、缺失输入根、输入类型错配、顺序错配和值摘要错配喵
3. 随机追踪必须与规范化生成身份绑定，不能只依赖一个无法反查原输入的八位 Seed 指纹喵
4. `buildUniverseCausalGraph` 应降为内部装配能力，或改为只接收由同一次受控重放产生且不可自由拼接的验证信封喵
5. 新增上述伪造、缺失、额外、重排和跨宇宙拼接的直接负向契约测试喵

## 5. P2 问题喵

### P2-01：随机引用校验没有证明决策属于引用声明的随机流喵

证据如下喵

- `src/sim/causality-query.ts:210-235` 分别按 `reference.streamId` 查找流，又从全局决策表按 `decisionId` 查找决策，但没有校验该决策实际位于所查到的流中喵
- 当前条件只比较流命名空间、样本序号、候选集和选中值，因此两个流在同一样本序号产生相同候选标识和选中值时，可以把甲流的 `streamId` 与乙流的 `decisionId` 拼成一个被接受的引用喵
- `tests/step-1-causality.test.ts:139-168` 使用同样的两个独立 Map 做正向核对，也没有断言决策所属流与 `reference.streamId` 相同喵
- 这与 `docs/non-functional-requirements.md:62` 明确要求同时通过流、命名空间、样本序号、候选集和选中值校验的约束冲突喵

影响如下喵

错误或被篡改的图可以把一个结果的随机证据挂到错误随机流上，而校验器仍报告有效喵

可验证整改条件如下喵

1. 决策索引必须保留所属流，并强制 `reference.streamId`、决策所属流和 `decisionId` 中编码的流身份完全一致喵
2. 新增至少两个随机流的负向测试，在保持样本序号、候选集和选中值相同的情况下交换 `streamId` 或 `decisionId`，并断言返回 `INVALID_RANDOM_REFERENCE` 喵
3. 正向测试必须直接验证每条引用的决策对象存在于该引用指定流的 `decisions` 集合中喵

### P2-02：无效因果目标会静默回退到另一个节点，与明确错误要求相反喵

证据如下喵

- `src/components/causalExplorerModel.ts:106-109` 在 `initialNodeId` 不存在时直接选择第一个派生节点喵
- `src/components/CausalExplorer.tsx:50-55` 在受控 `selectedNodeId` 不存在时也会回退到内部节点或默认节点，而不会显示目标无效状态喵
- `docs/non-functional-requirements.md:77` 明确要求因果查询目标不存在时显示明确空状态，不得自动替换为另一个结果并隐瞒错误喵
- 当前组件测试只覆盖有效初始节点和整图为空的情况，没有覆盖“图有效但请求目标不存在”的场景喵

影响如下喵

当投影 ID 过期、跨宇宙请求错配或外部调用传入错误节点时，玩家会看到另一个合法节点，容易误以为该节点就是所请求结果的原因喵

可验证整改条件如下喵

1. 明确区分“调用方没有指定目标”和“调用方指定了不存在的目标”两种状态喵
2. 对后者展示包含原目标标识的中文错误或空状态，不得选择其他节点喵
3. 为无效 `initialNodeId`、无效受控 `selectedNodeId` 和跨图节点 ID 增加组件测试与至少一条应用集成测试喵

### P2-03：架构文档宣称的门禁没有全部落到可执行测试中喵

证据如下喵

- `docs/quality-gates.md:136-154` 明确要求 `src/ui/causalView.ts`、`src/ui/causalProjectionSources.ts`、`src/ui/observationCausalProjection.ts`、`src/ui/lawComparisonCausalProjection.ts` 和 `src/ui/useLawComparisonModel.ts` 进入文件规模审计喵
- `tests/architecture.test.ts:91-120` 的实际规模上限 Map 没有这五个文件，因此这些文件可以无限增长而门禁仍然通过喵
- `docs/architecture.md:45` 规定组件不得调用宇宙生成器，但 `tests/architecture.test.ts:24-39` 的生成器黑名单只包含 `generateUniverse`、`generateGalaxies`、`generateCivilizations` 和 `applyInterventions`，遗漏当前主产品入口 `generateCausalUniverse` 喵
- `docs/legacy-module-migration.md:139` 因此把步骤 1 架构边界和规模保护线描述为“已关闭”，证据强度高于实际测试能力喵

影响如下喵

后续修改可以在不触发门禁的情况下让关键 UI 编排文件失控增长，或让组件直接调用新的主生成器，文档与自动化之间会持续漂移喵

可验证整改条件如下喵

1. 把质量门禁列出的全部步骤 1 文件加入可执行规模检查，并为每个文件给出与职责相称的上限喵
2. 组件生成器边界应采用能力或模块边界判定，不应依赖容易漏项的少量函数名黑名单喵
3. 至少增加一个针对 `generateCausalUniverse` 的架构负向夹具或 AST 级断言，证明组件直接调用当前或未来产品生成入口会失败喵
4. 文档只有在相应断言真实执行后才能继续使用“门禁已关闭”的表述喵

## 6. 建议项喵

### 建议-01：为查询建立可复用只读索引喵

`src/sim/causality-query.ts:21-37` 和 `src/sim/causality-query.ts:326-353` 在每次直接查询或递归查询时重新构建节点 Map，并在广度遍历中使用数组 `shift` 喵

当前静态图仍能通过性能预算，但步骤 2 以后因果历史持续增长时，这会成为重复查询成本和主线程抖动来源喵

建议为冻结图建立 WeakMap 只读索引，并使用游标队列替代频繁 `shift`，同时新增多规模图的查询增长曲线测试喵

### 建议-02：把观察公式与因果来源定义为同一份声明喵

当前 `observationProjection.ts` 负责数值公式，`observationCausalProjection.ts` 另行猜测来源，两份实现已经出现 P1-01 所示漂移喵

建议让每项投影返回数值、结构化操作数、计算规则 ID 和展示说明，再由展示层与因果层共同消费喵

### 建议-03：减少因果界面对内部标识的默认暴露喵

`src/components/CausalExplorer.tsx:188-199` 默认向普通玩家展示 `subjectId`，这对调试有价值，但会增加内部枚举感并削弱产品语言的一致性喵

建议把内部标识放入可展开的“技术证据”区域，默认优先展示中文对象名、规则、状态和值喵
