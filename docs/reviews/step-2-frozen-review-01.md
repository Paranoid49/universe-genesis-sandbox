## 4. P1 问题喵

### P1-01：运行控制改变状态身份，导致后续因果状态链断裂喵

- 严重等级：P1 喵
- 涉及位置：`src/sim/runtime-state.ts:173-182`、`src/sim/runtime-state.ts:291-300`、`src/sim/runtime-causality.ts:149-173`、`src/ui/useRuntimeUniverseModel.ts:78-95` 喵
- 现实触发或复现方式：启动默认运行宇宙，单步推进一次，把速度从 1 倍调整为 4 倍，再执行一次单步推进喵
- 独立复现证据：咪使用临时 Chromium E2E 用例按上述步骤操作，第二次推进后页面稳定出现“运行因果网络校验失败”，命中 `MISSING_CAUSE` 或 `NO_ROOT_PATH`，临时用例随后已删除喵
- 实际影响：玩家执行冻结基线明确要求的“推进—调速—继续推进”后，运行主界面进入顶层错误恢复界面，当前宇宙无法继续正常浏览或追因喵
- 判断依据：冻结 R2 要求速度调整不能破坏确定性时间步，R10 和 C1 要求运行控制形成可用闭环，C4 及步骤 1 不变量要求所有运行变化保持根可达的因果闭包喵
- 根因：`configureUniverseClock` 把 `status` 或 `speed` 写入 `UniverseState` 并重新计算 `state.id`，却不生成状态转换或控制输入喵
- 根因后果：后续转换的 `beforeStateId` 指向一个因果网络从未登记的控制后状态；同时相同世界演化仅因速度不同就产生不同状态 ID 和转换 ID，控制调度污染了确定性状态身份喵
- 整改建议：把运行状态与速度从决定世界事实身份的模拟状态中分离为调度控制状态，或者把每次控制变更建模为显式、有因果记录的类型化控制转换喵
- 必要回归：覆盖“推进后调速再推进”“连续运行后暂停再单步”“恢复后改速再推进”，并逐步比较状态哈希、转换 ID、因果闭包和世界对象序列喵

### P1-02：对象的实际字段变化没有完整进入 `StateDiff` 和值级因果记录喵

- 严重等级：P1 喵
- 涉及位置：`src/sim/runtime-state.ts:102-113`、`src/sim/runtime-events.ts:7-33`、`src/sim/runtime-history.ts:10-24` 喵
- 现实触发或复现方式：任意推进一个时间步，比较推进前后的持久对象与该转换的 `differences` 喵
- 实际影响：每一步都会改变对象的 `revision` 和 `updatedAtTick`，但差异日志只记录凝聚度、能量和可选状态变化；事件和差异因果节点无法说明这两个可见世界事实为何改变喵
- 进一步影响：当数值被钳制而没有记录到属性差异时，仍可能出现对象版本和更新时间变化但没有对应对象事件的情况，状态变化与历史投影不再一一核对喵
- 判断依据：冻结 R2 要求类型化转换明确保存可验证状态差异，R4 要求历史来自已提交真实差异，R7 要求派生事件和解释可以重建，步骤 1 不变量要求所有状态变化拥有准确直接原因和值级入口喵
- 整改建议：生成结构化前后状态差异时覆盖所有被提交的对象字段，或者取消不代表世界变化的版本递增；历史反投影必须只消费完整差异语义，不能通过“每个转换减一”猜测对象版本喵
- 必要回归：自动比较每次转换前后对象的所有字段与 `StateDiff` 操作集合，并验证每项可见变化在运行因果网络中拥有准确字段节点喵

### P1-03：运行存档只校验指纹和数量，没有校验转换历史的语义一致性喵

- 严重等级：P1 喵
- 涉及位置：`src/sim/runtime-state.ts:284-300`、`src/sim/runtime-archive.ts:26-80` 喵
- 现实触发或复现方式一：克隆一个有效运行状态，把首个转换差异的 `before` 改为 999，再调用 `createRuntimeArchive` 和 `restoreRuntimeArchive` 喵
- 现实触发或复现方式二：把转换的 `inputIds` 改为一个输入日志中不存在的 ID，再创建并恢复存档喵
- 独立复现证据：两种临时定向测试都通过了创建与恢复；第二种状态只有在稍后构建因果网络时才因 `MISSING_CAUSE` 失败，临时测试文件随后已删除喵
- 实际影响：损坏或拼接的历史可以被系统自己重新封装成校验和正确的“有效存档”，恢复后保存的差异、输入、随机决定、前后状态和当前快照可能互相矛盾喵
- 判断依据：冻结 R8 和 C8 要求状态哈希错配、历史缺失、随机流不完整和身份拼接明确拒绝；必要威胁模型还明确要求篡改差异能够被完整性校验发现喵
- 根因：`assertUniverseState` 只检查版本、当前 `state.id` 和“转换数等于提交 ID 数”，而 `state.id` 本身不包含 `transitions` 内容；存档层只重新计算外层指纹，没有重放或验证转换链喵
- 整改建议：新增运行状态语义校验器，逐项校验转换版本、ID、前后状态链、提交顺序、输入引用、规则引用、随机决定与流游标、差异前后值、对象快照和最终状态一致性喵
- 必要回归：每个语义破坏点提供直接负向测试，并证明合法存档恢复后的下一步与未中断路径仍完全一致喵

### P1-04：连续运行中的存档错误不会进入停止状态，错误后仍自动推进喵

- 严重等级：P1 喵
- 涉及位置：`src/ui/useRuntimeUniverseModel.ts:40-52`、`src/ui/useRuntimeUniverseModel.ts:97-137` 喵
- 现实触发或复现方式：开始连续运行，完成一步后让存储适配器拒绝写入，再点击保存检查点并等待下一次定时调度喵
- 独立复现证据：临时 UI 定向测试确认错误提示出现后，模拟位置仍从第 1 步自动推进到第 2 步，临时测试文件随后已删除喵
- 实际影响：玩家看到“存档失败”后可能合理认为宇宙已经停止在错误检查点，但系统会在异步锁释放后继续改变世界状态，增加未保存历史并扩大恢复差异喵
- 判断依据：冻结 R12 要求错误恢复时释放调度任务，质量要求 6.4 明确要求连续运行、恢复和存储任务异常时进入可恢复错误状态且不得继续静默推进喵
- 根因：保存和恢复只在异步任务期间通过 `operationInProgress` 暂时阻止 `advance`，失败分支没有把时钟提交为暂停状态；`finally` 解锁后原有运行计时器继续生效喵
- 整改建议：存档或恢复任务开始时以原子方式进入暂停或明确的错误状态，失败后保持该状态并要求玩家显式继续；错误路径同时清理调度器和待处理任务喵
- 必要回归：覆盖运行中保存失败、运行中恢复失败、IndexedDB 打开失败、事务失败及失败后真实时间等待，确认状态哈希和步数不再变化喵

### P1-05：任意查询参数都会绕过默认运行主流程并进入旧产品喵

- 严重等级：P1 喵
- 涉及位置：`src/App.tsx:12-19` 喵
- 现实触发或复现方式：访问 `/?utm_source=test`、`/?debug=1` 或任何非空且不是旧分享载荷的查询字符串喵
- 实际影响：常见分析参数、部署参数或无关查询参数会让新访客直接进入旧版隔离兼容视图，默认产品再次表现为一次性静态事件浏览器喵
- 判断依据：冻结 R11 和 C11 要求默认启动进入运行中宇宙，只有明确保留的 UGS070、I1 和 A1 兼容输入可以进入只读兼容路径喵
- 根因：`startsInLegacy` 使用 `Boolean(resolvedSearch)` 判断，而不是识别受支持的旧载荷参数喵
- 整改建议：增加严格的旧分享查询识别函数，只在成功识别受支持的兼容格式时进入 `LegacyApplication`，未知查询参数必须保留默认运行流程并提供必要提示喵
- 必要回归：分别覆盖 UGS070、I1、A1、未知参数、空参数和混合参数，证明旧兼容能力保持且默认主流程不被误切换喵

### P1-06：冻结要求的“暂停命令 100 毫秒”独立性能门禁没有实现喵

- 严重等级：P1 喵
- 涉及位置：`tests/performance/runtime.performance.ts:1-88`、`docs/step-2.md:50-56`、`docs/quality-gates.md:136` 喵
- 现实触发或复现方式：检查当前步骤 2 非覆盖率性能用例，它只采样单步、一百步、快照和恢复四类指标，没有测量暂停命令到停止下一次调度的延迟喵
- 实际影响：无法证明低速、高速或时间步接近完成时，玩家的暂停命令会在冻结预算内阻止下一步调度；当前发布快照的“性能门禁通过”没有覆盖全部冻结预算喵
- 判断依据：冻结质量要求 6.3 明确规定暂停命令在当前时间步提交完成后 100 毫秒内停止调度下一步，6.1 要求执行独立非覆盖率性能测试喵

## 6. 建议项喵

### 建议-01：提供“以相同 Seed 重新启动当前宇宙”的明确入口喵

当前 `RuntimeSession` 的 React `key` 只由 Seed 和模板组成，玩家在已经运行后再次点击相同 Seed 和模板的“创世”不会重置会话喵

冻结基线只要求能够启动新宇宙，没有明确要求同参数重启，因此本项不阻塞步骤 2 喵

### 建议-02：把 IndexedDB 版本迁移策略显式化喵

当前 `onupgradeneeded` 只创建第一版对象仓库，公共 `RuntimeStorageAdapter` 也没有显式迁移入口或迁移结果类型喵

当前数据库只有版本 1，尚无现实旧版本迁移路径，因此本项只建议在步骤 2 关闭文档中明确“当前迁移能力”的边界，不要求提前实现步骤 7 的长期存档迁移喵
