## 6. P1 问题详细证据喵

### P1-01：观察入口不能从对象、尺度、时间和现象任意开始喵

- 严重等级：P1 喵
- 涉及位置：`src/components/pages/ObservationPage.tsx:15-25`、`src/ui/useObservationWorkbench.ts:45-49`、`src/sim/observation.ts:15-18`、`src/sim/observation.ts:21-29` 喵
- 现实触发或复现方式：进入“观察”页面，玩家只能点击四个观察方式并执行观察，页面没有对象选择、尺度入口、时间入口或现象入口，工作台固定取 `context.visibleObjectIds[0]` 作为观察对象喵
- 实际影响：玩家无法围绕自己选择的对象或问题开始研究，也无法从尺度、时间和现象任意切入；即使运行状态未来存在多个对象，界面仍会静默观察排序后的第一个对象喵
- 判断依据：冻结 R3 明确要求“按对象、尺度、时间和现象组织的观察入口，玩家可以从其中任意一种方式开始”，C3 要求多条互不依赖的探索路径，C1 要求玩家从当前宇宙自主选择观察入口喵
- 根因：`ObservationMethod.scale` 和 `phenomenon` 目前只是方法元数据，没有形成可选入口；`ObservationRequest` 由 Hook 内部自动组装，玩家没有对象和时间参数控制权喵
- 整改建议：建立只读观察入口模型，分别提供对象、尺度、时间和现象入口，最终统一转换为显式 `ObservationRequest`；不得通过读取完整状态临时填充入口喵
- 必要回归：以至少两个可见对象、两个时间位置和不同尺度／现象夹具，分别从四类入口开始并获得可追溯信号，证明任一路径不依赖固定对象或固定顺序喵

### P1-02：新主流程直接持有完整 `UniverseState`，认知边界可被页面属性绕过喵

- 严重等级：P1 喵
- 涉及位置：`src/components/RuntimeApplication.tsx:79-89`、`src/components/pages/ObservationPage.tsx:8`、`src/components/pages/RuntimePage.tsx:6`、`src/ui/useObservationWorkbench.ts:18-22`、`src/sim/observation.ts:21-29`、`tests/step-3-architecture.test.ts:6-27` 喵
- 现实触发或复现方式：检查新主流程装配，`ObservationPage` 接收包含完整 `state`、对象、事件和因果网络的 `RuntimeUniverseController`，`RuntimePage` 同样接收完整控制器，工作台 Hook 直接接收 `UniverseState` 并通过 `Object.keys(state.objects)` 枚举全部底层对象喵
- 实际影响：当前页面虽然没有把全部字段渲染出来，但任何页面重构、调试组件、缓存或选择器都能直接读取未观察对象数量、属性、规则和完整历史；冻结要求的认知边界依赖开发者自律，而不是接口能力约束喵
- 判断依据：冻结 R1 要求新观察界面只能直接获得可用信号、证据和公开公理，C2 明确要求底层事实不能通过页面属性泄露，R11 要求条件视图只消费观察上下文与可见引用，R12 和 C12 要求新主流程不存在旧式完整事实依赖；必要威胁模型要求新主流程只接收观察上下文与可见结果喵
- 根因：当前架构测试只扫描 `RuntimeApplication.tsx` 是否出现旧类型名称，没有检查页面 Props、Hook 参数或控制器可达字段，因而无法发现完整状态引用已经穿过页面边界喵
- 整改建议：在容器层将运行控制、公开时钟、观察上下文、可见对象引用和因果查询能力拆成最小只读端口，页面不得获得完整 `UniverseState`、完整对象表或完整运行控制器喵
- 必要回归：增加类型或依赖级负向门禁，分别在观察页、概览页、研究页、条件视图和缓存中尝试读取 `state.objects`、`state.rules`、完整差异和旧选择器时必须失败喵

### P1-03：未知四态和冲突解释没有形成可运行、可保存的产品能力喵

- 严重等级：P1 喵
- 涉及位置：`src/sim/contracts/observation.ts:5`、`src/sim/observation.ts:33-40`、`src/sim/observation.ts:63-68`、`tests/step-3-observation.test.ts:43-49` 喵
- 现实触发或复现方式：检查 `KnowledgeStatus`，类型包含 `unobserved`、`insufficient` 与 `confirmed-absent`，但没有“已观察但无结果”的独立状态；执行“近期变化”无语义差异时直接返回 `confirmed-absent`，界面把它显示为确认未出现喵
- 冲突复现方式：当前测试只把两条手工证据传给 `knowledgeStatusFromEvidence` 并断言字符串 `conflicted`，实际观察结果没有问题身份、解释身份或冲突关系，研究存档测试也没有保存两种冲突解释并验证全部来源恢复喵
- 实际影响：玩家无法区分“仪器完成观察但没有给出结果”和“证据确认对象或现象不存在”；多个观察对同一问题给出不同解释时，系统没有稳定聚合边界，后写入结果只能作为互不关联的信号存在喵
- 判断依据：冻结 R4 和 C5 明确要求未观察、证据不足、已观察但无结果和确认不存在四种状态可区分，并要求自动化证明冲突解释可以同时保存且不丢失来源喵
- 根因：知识状态被当作单次信号标签，而不是围绕稳定问题或解释集合计算的认知投影喵
- 整改建议：补充独立无结果状态，定义稳定问题身份和解释身份，保存支持、反对与中性证据的关系，并让冲突状态由同一问题的证据集合确定喵
- 必要回归：直接覆盖四种状态的生成与 UI 文案，并创建两种相反解释，经过研究存档保存、刷新恢复后仍保持冲突状态、解释身份和全部来源喵

### P1-04：主导航和当前宇宙概览没有完成冻结的信息架构喵

- 严重等级：P1 喵
- 涉及位置：`src/components/AppChrome.tsx:94-112`、`src/components/pages/RuntimePage.tsx:6-19`、`src/components/RuntimeApplication.tsx:86-89` 喵
- 现实触发或复现方式：默认导航只有“当前宇宙、观察、研究记录、已发生历史、旧版兼容”，没有存档组织入口；当前宇宙页只显示时钟、状态 ID、关注数量、观察数量和固定“保持未知”卡片喵
- 实际影响：玩家不能从新主信息架构浏览或组织运行检查点与研究存档，只能使用当前页上的两个即时按钮；概览也没有展示近期可观察变化、具体关注内容或由观察得到的当前可知领域结论喵
- 判断依据：冻结 R7 和 C8 要求新主导航围绕当前宇宙、观察、研究记录、已发生历史和存档组织，R8 要求新概览由当前可知状态、近期可观察变化、未知区域和玩家关注内容组成喵
- 根因：实现只把步骤 2 的运行页裁剪为 `current`／`history` 两个视图，并用研究数量替代了观察驱动的概览模型，没有建立独立的认知概览投影和存档导航责任喵
- 整改建议：增加新主流程内的存档组织入口，并建立只消费公开时钟、已观察信号、研究关注和未知状态的概览模型；不要复用旧图书馆或提前实现步骤 4 分支喵
- 必要回归：概览在没有观察时不泄露事实，完成观察和关注后出现可追溯的当前可知结论、近期观察变化与关注内容；导航可进入步骤 2 检查点和步骤 3 研究存档的组织视图喵

### P1-05：法则认知分层与条件对象视图没有实现喵

- 严重等级：P1 喵
- 涉及位置：`src/sim/contracts/observation.ts:8-29`、`src/sim/observation.ts:15-18`、`src/components/pages/ObservationPage.tsx:15-42`、`src/components/RuntimeApplication.tsx:86-89`、`docs/legacy-module-migration.md:192-194` 喵
- 现实触发或复现方式：执行“规律追踪”只能看到已提交转换中的规则 ID，数据和界面没有分别呈现公开公理、观察支持的规律、内部观点边界和未知规则；完成结构观察后也不存在由可见对象引用进入的传统空间条件视图喵
- 实际影响：玩家无法判断某条法则是创世时公开设定、观察归纳、内部实体观点还是仍未知；传统空间对象即使已经获得信号，也只能停留在通用信号卡片，无法进入冻结要求的条件对象视图喵
- 判断依据：冻结 R10 和 C9 要求四类法则认知在数据与界面可区分，R11 和 C10 要求传统空间对象能够通过观察上下文进入条件视图，并在无对象或无能力时隐藏入口喵
- 根因：实现把法则分层简化为单一 `rule-trace` 方法，把条件对象视图整体留给未来泛化；迁移台账 S3-08 甚至写明“通用条件对象视图由步骤 5 泛化”，与步骤 3 冻结要求直接冲突喵
- 整改建议：在步骤 3 范围内实现最小法则认知投影和传统空间条件适配器，只消费观察上下文与可见引用；不要求实现步骤 5 的动态规则包或非传统空间泛化喵
- 必要回归：分别验证四类法则状态的类型和界面标识，并验证有可见传统空间信号时出现条件入口、无信号或无能力时入口、路由和可访问名称均不存在喵

### P1-06：研究存档无法验证信号协议、运行历史身份和证据来源真实性喵

- 严重等级：P1 喵
- 涉及位置：`src/sim/contracts/observation.ts:32-55`、`src/sim/contracts/observation.ts:119-125`、`src/sim/research-archive.ts:24-49`、`src/sim/research-archive.ts:56-74`、`src/sim/observation.ts:90-116` 喵
- 现实触发或复现方式：通过公共 `createResearchArchive` 传入同宇宙 ID 但修改过 `ObservableSignal.version`、`stateId`、`methodId`、信号 `id`、证据 `id`、`sourceSubjectIds` 或 `causalNodeIds` 的记录，当前 `validateNotebook` 只检查信号的 `universeDefinitionId` 和引用集合，随后会为篡改内容生成新的合法外层校验和喵
- 身份缺口：`ObservationContext` 有 `runtimeHistoryId`，但 `ObservableSignal` 和 `EvidenceRecord` 没有运行历史身份；`EvidenceRecord` 也没有冻结要求规定的独立协议版本，研究记录无法证明来源属于当前运行历史喵
- 接口缺口：冻结公共契约要求 `ResearchStorageAdapter` 提供列举能力，当前类型只有 `migrate`、`put`、`get` 和 `remove` 喵
- 实际影响：调用方错误、迁移错误或本地篡改可以把伪造来源重新封装成校验和正确的研究存档；后续页面会把它作为真实观察证据展示，且无法组织多个研究记录喵
- 判断依据：冻结 R2、R5、R12、C4、C6 和版本身份边界要求结果绑定宇宙、运行历史、方法版本、时间和参数，证据记录拥有独立版本，缺失、跨宇宙或篡改来源必须拒绝；证据来源伪造威胁模型要求明确验证来源引用喵
- 根因：外层校验和只证明序列化内容未在校验和生成后改变，不证明内容本身由观察核心产生；存档校验没有重算信号和证据身份，也没有接收可验证的运行历史来源集合喵
- 整改建议：给信号和证据增加明确版本与运行历史身份，建立不依赖外层校验和的语义校验器，逐项验证方法、参数、信号身份、证据身份和来源引用；补齐存储列举接口喵
- 必要回归：每个公开拒绝原因都使用重新计算过外层校验和的篡改夹具，分别修改协议、历史身份、方法、时间、对象、可见值、证据强度和来源引用时必须拒绝喵

### P1-07：研究保存失败会残留旧成功提示并丢失本次输入喵

- 严重等级：P1 喵
- 涉及位置：`src/ui/useObservationWorkbench.ts:116-130`、`src/components/pages/ResearchPage.tsx:19-20`、`tests/ui/observation-workbench.test.tsx:55-76` 喵
- 现实触发或复现方式：先成功保存一条笔记，使页面显示“笔记已保存”，随后让存储适配器或 IndexedDB 拒绝下一次写入，再输入第二条笔记并点击保存喵
- 实际影响：失败分支只设置 `error`，没有清除旧 `status`，页面会同时显示保存失败错误和旧“笔记已保存”成功状态；`ResearchPage` 无论 `addNote` 或 `addHypothesis` 是否成功都会清空文本框，玩家本次未保存内容也会丢失喵
- 判断依据：冻结 R5 和 C6 要求写入失败明确提示、保留先前有效数据且不得显示虚假保存成功，存储失败威胁模型要求失败时不更新页面成功状态喵
- 根因：工作台持久化函数只返回布尔值给 `observe` 使用，笔记和推测调用方忽略结果；成功状态和错误状态也不是互斥状态机喵
- 整改建议：失败开始或失败完成时清除不再适用的成功状态，只有持久化返回成功才清空输入；最好使用带操作身份的提交结果，避免旧操作状态污染新操作喵
- 必要回归：先成功保存再触发同类保存失败，断言旧成功文案消失、错误可访问、旧有效记录不变且本次未保存输入仍在文本框中喵

### P1-08：强制浏览器与性能门禁没有覆盖冻结要求的完整场景喵
