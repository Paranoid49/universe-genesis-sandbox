## 3. 第八轮问题复核

### P1-01：认证校验缓存污染——已关闭

`src/sim/causality.ts` 使用私有 `WeakSet` 登记生产图，公共成功结果对象及其空 `issues` 数组均冻结。定向测试实际尝试修改 `valid`、替换 `issues` 和追加问题，并证明后续校验与序列化不变。非认证图每次生成独立失败结果，修改一次返回的问题对象不会污染下一次校验。

### P1-02：主要旧页面就地追因——未关闭

第八轮整改新增了概览指标、时间线当前事件、文明、神话、文明事件以及部分干预结果的追因按钮，但这只是抽样覆盖，不是“当前用户可见派生结果”的完整覆盖。

### P2-01：绑定语义依赖硬编码实体名单——已关闭

`CausalEntityLocator` 已携带 `containerKind`，`causality-domain-locators.ts` 的统一位置注册表同时为创建、解析和值指纹提供容器语义，`randomBindingKind` 直接消费定位器语义。字段实体和集合成员的语义错配会被结构校验拒绝。

### P2-02：完整导出语法认证门禁——未完全关闭

当前夹具已覆盖匿名默认导出、默认表达式、导出类、实例方法、静态方法和 `export assignment`。但是 `exportedSymbolsReaching` 只登记函数、类、变量、命名导出声明和导出赋值，没有处理 `ModuleDeclaration`，因此导出命名空间仍是可绕行的命名导出入口。

## 4. P1 问题

### P1-01：步骤 1 的“任意可见结果就地追因”仍只完成抽样闭环

权威要求是 `docs/milestones.md` 第 63 行的“点击任意结果、指标、状态或事件”均可追因，以及第 87 行的“所有用户可见的派生结果都拥有直接原因和规则来源”。当前实现虽然为因果图建立了大量节点，但界面入口仍存在系统性缺口。

直接证据如下：

- `OverviewPage` 只为七个指标增加追因入口；宇宙名称、描述、标语、分享结果以及纪元、星系、文明、神话等统计仍直接展示而无就地入口。
- `SpaceExplorer` 已展示星系、恒星系、行星、生物圈、文明种子及其大量状态值，但正向实体没有追因入口；现有入口只覆盖“未形成生物圈”和“未形成文明种子”两个负事实。
- `TimelineImpactPanel` 展示局部偏置、数值、解释和来源，但没有追因入口。
- `LawsPage` 展示法则领域、具体法则和法则关系，但只有 Seed 对比结果提供追因入口，当前宇宙本身的法则和关系没有入口。
- `LogsPage` 展示重要事件、稀有发现和潜在终局，因果图也为这些观察结果建立了节点，但页面完全没有追因入口。
- 文明页和奇迹页的统计、文明属性、目标变异等可见派生状态也没有逐项入口；干预目标变异仅显示为文字，未连接已经存在的 `target_mutation` 节点。

现有 UI 与 E2E 用例只依次点击 `.metric-tile`、`.event-detail`、文明详情、神话、文明历史和奇迹区域中的首个 `.trace` 按钮。测试没有建立“页面可见结果清单必须与可定位因果主题一一对应”的契约，因此绿色门禁无法发现上述漏项。

整改条件：

1. 建立统一的“可见结果到唯一因果主题”展示契约，禁止各页面继续靠零散按钮人工抽样接入。
2. 对概览、空间、时间线影响、法则、日志、文明和奇迹页面当前展示的全部领域结果逐项登记保留、聚合追因或单项追因语义；聚合节点必须明确覆盖哪些字段，不能用过宽实体节点掩盖字段级原因差异。
3. UI 测试按页面清单验证每类可见派生结果均有可访问入口，E2E 至少覆盖空间正向实体、当前法则、观察日志和目标变异的进入、唯一定位与返回闭环。
4. 主题不存在或不唯一时显示目标和失败原因，不得只显示“追因失败”。

## 5. P2 问题

### P2-01：返回功能只保存页面编号，不能保持用户浏览上下文

`useCausalViewController` 的返回上下文只有 `originPage`。例如 `CivilizationPanel` 的搜索词、路径筛选和分页由组件内部 `useState` 保存，进入因果页时组件卸载，返回后这些状态全部重置。现有 UI 和 E2E 只断言返回后页面标题可见，没有验证筛选、分页、选中对象、焦点或滚动位置仍可继续。

这与 `docs/non-functional-requirements.md` 第 78 行“保存返回原页面继续浏览的上下文”不一致。

整改条件：定义可序列化的页面返回上下文，至少保存会改变当前结果集合或用户位置的筛选、分页、选中对象和焦点目标；为文明筛选后追因返回、空间选中对象追因返回及至少一个长列表焦点恢复增加 UI 与 E2E 用例。

### P2-02：认证能力门禁仍可被导出命名空间绕过

`tests/architecture.test.ts` 的 `exportedSymbolsReaching` 在顶层语句扫描中没有处理 `ts.ModuleDeclaration`。以下合法 TypeScript 语法会导出可调用认证写入能力，但当前分析器不会把 `Unsafe` 登记为导出符号：

```ts
export namespace Unsafe {
  export function certify() {
    return certifyBuiltGraph();
  }
}
```

这仍属于命名导出和 helper 绕行，违反“认证写入能力不得通过命名导出公开”的边界要求。

整改条件：要么递归分析导出命名空间及其成员可达性，要么由架构门禁明确禁止生产模拟模块使用运行时命名空间；增加上述失败夹具，并补充嵌套命名空间和命名空间导出别名用例。

## 6. 建议项

1. 将 `TraceCauseButton` 的可访问名称改为包含结果名称的“追溯……原因”，避免页面出现大量同名“追因”按钮，也使 E2E 不再依赖 CSS 结构选择器。
2. 将页面可见结果覆盖矩阵生成成测试数据，由组件和测试共同消费，避免文档、页面和测试分别维护不一致的人工清单。
3. 将认证可达性门禁逐步迁移到 TypeScript 类型检查器符号图，或者采用“认证模块禁止任何运行时导出、仅白名单公开工厂”的更窄语法策略，降低自制 AST 数据流分析器长期漏判风险。
