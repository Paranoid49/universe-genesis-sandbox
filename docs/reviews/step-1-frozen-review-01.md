## 3. 阻塞问题

### P1-01：可见结果契约只覆盖部分页面，法则、干预与日志中的具体派生值没有值级追因绑定喵

- 严重等级：P1 喵
- 违反要求：R3、R8、C1，以及冻结威胁模型中的“可见结果主题和值指纹”防御要求喵
- 涉及位置：`src/components/pages/LawsPage.tsx:20-40`、`src/components/MiraclePanel.tsx:37-40`、`src/components/MiraclePanel.tsx:90-119`、`src/components/common.tsx:100-109`、`tests/architecture.test.ts:145-157`、`tests/ui/app-interaction.test.tsx:105-154` 喵
- 触发或复现方式：打开“法则与解释”页面，可看到法则评级标签、评级解释、特征、规则值、极性、规则解释、影响目标、代价和关系影响等多个独立派生值，但这些值只共享 `law.id`、`rule.id` 或 `interaction.id` 的宽泛追因按钮，没有 `data-result-subject` 与 `data-result-value` 的逐值登记喵
- 触发或复现方式：打开“造物主干预”页面并施加奇迹，反噬状态、概率偏移、指标变化及日志字段同样直接渲染，仅在整个条目旁放置宽泛主题按钮，无法验证当前显示的具体值是否与该主题的生产事实一致喵
- 触发或复现方式：把 `LawsPage.tsx` 中的 `{rule.value}` 临时替换为 `{rule.value + 1}`，或把 `MiraclePanel.tsx` 中的 `{signed(delta)}` 替换为任意错误文案，现有架构门禁仍不会命中，因为 `protectedFiles` 只包含空间、文明、时间线面板、概览和时间线页面喵
- 触发或复现方式：现有 UI 用例先枚举 `[data-t]` 并验证主题存在，再只对已经存在的 `[data-result-subject]` 核对值指纹，因此完全漏登的结果不会进入后续断言喵
- 实际影响：玩家点击某个法则或干预条目的“追因”时，只能看到宽泛对象原因，无法确认眼前某个评级、代价、概率偏移或指标差值的直接原因与规则来源喵
- 实际影响：展示层发生字段错绑、格式化值引用错误或主题复用错误时，当前发布门禁仍可全绿，不能提供 C1 明确要求的“值与主题一致性测试、漏登负向门禁”证据喵
- 判断依据：C1 使用“所有用户可见派生结果”这一全称范围，并明确要求值与主题一致性及漏登负向门禁，当前实现按文件白名单保护部分页面，不能证明全称条件喵
- 整改建议：把法则、日志、干预页面中具有领域意义的可见派生值全部接入统一结果契约，按需要建立精确生产节点或经过认证的状态值投影，不得用一个宽泛实体主题替代多个不同字段喵
- 整改建议：把架构门禁从人工挑选的五文件名单改为覆盖全部产品页面和共享展示组件，或建立显式允许清单并对未登记派生 JSX 做负向拒绝喵
- 整改建议：UI 测试应从实际渲染的全部派生值来源反向验证登记完整性，确保删除登记包装、改错主题或改错原始值都会失败喵
- 可验证关闭条件：上述页面不存在未登记的领域派生值；每项值都有精确主题、原始值指纹和可达追因入口；分别删除包装、改错主题、改错值时门禁稳定失败；完整发布门禁重新通过喵
