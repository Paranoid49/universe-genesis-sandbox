## 1. 结论摘要喵

当前实现的确定性、生产因果图认证、随机证据、构建体积和发布纪律均较强，第九轮要求的导出命名空间语法登记、文明与空间页面状态保持以及大量旧页面追因入口也确有实现喵

## 3. 第九轮问题复核喵

第九轮整改确实为概览、空间对象、时间线事件、法则、日志、文明、神话和奇迹结果增加了大量入口，但现有实现仍以整实体或整事件按钮代替字段级可见结果，且自动化只检查已经存在的 `[data-t]` 元素喵

因此，该整改只把入口数量显著扩大，并没有证明 `docs/step-1.md` 第 10 行的“选择任意可见结果”、第 86 行的“所有用户可见派生结果都有对应因果节点”以及 `docs/quality-gates.md` 第 68 行的逐项覆盖要求已经成立喵

## 4. P1 问题喵

### P1-01：页面追因契约仍遗漏字段、效果和观察结果粒度喵

权威要求位于 `docs/milestones.md` 第 87 行、`docs/step-1.md` 第 10、11、86 行以及 `docs/quality-gates.md` 第 68 行，要求每一项用户可见派生结果都能够进入其直接原因、适用规则和后果喵

直接证据如下喵

- `src/components/common.tsx` 第 32 行的 `AttributeBar` 只接收 `label` 与 `value`，没有主题键或追因入口喵
- `src/components/SpaceExplorer.tsx` 第 130 至 171 行展示行星宜居性、魔法饱和、大气、水量、稳定、生物圈魔法适应、文明概率以及文明候选五项属性，但这些值只能借用行星、生物圈或文明候选整实体按钮喵
- `src/components/CivilizationPanel.tsx` 第 114 至 119 行展示科技、魔法、信仰、扩张、稳定和灭绝风险，但没有任何字段级追因入口喵
- `src/components/TimelinePanels.tsx` 第 26 行开始逐项展示事件效果，而 `src/sim/causality-history.ts` 第 83 行已经为每项效果建立独立 `event_effect` 节点，页面却只提供整个事件的追因按钮喵
- `src/components/ObservationConsole.tsx` 第 162 至 165 行展示当前事件、来源和当前层级关联节点，但没有连接现有事件节点或事件关联投影的就地追因入口喵
- `tests/ui/app-interaction.test.tsx` 第 118 行只遍历已经存在的 `[data-t]`，并通过每页最低按钮数量判断覆盖，无法发现完全没有按钮的字段或事件效果喵
- `tests/step-1-causality.test.ts` 第 805 行的 `visibleDerivedSubjects` 由手工列出的实体级主题构成，没有登记上述字段级显示结果，因此测试名称“全部当前可见派生结果”大于实际覆盖范围喵

风险是不同属性可能由不同随机决定、规则和前置状态形成，玩家追溯“为什么文明概率是 42”时只能得到整个生物圈的宽泛原因，无法验证具体数值，也无法从该具体原因查看它已经产生的后果喵

可验证整改条件如下喵

1. 建立统一的可见结果清单或展示契约，使每个实际渲染的派生字段、事件效果、观察关联和状态值都绑定唯一主题，或者绑定一个明确声明所覆盖字段集合的聚合主题喵
2. 为行星、生物圈、文明候选、文明属性和时间线事件效果建立字段级节点与就地入口，禁止用整实体节点掩盖不同字段的原因差异喵
3. UI 测试必须从结果清单生成组件断言，或者反向扫描实际结果标记并要求每项都有主题，不能继续只检查已有 `[data-t]` 喵
4. E2E 至少覆盖一个行星数值、一个文明数值、一个事件效果和一个观察事件关联的进入、唯一定位、直接原因、适用规则和返回闭环喵

### P1-02：聚合统计节点展示的数量没有完整集合因果证据喵

`src/sim/causality-summary-collections.ts` 第 14 行的 `space.stats` 描述同时宣称星系、恒星系、行星、生物圈和文明候选五类数量，但第 16 行的直接原因只有星系集合边界和星系节点喵

在当前边方向中，恒星系是星系的后果，行星是恒星系的后果，生物圈和文明候选又是行星的后果，从 `space.stats` 向原因回溯不会经过这些实际成员或它们各自的集合数量决定，因此无法解释为什么这些数量恰好是当前值喵

同一文件第 28 至 30 行的 `civilization.stats` 还展示神话类型数量，但原因只包含文明节点，没有包含独立的神话节点；现有闭包校验只能证明这些原因存在并可达根，不能证明它们足以支撑节点描述中的每个统计值喵

风险是因果图在结构上合法，却在语义上给出不完整答案，玩家会看到“统计由若干星系导致”，但无法找到恒星系、行星、生物圈或神话集合为何具有当前基数喵

可验证整改条件如下喵

1. 为恒星系、行星、生物圈、文明候选、文明路径、神话类型和高风险判定建立独立集合边界或统计节点，每个统计值只引用实际成员、筛选谓词和集合形成规则喵
2. 如果界面继续使用一个总览聚合节点，该节点必须把上述细粒度统计节点作为直接原因，并在描述中明确每个字段对应的子节点喵
3. 增加语义负向测试，删除任意一个成员、神话节点或筛选证据时，统计节点认证或专用统计校验必须失败，不能只依赖一般根可达检查喵
4. 因果查询测试必须断言从每个统计值进入后可以看到对应集合边界，而不是只断言主题存在且唯一喵

## 5. P2 问题喵

### P2-01：返回上下文对零坐标和非 `data-t` 入口不完整喵

`src/ui/causalView.ts` 第 75 行只有在 `scrollX` 或 `scrollY` 为真值时才执行 `window.scrollTo`，因此来源页位于 `(0, 0)`、用户在因果页向下滚动后点击返回时，不会恢复到顶部喵

同文件第 46 至 48 行只从 `data-t` 获取返回焦点，而第 79 行的 `openCausalProjection` 没有传入通用焦点标识；`src/components/ObservationConsole.tsx` 第 144 行的八个观察追因按钮也没有 `data-t`，所以返回时无法精确恢复到触发按钮喵

现有 `tests/e2e/app.spec.ts` 第 152 行用例只覆盖文明页面的非零滚动位置和空间页面的 `data-t` 入口，不能发现上述边界喵

风险是键盘用户或从页面顶部开始探索的用户返回后会失去位置与焦点，无法连续浏览，且文档声称的“精确焦点与滚动位置”会与真实行为不一致喵

可验证整改条件如下喵

1. 返回时无条件执行 `window.scrollTo(context.scrollX, context.scrollY)`，并增加来源坐标为 `(0, 0)`、因果页滚动后返回的真实浏览器测试喵
2. 返回上下文使用与 `data-t` 无关的通用焦点令牌或元素标识，观察投影和 Seed 对比入口也必须显式保存并恢复触发控件喵
3. Chromium、Firefox、WebKit 和移动端 Chromium 均验证零坐标、非零坐标、重复主题和非主题投影入口四类返回路径喵

### P2-02：AST 认证门禁遗漏 `ImportEqualsDeclaration` 导出别名喵

`tests/architecture.test.ts` 第 283 行的 `exportedSymbolsReaching` 只在顶层登记函数、类、`ModuleDeclaration`、变量、普通命名导出和导出赋值，第 296 行新增的命名空间处理没有覆盖 `ImportEqualsDeclaration` 喵

以下合法 TypeScript 可以把包含认证调用的内部命名空间以运行时别名导出，但当前分析器不会把 `Alias` 识别为可达认证写入能力的导出符号喵

```ts
namespace Unsafe {
  export function certify() {
    return certifyBuiltGraph();
  }
}
export import Alias = Unsafe;
function certifyBuiltGraph() {
  return {};
}
```

咪使用 TypeScript 6.0.3 的 `transpileModule` 独立验证该片段无诊断，并生成 `exports.Alias = Unsafe`，所以这不是只存在于类型空间的理论语法喵

风险是后续维护者可以在不触发架构门禁的情况下公开认证写入能力，破坏“认证只存在于受控私有闭包”的边界喵

可验证整改条件如下喵

1. 递归登记并分析带 `export` 的 `ImportEqualsDeclaration`，增加 `export import Alias = Unsafe`、嵌套别名和多级别名夹具喵
2. 更稳妥的长期方案是使用 TypeScript 类型检查器符号图分析实际导出符号，或者对持有认证写入能力的模块实施严格导出白名单喵
3. 架构门禁文档不得在上述夹具失败前声称“命名空间别名绕行已全部关闭”喵

## 6. 建议项喵

1. 将“可见结果定义、因果主题、显示组件和测试期望”收敛为同一份类型化元数据，避免页面、构建器和测试各自维护人工清单喵
