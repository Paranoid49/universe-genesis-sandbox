## 2. P1-01 整改：可见派生值缺少值级追因登记

### 2.1 根因

旧的结果契约保护范围主要覆盖空间、文明、时间线、概览和观察页面。法则、干预及共享日志组件仍允许多个领域派生值共用宽泛实体主题，架构门禁也没有把这些文件纳入漏登扫描。

### 2.2 实现整改

- 新增 `src/sim/state-value-interface-definitions.ts`，为法则领域、法则、法则关系、干预状态、概率偏移和干预日志字段定义精确状态主题。
- `LawsPage.tsx` 的法则标题、评级、解释、特征、规则名称和值、极性、解释、影响目标、代价及法则关系字段全部接入统一 `ResultValue` 契约。
- `MiraclePanel.tsx` 的反噬等级、概率偏移、指标变化、日志字段、实体变化和反噬事件全部接入统一结果契约。
- `common.tsx` 的 `LogColumn` 为每条观察日志登记精确主题和原始值指纹。
- 因果契约升级为 `ugs-causality@13`，随机结果绑定契约升级为 `ugs-random-bindings@5`；旧领域规则 JSON 未变化，因此规则版本保持 `ugs-ruleset@0.7.0`。
- 架构门禁扩大到 `LawsPage`、`LogsPage`、`MiraclePanel` 和共享组件，并增加法则值、干预值漏登及错误主题的负向夹具。
- UI 测试增加已施加干预后的概率、指标和日志字段真实值登记用例，UI 测试总数由 43 项增加到 44 项。

### 2.3 直接回归及修复

新增 `ResultValue` 后，`.law-tags span` 同时为结果包装层和内部层设置浅色背景，透明背景的追因按钮因此形成金色文字叠加浅黄色背景，对比度只有 1.08，不满足 WCAG 2.1 AA。

`src/styles-features.css` 已为法则标签内部的追因按钮设置不透明深色背景，并保持全局追因按钮及其他页面样式不变。Chromium、Firefox、WebKit 和移动 Chromium 的法则页 axe 扫描均通过。

### 2.4 关闭证据

- 每个评审点名的法则、干预和日志派生值均具有精确主题、原始值指纹及追因入口。
- 删除登记包装、改错主题或引入未登记法则／干预字段时，架构负向夹具会失败。
- 已施加干预后的真实概率、指标和日志字段由 UI 测试与生产事实逐项核对。
- 四浏览器定向无障碍用例 4 项全部通过。
- 完整发布门禁通过。

## 3. P1-02 整改：当前发布事实和 P2 关闭规则冲突

### 3.1 根因

当前文档曾在正文与快照中手工维护多套测试数字，架构测试只抽取第一条快照，不能发现同一文档内的第二套数字。质量门禁中的旧 P2 归零表述也没有与冻结基线同步。

### 3.2 文档与门禁整改

- `docs/step-1.md` 中旧 E2E 50 项表述已改为实际的 54 项通过、2 项跳过。
- `docs/quality-gates.md` 和 `docs/legacy-module-migration.md` 的 P2 规则统一为修复、证据驳回、降级或登记技术债并明确处置，不再要求所有 P2 必须归零。
- 架构测试扫描当前文档中的全部测试数量和体积表述，而不是只读取第一条快照。
- 增加第二套当前数字和冲突 P2 规则的负向夹具，故意插入冲突时测试会失败。
- 历史测试数字只保留在对应历史评审和整改报告中，不再作为当前事实来源。

### 3.3 关闭证据

- 当前文档使用同一次完整发布门禁的唯一快照。
- P2 处置规则与 `docs/step-1-freeze.md` 保持一致。
- 架构测试 9 项通过，并包含当前数字与关闭规则冲突的负向验证。
- 完整发布门禁通过。
