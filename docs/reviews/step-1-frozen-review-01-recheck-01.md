### 3.2 P1-01 仍未关闭的问题喵

严重等级：P1 喵  

违反要求：R3、R8、C1，以及原 P1-01 明确列出的“改错值时门禁稳定失败”关闭条件喵  

涉及位置：`tests/architecture.test.ts:434-451`、`tests/ui/app-interaction.test.tsx:132-155`、`src/components/pages/LawsPage.tsx:31` 喵  

触发或复现方式：把 `src/components/pages/LawsPage.tsx:31` 的可见子内容从 `{rule.value}` 改为 `{rule.value + 1}`，同时保持 `ResultValue` 的 `value={rule.value}` 和 `subjectId` 不变，再运行当前架构测试和 UI 结果登记测试喵  

当前架构扫描在 `hasResultContractAncestor` 返回真时会跳过 `ResultValue` 内部的全部 JSX 表达式，因此不会检查包装内部显示表达式是否仍与 `value` 属性一致喵  

当前 UI 通用断言核对的是 `data-result-value` 与 `stateValueResult`，并核对主题集合是否存在，但没有核对玩家实际看到的文本是否与登记的原始值或规定格式化结果一致喵  

因此上述改动会让玩家看到错误规则值，而追因仍展示正确的生产值和主题，现有两类门禁缺少能够命中该差异的断言喵  

实际影响：展示层发生字段错绑、算术偏移或格式化引用错误时，发布门禁仍可能全绿，玩家看到的“果”与其追因证据不一致，直接破坏 C1 要求的值与主题一致性证据喵  

判断依据：原 P1-01 的关闭条件不仅要求存在包装、主题和指纹，还明确要求分别删除包装、改错主题、改错值时门禁稳定失败；当前只充分覆盖了漏包装和主题／指纹登记，未覆盖包装内部显示值错误喵  

整改建议：增加一个保留正确 `subjectId` 与 `value`、但故意把 `ResultValue` 子内容改错的负向夹具，并让架构或 UI 门禁比较显示语义与登记值喵  

对于 `signed`、标签翻译和列表等格式化字段，应通过受控格式化器或显式预期显示值完成比较，避免把合法格式化误判为原始值错误喵  

### 3.3 P1-01 判定喵

当前生产代码的已检查字段没有发现现实错值，逐值登记和局部无障碍回归均已修复，但原报告规定的错误值负向门禁仍缺失，因此 P1-01 未关闭喵
