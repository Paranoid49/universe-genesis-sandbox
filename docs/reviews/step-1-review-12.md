# 步骤 1 第十二轮严格架构与产品复审报告喵

评审日期：2026-07-14 喵

评审者：步骤 1 第十二轮全新独立评审 agent 喵

评审提示词：作为一个严格系统架构师，对当前项目实现进行打分，涉及角度：易用性，可扩展性，可维护性，边界是否清晰，当前阶段功能是否完整，文档要求是否明确等等喵

## 1. 结论摘要喵

第十一轮的工程整改显著提高了字段入口数量、统计图结构和自动化覆盖，但三项 P1 与一项 P2 尚未达到其报告中给出的可验证关闭条件喵

状态值投影已经展示公式、操作数和一次精确作用域决定，但加权选择仍没有暴露候选快照、逐候选权重、资格判定和情境加成，所谓随机决定节点也只是由 Seed 支撑的文字副本，没有携带随机引用或完整参数证据喵

统一 `ResultValue` 组件解决了部分页面的入口一致性，却没有形成可以穷举所有可见派生值的契约，架构门禁只匹配一份手写正则白名单，而且当前筛选事件数量被错误绑定到总时间线数量主题，自动化仍无法发现值与主题语义不一致喵

路径、神话和高风险统计已经新增分组与判定节点，但专用校验器只检查顶层统计节点，交换分组成员、篡改成员字段、反转纳入／排除判定或改变阈值规则都不会被该校验器发现喵

事实文档虽然增加了统一快照行，但同一正文仍同时保留第十轮和第十一轮互相冲突的测试数量、覆盖率与体积数据，原 P2 文档一致性问题没有真正关闭喵

此外，定向测试首次执行时性能契约以 1512.6385 毫秒超过 1500 毫秒预算而失败，随后三次隔离重跑与完整发布门禁通过，说明非覆盖率性能门禁存在可复现的边缘抖动，且 `check:release` 只在覆盖率模式使用 3000 毫秒预算，不能证明 1500 毫秒日常预算稳定成立喵

本轮结论为 P0 零项、P1 三项、P2 两项、建议项三项，步骤 1 不允许标记完成，也不允许进入步骤 2 喵

## 3. 第十一轮问题复核喵

### P1-01：按需状态值投影缺少值级完整原因和精确规则——未关闭喵

`src/sim/state-value-projection.ts` 第 33 至 40 行把随机决定转换为一个只直接依赖 `input.seed` 的投影证据节点，描述只包含决定 ID、流、作用域、操作、样本和选中值，没有携带候选集、候选顺序、标签、权重或类型化参数，也没有使用图中的随机引用契约喵

`src/sim/state-value-definitions.ts` 第 171 至 173 行把文明路径公式概括为九个文明与法则操作数，而真实算法在 `src/sim/civilizations.ts` 第 134 至 164 行还包含逐路径资格阈值、`civilizationPathProfiles` 档案偏置、`civilizationPathWeight`、`pathSituationBoost` 和筛选后的候选权重集合喵

同类缺口也存在于神话类型，`src/sim/state-value-definitions.ts` 第 178 至 179 行只登记文明数值与一个拼接的起源环境字符串，真实算法在 `src/sim/civilizations.ts` 第 195 至 223 行使用每种神话档案的四类偏置及逐类型环境分支喵

当前测试只在 `tests/step-1-causality.test.ts` 第 812 至 832 行检查一个行星宜居性投影存在五个编号操作数、决定描述包含作用域文字和克隆规范被拒绝，没有对加权选择的候选快照、逐项权重、资格分支、参数变异或公式重算建立负向矩阵喵

这意味着玩家看到的“精确随机决定”仍只是最终抽样摘要，无法验证该决定是否在真实候选集合和真实权重上执行，也无法从候选被排除或权重变化追到本次结果喵

可验证整改条件如下喵

1. 加权选择、候选选择和分支分类的字段投影必须直接复用生成阶段记录的类型化参数与候选快照，不得由展示层重新拼装不完整公式喵
2. 决定证据必须保留操作类型、候选顺序、候选值、标签、权重、原始样本、选中值和精确 `decisionId`，并通过随机引用或等价的受认证绑定连接到生成追踪喵
3. 文明路径、神话类型、星系类型、恒星系类型和行星类型至少各增加资格阈值、档案偏置、候选权重和选中决定的正向重算与逐参数负向用例喵
4. 删除候选、交换权重、改变资格阈值、移动决定到其他流或只保留最终选中值时，字段投影认证必须失败喵

### P1-02：统一可见结果契约无法发现漏登和错误主题映射——未关闭喵

`tests/architecture.test.ts` 第 405 至 412 行只用正则识别恒星系数量、行星数量、少量事件字段、指标字段、影响字段和神话字段，完全不覆盖宇宙名称、Seed、分享码、星系／恒星系／行星／文明名称、空间属性以及未来新增的任意领域字段喵

因此 `src/components/SpaceExplorer.tsx` 第 40 行和第 62 行的宇宙名称与星系名称可以裸渲染，`src/components/pages/OverviewPage.tsx` 第 40 至 41 行的标准 Seed 与分享码也没有通过 `ResultValue` 契约，而架构测试仍保持绿色喵

更严重的是，`src/components/pages/OverviewPage.tsx` 第 50 行把随界面筛选变化的 `filteredTimelineCount` 绑定到表示完整历史总数的 `timeline.count`，同一个主题同时展示两个可能不同的值喵

`tests/ui/app-interaction.test.tsx` 第 118 至 145 行只分别确认按钮主题在图中唯一、状态投影可以构建，以及标记内部存在同字符串入口，没有校验显示值是否与目标节点、状态定义或统计描述一致，因此上述错误映射不会失败喵

这违反 `docs/quality-gates.md` 第 68 行“每一个旧界面可见派生结果都有唯一可查询因果节点或明确一对多历史节点”的要求，也无法证明完全未登记的值会被自动化发现喵

可验证整改条件如下喵

1. 建立类型化结果描述符或由领域视图模型生成的统一渲染清单，使受保护页面不能直接裸渲染登记范围内的领域值喵
2. 结果描述符必须同时绑定取值函数、主题、策略和期望值指纹，测试必须核对实际显示值与因果节点或按需状态定义的值一致喵
3. 界面筛选数量必须使用明确的界面状态主题或不声明为宇宙事实，禁止借用 `timeline.count` 等数值不同的生产主题喵
4. 对裸宇宙名称、裸对象名称、裸 Seed、裸分享码、错误主题复用和新增未知字段分别增加会失败的架构或组件负向夹具喵

### P1-03：统计语义校验不能证明分组、字段和筛选判定正确——未关闭喵

`src/sim/causality-summary-group-evidence.ts` 第 12 至 22 行确实创建了字段、成员和分组节点，第 27 至 38 行也为每个文明创建了高风险纳入／排除节点喵

但是 `src/sim/causality-summary-validation.ts` 第 6 至 24 行只读取顶层统计节点的直接原因集合与描述，第 70 至 84 行只期望分组节点 ID和判定节点 ID存在，从未检查分组节点实际包含哪些成员、成员是否指向正确字段、字段值是否等于领域值、判定正负是否符合阈值或阈值规则是否仍为 65 喵

`src/sim/causality-summary-group-evidence.ts` 第 35 至 36 行把阈值 65 写在自由文本中并只引用通用 `axiom:summary-filtering`，没有独立、可版本化、可查询的谓词节点喵

`tests/step-1-causality.test.ts` 第 835 至 858 行只覆盖顶层缺少原因、顶层描述变化、顶层额外原因以及成员描述正则存在，没有覆盖交换路径分组成员、交换神话分组成员、篡改字段值、反转纳入／排除、改变阈值和断开字段来源喵

因此当前统计图可以在顶层数量和原因 ID完全不变的情况下，让玩家沿原因路径看到错误的成员归组或筛选结论，专用语义校验仍返回空问题数组喵

可验证整改条件如下喵

1. `summaryCollectionEvidenceIssues` 必须递归校验字段节点、成员节点、分组节点、筛选判定节点和顶层统计节点的精确语义，不只比较顶层原因集合喵
2. 分组统计必须校验每个领域实体恰好进入其真实字段值对应的一个分组，禁止缺失、重复、错组和跨类别成员喵
3. 高风险统计必须使用独立的版本化谓词节点，逐项校验字段值、运算符、阈值、纳入／排除结果以及正负成员完整集合喵
4. 增加额外原因、错组、错误字段值、阈值 65 改写、正负成员互换、成员重复和字段来源缺失的完整负向矩阵喵

## 4. P2 问题喵

### P2-01：当前事实文档仍同时陈述互相冲突的发布事实喵

`docs/step-1.md` 第 126 至 129 行仍声明 E2E 50 项、旧覆盖率和 JavaScript gzip 79576 字节，而第 132 行又声明 E2E 54 项、新覆盖率和 JavaScript gzip 84833 字节喵

`docs/non-functional-requirements.md` 第 16 行仍把当前完整门禁体积写为 79576 字节，第 18 行则写为 84833 字节喵

`docs/quality-gates.md` 第 125 行把第十轮 79576 字节写入当前体积表，第 129、133 和 135 行又并列保留第十轮与第十一轮快照，正文没有把历史记录和当前唯一事实明确分区喵

架构测试只抽取每份文档中一条 `当前发布快照` 和一条 `当前定向测试组成`，所以无法发现同一文档其他正文仍有冲突数字喵

可验证整改条件如下喵

1. 当前事实章节只能保留一组当前数据，历史快照必须移入明确标注的历史区或评审归档，不得继续使用“当前”措辞喵
2. 文档一致性门禁必须解析所有带测试数量、覆盖率、E2E 和体积语义的当前陈述并与唯一机器可读快照比较喵
3. 人为修改任一正文数字或同时写入两组当前快照时，门禁必须失败喵

### P2-02：1500 毫秒性能契约在定向套件中出现边缘失败，完整发布门禁不能覆盖该预算喵

本轮首次执行 `npx vitest run tests/step-1-causality.test.ts tests/causality-factory-guards.test.ts tests/architecture.test.ts` 时，31 项中 30 项通过、1 项失败，`tests/step-1-causality.test.ts` 第 653 行记录实际 1512.6385 毫秒超过 1500 毫秒预算喵

随后将该性能用例隔离重跑三次均通过，完整 `npm run check:release` 也通过，表明问题是边缘抖动而不是稳定超时喵

但第 652 行在覆盖率生命周期下使用 3000 毫秒预算，而 `check:release` 只运行覆盖率模拟测试，没有执行非覆盖率的 1500 毫秒契约，所以完整门禁绿灯不能证明日常预算稳定满足喵

可验证整改条件如下喵

1. 发布门禁必须显式执行非覆盖率性能契约，或提供独立稳定的性能脚本与采样策略，不能只依赖覆盖率模式的双倍预算喵
2. 使用预热、多个样本的中位数／高分位或分离生成与查询阶段等方式降低单次调度噪声，同时不得直接放宽既定预算规避失败喵
3. 在普通环境连续执行约定次数后应稳定通过，并在文档中记录采样方法、机器无关假设和失败判定喵

## 5. 建议项喵

1. 将生成规则的公式元数据、字段取值函数、候选参数记录和投影说明收敛到同一类型化定义，避免 `galaxies.ts`、`civilizations.ts` 与 `state-value-definitions.ts` 三方手工镜像喵
2. 为因果投影建立来源宇宙身份绑定，`buildStateValueCausalProjection` 除检查主题存在外还应验证传入领域对象与受认证图属于同一生成身份，避免未来调用方把合法图与伪造读取模型拼接喵
3. 将发布快照生成成单一机器可读工件，再由文档引用或自动生成，减少每轮手工复制覆盖率与体积数据造成的漂移喵
