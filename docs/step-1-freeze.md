# 步骤 1 冻结验收口径

冻结日期：2026-07-14  
适用范围：步骤 1“可追因的宇宙生成器”  
唯一需求来源：`docs/milestones.md` 第 4 节及其共同交付规则

本文件只把既有里程碑要求整理为可执行检查项，不新增功能、不提高既有完成条件，也不允许后续评审改变验收口径。

## 1. 冻结需求

| 编号 | 冻结需求 | 来源 |
| --- | --- | --- |

| R2 | 宇宙公理、初始状态、外部输入、状态变化、事件和观察结果进入同一因果模型。 | 里程碑步骤 1 实现重点 |

| R4 | 因果校验必须拒绝孤立结果、丢失根因、无规则支撑的变化和未授权因果循环。 | 里程碑步骤 1 实现重点 |

| R11 | 当前架构、质量门禁和非功能要求必须与实际步骤 1 实现一致。 | 共同交付规则与步骤 1 实现重点 |

## 2. 冻结非目标

以下事项明确不属于步骤 1，评审不得将其作为 P0、P1 或 P2 提出：

- 不实现模拟时钟、可继续状态转换、持续演化和运行分支存档，这些属于步骤 2。
- 不重构认知边界、未知状态、研究记录簿和自由观察路径，这些属于步骤 3。
- 不实现不可变历史分支、共同祖先实验和新分享包，这些属于步骤 4。
- 不替换固定模板、固定法则领域、固定空间拓扑和固定观察通道，这些属于步骤 5。
- 不实现通用自主实体、认知、意图和行动协议，这些属于步骤 6。
- 不以更优雅的架构、未来扩展偏好、理论攻击面或低收益重构作为阻塞条件。

## 4. 评审与问题管理规则

- 首次完整评审只依据本文件，不得提出步骤 2 至步骤 6 的要求。
- 首次完整评审后的复查只检查未关闭问题、本轮修改和直接回归，不得重新开放式扫描整个项目。
- 已关闭问题只有出现违反同一冻结条件的新证据时才能重新打开。
- 同一根因的字段、语法和案例必须合并为一个问题。
- 每轮问题必须引用 R 或 C 编号，并提供位置、复现方式和现实影响；无法引用者只能作为建议项。
- 步骤 1 最多进行三轮整改复查，仍有争议时启动只裁决现有问题的裁决 Agent。
- 有效 P0、P1 必须关闭；P2 必须修复、驳回、降级或登记技术债；建议项不阻塞完成。

## 5. 冻结兼容边界、质量要求与必要威胁模型

本节只显式整理 R1 至 R11 与 C1 至 C11 已经包含的边界，不增加新的完成条件。

- 接口与数据边界：`generateUniverse`、旧 `UniverseSummary`、UGS070 分享、I1 干预载荷、A1 图书馆和主要旧页面在步骤 1 继续兼容；新增因果事实不得只存在于旧聚合对象，因果图不得被旧 JSON、分享或 A1 载荷隐式携带。
- 版本边界：规则版本与旧领域 JSON 保持兼容；因果图、随机绑定和输入清单使用独立版本，证据契约变化必须升级对应版本并更新内容哈希基线。
- 质量边界：必须通过类型检查、静态检查、模拟与 UI 覆盖率、真实浏览器、构建、体积、文件规模、无障碍、文档一致性和独立性能门禁。
- 需要防御的威胁：伪造或污染因果图、输入清单、随机流、抽样参数、候选顺序、候选权重、候选资格、结果绑定、统计成员、统计字段、筛选谓词、可见结果主题和值指纹，以及损坏分享或存档输入造成的静默等价恢复。
- 明确不属于本步骤的威胁：远程账户、多人协作、云端权限、网络服务攻击、长期运行分支存档和后续规则包执行沙箱；这些能力尚未进入步骤 1 产品范围。

## 6. 历史评审处理

步骤 1 冻结前产生的评审报告保留为历史证据，但不再拥有新增验收条件的效力。主 Agent 只处理能够回链到本文件 R／C 条目的问题。冻结后的下一名独立 Agent 使用老大指定的新提示词执行首次完整评审，之后最多进行三轮有限复查。
