# 步骤 1：可追因的宇宙生成器

状态：已完成。冻结后首次完整评审的两项 P1 已经两轮有限复查全部关闭，最终 P0、P1、P2 和建议项均为零，完整发布门禁通过。

冻结验收口径见 [step-1-freeze.md](step-1-freeze.md)，第十二轮问题裁定见 [step-1-review-12-adjudication.md](reviews/step-1-review-12-adjudication.md)。

本步骤把当前静态宇宙生成结果纳入统一因果模型，同时保持现有创世、浏览、分享和本地存档路径可运行。本步骤不提前实现持续时间推进、自由观察、正式分支、多本体规则包或自主实体运行时。

## 1. 用户闭环

1. 玩家使用当前 Seed 与模板生成宇宙。
2. 玩家进入因果查询界面并选择任意可见结果。
3. 界面展示该结果的直接原因、适用规则和向前产生的后果。
4. 玩家可以沿原因继续回溯，直到宇宙公理、初始状态或显式外部输入。
5. 同一显式输入再次生成时，宇宙结果和完整因果图完全一致。

## 2. 因果契约

因果模型必须至少包含：

- 因果节点，用于表示根输入、规则、状态、确定性选择、事件、结果和观察投影。
- 因果边，用于表示一个节点对另一个节点的直接作用关系。
- 合法根集合，只允许宇宙公理、初始状态和显式外部输入成为无原因节点。
- 规则引用，用于说明一次变化由哪些宇宙规则允许。
- 目标引用，用于把因果节点映射回指标、事件、空间对象、实体或展示结果。
- 随机元数据，用于记录 PRNG 算法版本、随机流身份，以及每次决定唯一对应的 `decisionId`、单个抽样序号、候选集和选中值。
- 类型化随机调用转录，用于记录实际执行的操作、范围、概率、候选顺序、权重、原始样本和选中值。
- 随机结果绑定，用于把每项决定绑定到结果节点、结果主题、节点类型、携带真实容器语义的类型化领域定位器、领域值指纹、作用域以及字段／集合／负事实语义。
- 生成清单，用于把 Seed、规则版本、创世模板和每一项有序干预绑定到唯一输入根与同一次随机追踪身份。
- 跨图组合证据，用于同时绑定 Seed 对比的左右宇宙身份、左右来源节点、两侧评分、差值和最大差异裁决。

完备原因由宇宙公理、变化前状态、模拟与规则版本、随机流状态和有序显式输入组成。完备原因相同时必须产生相同结果；相同结果允许来自不同原因，因果历史不得因此合并。

## 3. 覆盖范围

当前用户可见的以下结果必须进入因果图：

- Seed、模板和规则版本等根输入。
- 结构化法则、法则领域评分和法则关系。
- 所有宇宙指标及其影响来源。
- 时间线事件、触发事件、事件效果和事件来源。
- 时间线影响摘要的兼容投影。
- 星系、恒星系、行星和生物圈。
- 文明、文明历史和神话档案。
- 玩家干预、干预结果、目标变化和反噬事件。
- 宇宙名称、摘要、解释、分享结果和观察日志等展示投影。

旧字段可以继续存在，但不得成为无法追溯的事实孤岛。

## 4. 闭包校验

校验器必须拒绝：

- 重复节点或重复边。
- 指向不存在节点的原因、后果或规则引用。
- 非法无原因节点。
- 无法从合法根递归到达的派生节点。
- 未经宇宙公理许可的因果循环。
- 只把随机数写成原因，却没有规则与前置状态的结果。
- 输入根与规范化显式输入清单之间存在额外、缺失、重排、类型或值错配的图。
- 随机决策不属于引用声明的随机流，或随机追踪与生成清单不属于同一次生成。
- 随机引用与结果绑定不一致，或完整随机调用转录与受控重放结果不一致。

允许的因果闭环必须显式引用使其成立的高阶公理和一致性约束，并精确登记闭环成员、闭环边、高阶公理和约束集合。闭环成员必须与登记完全一致，闭环不得包含合法根；任何未授权循环仍会被拒绝。

## 5. 展示要求

- 因果查询必须提供“为什么发生”和“产生了什么后果”两个方向。
- 所有查询必须有文字路径和键盘操作能力。
- 节点标签必须使用中文领域语言，不直接暴露内部枚举。
- 空原因、空后果和无法找到目标时必须提供明确说明。
- 解释文案只能由因果事实生成，不得作为新的根事实写回因果图。
- 观察台对摘要、几何、生命、文明、魔法、神性、因果和事件关联八类结果分别提供单项追因；强度数值与来源主题必须由同一份度量定义产生，每次操作只追加并聚焦一个观察投影节点。
- Seed 法则对比必须显式展示“右值 - 左值”公式，并先用组合证据同时校验左右宇宙身份、来源、评分和最大差异裁决，再在各自独立的因果图中展示本侧依据，禁止合并两侧随机轨迹或因果历史。

## 6. 兼容边界

- `UniverseSummary` 在本步骤中保留为旧界面兼容读取模型；因果图以非枚举、冻结的只读值挂载，不能被展示层改写。
- `generateUniverse` 保留兼容惰性边界；生产显式 API `generateCausalUniverse` 必须在返回前物化因果图、执行完整闭包校验，并拒绝重放不一致或无效图。
- 低层因果装配器不从公共 `src/sim` API 导出；受控工厂根据显式输入自行重新执行完整生成，认证写入能力仅存在于模块私有闭包中，普通模块和测试夹具不能导入或调用。
- 当前主要页面、分享码、本地 A1 图书馆和导入导出保持可运行。
- 本步骤新增因果数据不得改变相同输入下既有领域字段的值。
- 模拟核心内容变化必须同步更新内容哈希基线；领域结果变化时升级规则版本与短码，仅因果证据契约变化时升级因果契约版本。
- 旧阶段固定数量测试只属于兼容形状测试，不再代表新产品永久要求。

## 7. 自动化验收

- 所有用户可见派生结果都有对应因果节点。
- 所有非根节点至少有一个直接原因。
- 每个派生节点都能递归到达合法根。
- 每条边的原因与后果节点都存在，双向查询结果一致。
- 同一输入两次生成得到完全相同的因果图。
- PRNG 算法版本、随机流身份和抽样序号可见且稳定；每个决定引用精确对应一次随机抽样，不用流级范围冒充决定证据。
- 不同原因可以产生相同结果而不合并历史节点。
- 孤立节点、悬空引用和非法循环会被校验器拒绝。
- 因果查询组件通过交互、键盘与无障碍测试。
- 完整 `npm run check` 与步骤 1 独立严格架构评审通过。
