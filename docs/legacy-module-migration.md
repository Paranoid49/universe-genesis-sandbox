# 旧产品模块迁移矩阵

版本：1.0  
状态：产品重构实施路线的强制组成部分

本文件逐项规定当前旧产品模块在新产品路线中的最终处理方式。任何模块都必须明确归入保留、保留并增强、降级、替换或删除之一，不允许以“后续再看”作为阶段完成条件。

## 1. 处理方式定义

- 保留：现有职责符合新产品定义，只允许进行兼容性和工程质量调整。
- 保留并增强：保留核心职责，但必须补充新产品要求的数据、接口或行为。
- 降级：模块不再充当事实来源、核心流程或一级入口，只作为展示、兼容或条件视图存在。
- 替换：旧模块在负责步骤完成后退出新产品主流程，由新的状态、规则或交互模块承担职责。
- 删除：旧行为不进入新运行时；如需读取历史数据，只允许存在于隔离的只读兼容路径中。

每个步骤完成时，负责该步骤的矩阵条目必须具备实现、迁移、测试和文档四类证据。未完成的条目不得通过修改状态名称、保留隐藏入口或降低测试门槛规避验收。

## 6. 分步骤关闭范围

- 步骤 1 关闭因果契约、解释来源、旧测试分层、当前架构与门禁换轨，并把 `UniverseSummary` 降为兼容读取模型。
- 步骤 2 关闭一次性生成器、预制时间线、`timelineImpact`、固定状态来源、不可继续存档和旧时间浏览问题。
- 步骤 3 关闭旧 Overview、Logs、九页后台式导航、全知观察和固定一级对象入口问题。
- 步骤 4 关闭单一分享码、无分支身份、可清空历史、实验与干预混同以及旧 Seed 对比问题。
- 步骤 5 关闭固定模板、固定法则领域、固定指标、固定空间拓扑、固定观察叠层和固定奇迹清单问题。
- 步骤 6 关闭即时生命种子、预制文明历史、预赋终局以及神话叙述与真实实体混同问题。
- 步骤 7 关闭 A1 作为运行存档、`localStorage` 大型状态存储、current-only 冒充恢复和单一 JSON 文本框导出问题。
