# 步骤 3 冻结验收口径：自由探索的观察台

状态：已冻结，实施中，尚未完成验收

冻结日期：2026-07-14

唯一上位产品事实：`docs/product-definition.md`

实施顺序：`docs/milestones.md`

旧模块去留：`docs/legacy-module-migration.md`

本文件是步骤 3 实现、测试、文档和独立评审的唯一验收基线。冻结后的实现与评审不得增加需求、提高完成标准或评价步骤 4 及后续能力。确实影响范围的歧义由主执行 Agent 依据当前事实来源裁定；只有会显著改变产品范围时才请求用户裁决。

## 1. 工作单元目标

把步骤 2 的“可以观看持续演化状态与历史”迁移为“玩家围绕自己选择的问题，通过有限观察逐步积累证据”的自由探索产品。

步骤 3 的产品闭环是：玩家进入当前宇宙，自主选择对象、时间、尺度、现象或观察方式，获得带来源和不确定性的可观察信号，将感兴趣内容加入关注，记录笔记或推测，并从观察历史继续自己的研究。

步骤 3 不规定玩家必须寻找生命、文明、神明、异常或任何特定答案，也不提供胜利条件、强制任务或唯一正确探索顺序。

## 2. 当前必须实现的要求

### R1：四层认知边界

- 明确分离宇宙事实、可观察信号、内部实体认知和玩家推测。
- 新观察界面只能直接展示玩家当前可获得的信号、证据和明确公开的创世公理，不得绕过观察系统平铺底层 `UniverseState`。
- 内部实体认知在步骤 3 只建立兼容边界，不要求实现步骤 6 的自主实体；没有内部认知数据时不得生成占位信念。
- 玩家笔记和推测属于玩家研究数据，不得自动升级为宇宙事实、规则或因果节点。

### R2：可追溯的观察结果

- 每个观察结果必须记录观察对象、观察方式、观察时刻或时间范围、来源事实或已提交历史、证据强度和稳定身份。
- 同一宇宙状态、观察方式和显式观察参数必须产生相同观察结果。
- 观察结果可以不完整、模糊或互相冲突，但不能凭空增加底层不存在的事实。
- 玩家能够从信号查看其证据来源；能够公开追因的内容继续复用步骤 1、2 的因果查询，不得建立绕过因果闭包的解释来源。

### R3：自由观察入口

- 提供按对象、尺度、时间和现象组织的观察入口，玩家可以从其中任意一种方式开始。
- 至少提供多条互不依赖的观察路径，不能要求先发现某类对象或完成固定顺序才能使用其余通用观察能力。
- 观察入口由当前宇宙实际存在的对象、历史和当前步骤支持的观察方式决定，不得把星系、生命或文明设为通用入口前置条件。
- 步骤 3 可以继续使用传统空间兼容宇宙证明观察闭环，但观察契约和通用导航不能绑定传统空间层级。

### R4：未知、证据强度与解释冲突

- 未观察、证据不足、已观察但无结果和确认不存在必须是不同状态。
- 证据强度必须有稳定、可测试的计算或分级来源，不能由展示文案临时决定。
- 多个观察结果对同一问题给出不同解释时，必须保留各自证据和冲突关系，不得静默选择产品偏好的答案。
- 产品可以提示变化或异常作为可选线索，但不得把线索包装成必须完成的任务。

### R5：研究记录簿

- 玩家可以关注对象或现象、添加标签、记录笔记、保存推测并查看观察历史。
- 每条记录必须绑定宇宙定义和当前运行分支身份；没有步骤 4 分支时使用当前唯一运行历史身份，不能与其他宇宙混用。
- 笔记和推测可以引用观察证据，但修改文字不得改写宇宙状态、随机流、输入日志或因果历史。
- 记录簿保存失败必须明确提示并保持先前有效数据，不得显示虚假的保存成功。

### R6：不预设玩家目标

- 删除系统自动判定“重要事件”“稀有发现”“潜在终局”和默认探索终点的产品行为。
- 产品不得根据生命、文明、神明或任何特定内容是否出现来判断一次探索是否成功。
- 空无、长期稳定、没有目标对象或没有显著变化必须能够成为合法观察结果。
- 界面文案不得暗示玩家应当追求繁荣、毁灭、惊奇、生命或叙事高潮。

### R7：主导航与页面层级换轨

- 新主导航围绕当前宇宙、观察、研究记录、已发生历史和存档组织，并为步骤 4 的实验与分支保留清晰扩展位置，但不提前实现它们。
- 星系、文明、神话、奇迹和固定纪元不得作为永久一级栏目。
- 对象类型页面只在当前宇宙存在对应对象并且玩家拥有相应观察入口时作为上下文视图出现。
- 旧九页后台式导航只能留在明确隔离的旧版兼容视图中。

### R8：当前宇宙概览替换

- 旧 `OverviewPage` 退出新主流程，由当前可知状态、近期可观察变化、未知区域和玩家关注内容组成的新概览接管。
- 未经观察的对象数量、文明数量、神话数量、底层指标和完整法则不得直接展示。
- 概览中的每个领域结论必须来自观察结果、公开公理或已记录的玩家研究数据。

### R9：旧日志页退出

- 旧 `LogsPage` 及 `observationLog` 自动挑选重要事件、稀有发现和潜在终局的行为不得进入新主流程。
- 新研究记录簿只能记录实际观察证据、明确内部视角、玩家笔记和玩家推测。
- 旧日志只允许在隔离兼容视图中读取旧静态宇宙内容，不得成为新运行事实或研究数据来源。

### R10：法则与指标的认知呈现

- 区分玩家设定或明确公开的宇宙公理、由观察证据支持的规律、内部观点和未知规则。
- 步骤 3 不实现步骤 5 的动态规则包，但法则观察接口不得假设固定六领域是永久结构。
- 指标只能作为观察结果或当前可知状态的派生投影，不得绕过认知边界读取并展示底层固定七指标。
- 旧独立 Seed 对比只保留在兼容视图；共同祖先分支比较由步骤 4 实现。

### R11：条件对象视图

- `SpaceExplorer` 降为传统空间宇宙的条件视图，通用观察流程不得依赖“星系—恒星系—行星”层级。
- `CivilizationPanel` 在新主流程中不作为永久入口；完整自主实体视图由步骤 6 实现。
- 没有对应对象、没有可用观察方式或尚未获得相关信号时，不显示空白对象栏目或泄露底层存在性。
- 条件视图消费观察上下文和可见对象引用，不得直接接收完整旧 `UniverseSummary`。

### R12：兼容、持久化与主流程隔离

- 步骤 1 因果查询、步骤 2 运行控制、确定性状态、历史浏览、运行检查点和旧只读兼容能力不得静默损坏。
- 研究记录使用版本化存储契约，并与运行状态存档分离；研究数据损坏、未知版本或跨宇宙身份错配必须明确拒绝。
- 旧 A1、UGS070、I1 和旧页面继续只在隔离兼容视图中读取，不得自动升级为等价观察历史或研究记录。
- 新主流程不得依赖 `useUniverseAppModel`、完整 `UniverseSummary`、旧 `selectors.ts`、`observationLog` 或固定九页导航提供事实。

## 3. 明确不实现的内容

- 不实现步骤 4 的不可变分支身份、共同祖先比较、回到过去修改、界外实验、新干预协议或正式分享包。
- 不实现步骤 5 的可组合多本体规则包、动态空间拓扑、动态事件类别、动态指标集合或动态干预能力。
- 不实现步骤 6 的自主实体感知、记忆、信念、意图和行动，也不要求生命或文明出现。
- 不实现步骤 7 的长期创作档案、结构化编年史或正式三类文件导出。
- 不增加固定任务、成就、胜利条件、探索评分、重要性评分或默认推荐终点。
- 不把观察结果、玩家笔记或推测写入宇宙状态，除非未来步骤中对应宇宙规则明确规定观察会产生作用；该作用协议属于步骤 4、5。
- 不允许通过观察界面修改过去状态或清除已经发生的历史。
- 不增加新的固定事件池、生命配额、文明配额、神话配额或内容文案规模来冒充探索深度。

## 4. 可验证完成条件

### C1：自由探索闭环

玩家可以从当前宇宙选择一种观察入口，获得信号和证据，关注对象或现象，记录笔记或推测，离开后重新打开并继续研究。

### C2：认知边界

自动化负向门禁证明未经观察的底层事实、对象数量、完整法则和固定指标不能通过新主流程、选择器、缓存或页面属性泄露。

### C3：多路径探索

至少存在多条互不依赖的观察路径，玩家忽略生命、文明和任何预设主题仍能完成一次证据积累过程。

### C4：证据来源与确定性

相同宇宙状态、观察方式和显式参数产生完全一致的观察结果、证据强度、来源引用和结果身份；每个结果都能定位到实际状态或已提交历史。

### C5：未知与冲突

未观察、证据不足、无结果和确认不存在具有可区分状态；至少一个自动化场景证明冲突解释可以同时保存且不丢失来源。

### C6：研究记录持久化

关注、标签、笔记、推测和观察历史能够保存恢复；损坏版本、跨宇宙拼接和写入失败明确拒绝并保留先前有效记录。

### C7：无预设目标

新主流程不存在系统生成的重要性、稀有度、潜在终局、任务完成度或生命文明默认终点；空无和稳定状态可以形成合法观察记录。

### C8：新导航与概览

新主导航和概览围绕当前宇宙、观察、研究记录、已发生历史和存档组织，不直接显示未经观察的底层统计，旧九页导航退出新主流程。

### C9：法则认知分层

公开公理、观察支持的规律、内部观点占位边界和未知规则在数据与界面上可区分，旧固定法则平铺和独立 Seed 比较不进入新主流程。

### C10：条件对象视图

传统空间对象可以通过观察上下文进入条件视图；没有对应对象或观察能力时不显示空白星系、文明或神话入口，也不泄露其底层存在性。

### C11：运行状态不受浏览影响

纯观察、浏览、关注、笔记和推测不会改变 `UniverseState` 身份、逻辑时钟、输入日志、随机流、转换历史或因果网络。

### C12：兼容与隔离

步骤 1、2 永久不变量和只读兼容路径通过回归；新观察主流程不存在对旧完整生成器、`UniverseSummary`、旧选择器、`observationLog` 或固定九页应用模型的事实依赖。

### C13：纵向浏览器验收

真实浏览器至少覆盖一次“进入当前宇宙—选择观察方式—获得证据—查看来源—关注—记录笔记或推测—刷新—恢复研究”的完整路径，并覆盖键盘、窄屏、axe 和错误提示。

### C14：迁移矩阵关闭

`docs/legacy-module-migration.md` 中归属于步骤 3 的概览、日志、导航、观察、法则、条件对象视图、选择器、解释和研究记录职责均具有实现、迁移、测试、文档与兼容证据。

### C15：评审与问题处置

完成实现与完整发布门禁后，由未参与步骤 3 实现的全新独立 Agent 依据本冻结基线执行首次完整评审。有效 P0、P1 必须关闭，P2 必须修复、以证据驳回、降级或登记技术债并明确处置，建议项不阻塞完成。

## 5. 接口、数据与兼容边界

### 5.1 新公共契约

- `ObservationContext`：宇宙定义、当前运行身份、逻辑观察位置、玩家可用观察方式和认知边界。
- `ObservationMethod`：观察方式版本、适用对象或现象、显式参数、可见范围和结果投影规则。
- `ObservableSignal`：稳定身份、观察对象、观察方法、时间范围、可见值、不确定性和证据引用。
- `EvidenceRecord`：证据身份、来源事实或历史引用、证据强度、支持或反对的解释及可追因入口。
- `KnowledgeStatus`：未观察、证据不足、部分支持、冲突、确认与确认不存在等可区分状态。
- `PlayerFocus`：玩家关注的对象、现象或问题及标签。
- `PlayerNote`：玩家文字记录、创建和更新时间、引用证据与宇宙身份。
- `PlayerHypothesis`：玩家推测、状态、支持与反对证据引用，不自动成为宇宙事实。
- `ObservationHistoryEntry`：实际执行的观察参数、结果身份和时间顺序。
- `ResearchNotebook`：关注、笔记、推测和观察历史的版本化聚合，但不承载宇宙事实。
- `ResearchStorageAdapter`：版本、迁移、事务写入、读取、列举、删除和失败恢复接口。

具体类型名可以调整，但宇宙事实、观察结果、内部认知和玩家研究数据不得重新合并进单一聚合对象。

### 5.2 版本与身份边界

- 观察协议、证据记录、研究记录和研究存储必须拥有独立版本。
- 观察结果身份必须绑定宇宙定义、当前运行历史身份、观察方法版本、逻辑时间和显式参数。
- 玩家研究记录必须绑定宇宙定义和运行历史身份；状态相似或 Seed 相同不得自动合并记录。
- 改变可见范围、证据强度语义、观察结果身份或研究存档解释时，必须评估并更新对应版本。

### 5.3 已完成步骤不变量

- 步骤 1 的因果闭包、双向查询、规则来源、确定性随机证据和同果异因历史保持有效。
- 步骤 2 的 `UniverseState` 唯一事实来源、逻辑时钟、完整差异、运行因果、确定性恢复和存档语义校验保持有效。
- 模拟核心继续禁止读取 React、浏览器状态、系统时间、网络、本地存储或玩家笔记。
- 观察和研究数据对外只读投影不得获得修改宇宙状态的引用。

## 6. 质量要求

### 6.1 必须执行的门禁

- `npm run lint`。
- 应用与测试 TypeScript 类型检查。
- 步骤 1、2 永久不变量与兼容回归测试。
- 步骤 3 观察边界、信号、证据、未知状态、冲突、记录簿、存储和主流程隔离测试。
- UI 组件测试、真实浏览器 E2E、纯键盘路径、窄屏和 axe。
- 生产构建、体积检查和独立非覆盖率性能测试。
- 最终完整入口仍为 `npm run check:release`，并新增步骤 3 专属门禁，不得绕过步骤 1、2 检查。

### 6.2 覆盖率

- 模拟与观察核心全局语句、函数和行覆盖率不低于 90%，分支不低于 82%。
- UI 全局语句、函数和行覆盖率不低于 75%，分支不低于 70%。
- 观察投影、证据强度、研究存档校验和研究存储适配器必须设置独立文件阈值。
- 每个公开研究存档拒绝原因、知识状态转换和身份错配必须有直接负向测试。

### 6.3 性能与响应预算

在两次预热和至少十二个代表性多模板样本下执行非覆盖率测试：

- 单次观察投影 P50 小于 50 毫秒，P90 小于 100 毫秒。
- 一百条观察历史重建 P50 小于 500 毫秒，P90 小于 1000 毫秒。
- 研究记录保存 P90 小于 500 毫秒，恢复并完成完整性校验 P90 小于 1000 毫秒。
- 观察筛选、关注和笔记输入的界面反馈在普通桌面环境中 100 毫秒内可见。
- 性能优化不得改变观察结果、证据强度、来源引用或研究记录内容。

### 6.4 可靠性与资源

- 观察任务要么完整产生带来源的结果，要么明确失败，不留下半条观察历史。
- 研究记录写入要么完整提交，要么保留先前有效版本。
- 页面切换、组件卸载、错误和测试结束必须取消观察任务、计时器、工作线程和待处理存储任务。
- 测试结束必须关闭服务并删除 `coverage/`、`dist/`、`playwright-report/` 与 `test-results/`，确认 4173／5173 和项目 Node 进程无残留。
- 手动启动命令为 `npm run dev`，停止方式为同一可见终端按 `Ctrl + C`。

### 6.5 UI 运行时与体积

- 继续使用当前已验证的 Preact React-compat 边界，步骤 3 开始实现前复核其对记录簿表单、观察筛选和错误边界的适用性。
- JavaScript gzip 硬上限保持 115 KiB、保护线保持 110 KiB；CSS gzip 硬上限保持 10 KiB、保护线保持 8 KiB。
- 若必要能力突破保护线，必须通过正式架构决策说明原因、备选方案和新预算，不得直接放宽脚本。

## 7. 必要威胁模型

| 威胁 | 必须具备的防御与验证 |
| --- | --- |
| 全知事实泄露 | 新主流程只接收观察上下文与可见结果；依赖扫描和负向 UI 测试拒绝直接读取完整状态、旧选择器和 `UniverseSummary` |

## 8. 冻结后的评审规则

首次完整评审只能评价本文件 R1 至 R12、C1 至 C15、接口与兼容边界、质量要求和必要威胁模型，以及步骤 3 修改对步骤 1、2 已完成功能造成的直接回归。
