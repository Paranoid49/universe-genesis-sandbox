# 产品重构实施路线

本文件描述项目从当前“结构化结果与事件浏览器”迁移到目标产品的实施顺序。当前产品定义以 [product-definition.md](product-definition.md) 为唯一事实来源，所有旧模块的处理方式以 [旧产品模块迁移矩阵](legacy-module-migration.md) 为强制验收清单，原阶段 0 至阶段 8 文档只作为既有实现和历史决策的证据，不再决定后续产品方向。

[旧版完整里程碑](archive/legacy-phases/milestones_old.md) 保存此前的完整里程碑原文，不在本路线中修改。

新路线中的每个步骤必须同时关闭迁移矩阵中归属于该步骤的条目。任何旧模块如果没有明确的保留、增强、降级、替换或删除结论，视为该步骤规格不完整。

## 1. 当前基线

当前版本已经具备确定性 Seed、结构化法则、时间线、空间对象、文明与神话样本、干预、本地存档、分享和二维观察台等工程能力。

这些能力构成可复用的工程基线，但当前核心体验仍以生成完成后的结果浏览为主，尚未形成持续演化、自主探索和因果推理闭环。因此，原阶段 0 至阶段 8 的“完成”只代表对应工程规格完成，不代表目标产品已经成立。

## 2. 所有步骤共同遵守的交付规则

以下步骤必须按顺序交付，每一步都必须产生一款能够独立启动、操作、保存结果并通过自动化检查的产品版本。

- 不允许设置只交付底层模型、纯重构或者纯文档的实现阶段。
- 每一步都必须包含一个用户可以从界面完整体验的纵向闭环。
- 每一步完成后，上一版本的有效存档必须得到迁移、兼容或者明确拒绝，不能静默损坏。
- 每一步都必须保持确定性，同一显式输入和操作序列必须复现同一结果。
- 每一步都必须通过因果闭包检查，不允许新增无法追溯来源的结果。
- 每一步都必须通过 [quality-gates.md](quality-gates.md) 定义的质量门禁、构建、单元测试和用户界面测试，并遵守 [non-functional-requirements.md](non-functional-requirements.md) 的非功能要求。
- 每一步必须同步更新 [architecture.md](architecture.md)、[quality-gates.md](quality-gates.md)、[non-functional-requirements.md](non-functional-requirements.md) 和对应测试，不允许新实现继续受已被替换的旧架构与旧产品形状约束。
- 旧测试必须区分永久不变量、旧基线兼容与待退役形状断言；固定事件、纪元、对象、文明和奇迹数量不得作为新运行时的永久门禁。
- 旧测试的逐条分类、退出步骤和替代契约以 [旧产品形状测试退役登记](legacy-test-retirement.md) 为准；没有登记的旧形状不得随实现一起继续扩散。
- 被替换或删除的旧模块只能存在于明确隔离的只读兼容路径中，不得继续进入新产品主流程、主导航或事实来源。
- 如果一个阶段无法通过浏览器演示其新增价值，则该阶段不得标记完成。

统一运行方式：

```text
npm install
npm run dev
```

统一停止方式是在启动开发服务器的可见终端中按 `Ctrl + C`。

统一发布验收方式：

```text
npm run check:release
```

## 4. 步骤 1：可追因的宇宙生成器

本步骤的需求、非目标和可验证完成条件已经冻结，见 [步骤 1 冻结验收口径](step-1-freeze.md)。冻结后的评审不得增加需求、改变完成条件或评价后续步骤。

### 用户获得的产品

玩家仍可生成和浏览当前类型的宇宙，但点击任意结果、指标、状态或事件时，都可以沿因果关系逐层追溯，直到宇宙公理、初始状态或显式玩家输入。

### 实现重点

- 建立统一的因果节点、因果边、规则引用和来源契约。
- 把宇宙公理、初始状态、外部输入、状态变化、事件和观察结果纳入同一因果模型。
- 为现有生成流程补齐直接原因、适用规则、前置状态和确定性选择记录。
- 建立因果闭包校验器，拒绝孤立结果、丢失根因、无规则支撑的变化和未经宇宙宪法许可的因果循环。
- 固化“完备同因必得同果、同果不必同因、状态相同不代表历史相同”的运行契约。
- 记录 PRNG 算法版本、随机流身份和抽样序号，为步骤 2 的可序列化流状态建立兼容边界。
- 将 `UniverseSummary` 明确降为旧基线兼容读取模型，不再允许新增领域事实只存在于该聚合对象中。
- 将现有“解释文案”降级为因果事实的展示结果，不能继续充当事实来源。
- 在界面提供“为什么发生”和“产生了什么后果”两种查询方向。
- 将现有 `phase-*` 测试拆分为永久不变量、旧基线兼容和待退役形状断言，并为每项旧形状断言登记退役步骤。
- 更新当前架构、质量门禁和非功能需求，明确一次性静态生成流水线只属于旧基线，不是后续扩展目标。

### 可运行产物

一款能够生成宇宙、浏览现有内容并交互式查看完整因果链的网页产品。

完成证据：步骤 1 已升级到 `ugs-causality@13` 与 `ugs-random-bindings@5`。字段值按需投影记录公式、操作数和精确随机决定；全页面领域派生值登记精确主题、原始值指纹和实际显示文本并与生产事实核对；五类领域加权选择保留候选顺序、权重与资格；路径、神话和高风险统计具有字段、分组、独立谓词及深层负向校验。冻结后的首次独立评审发现两项 P1，经过两轮有限复查后全部关闭，最终 P0、P1、P2 和建议项均为零。详细证据见 [冻结后首次评审](reviews/step-1-frozen-review-01.md)、[第一轮有限复查](reviews/step-1-frozen-review-01-recheck-01.md)、[第二轮有限复查](reviews/step-1-frozen-review-01-recheck-02.md)、[整改报告](reviews/step-1-frozen-review-01-remediation.md) 与 [旧产品模块迁移矩阵](legacy-module-migration.md)。

### 完成标准

- 所有用户可见的派生结果都拥有直接原因和规则来源。
- 任意结果都能递归追溯到合法根因，不存在孤立节点。
- 允许的因果闭环能够追溯到高阶公理和一致性约束。
- 随机选择同时保留候选规则、前置状态和确定性抽样记录。
- 同一输入能够复现完全一致的结果与因果图。
- 不同原因形成相同状态时，因果历史仍保持独立且可以分别查询。
- 旧阶段测试中的固定内容数量已经登记为兼容断言或待退役断言，不再被解释为新产品永久需求。
- 迁移矩阵中归属于步骤 1 的条目均有实现、测试、文档和兼容证据。
- 原有主要浏览、分享和本地存档流程仍可使用。

## 5. 步骤 2：持续演化的宇宙

本步骤的需求、非目标、完成条件、兼容边界、质量要求和必要威胁模型已经冻结，见 [步骤 2 冻结验收口径](step-2-freeze.md)。冻结后的评审不得增加需求、改变完成条件或评价后续步骤。

### 用户获得的产品

玩家启动的不再是一份已经生成完毕的宇宙报告，而是一个可以暂停、推进和继续演化的宇宙。历史事件由实际状态变化产生，玩家能够观察对象和规律随时间发生变化。

### 实现重点

- 建立确定性的模拟时钟、状态快照和状态转换协议。
- 建立 `UniverseState` 作为新运行时唯一事实来源，规则、事件、指标和界面都只能消费或投影该状态。
- 保存 PRNG 算法版本、命名随机流、流游标和输入顺序，确保恢复后的下一次转换与未中断运行一致。
- 将预制完整时间线迁移为按时间步或条件触发的状态演化。
- 让事件成为状态变化的可读投影，而不是未来结果的事实来源。
- 删除新运行时中的 `timeline → timelineImpact → 空间／生命／文明` 反向生成链，旧 `timelineImpact` 只允许在旧存档只读查看器中存在。
- 将固定全局指标降为状态或观察结果的派生投影，并用模拟时钟替代随机年龄评分。
- 将空间对象和其他世界对象迁移为可以创建、变化和消亡的持久状态，不再只生成一次性代表样本。
- 支持暂停、单步推进、连续运行、速度调整和从存档继续。
- 建立新的运行状态存档信封，保存模拟位置、状态快照、输入日志、因果完整性信息和随机流状态。
- 浏览器实现使用能够承载版本化状态快照的存储适配器，`localStorage` 不再保存完整运行分支数据。
- 为长时间跨度建立分层精度和计算预算，避免无意义的微观全量模拟。

### 可运行产物

一款可以从初始状态启动、由玩家控制时间并持续产生可追因变化的宇宙观察产品。

### 完成标准

- 宇宙启动时不再以预制事件列表作为未来历史来源，界面也不提前平铺尚未演化到的结果。
- 每个事件都由真实状态差异生成，并进入因果网络。
- 相同初始输入和操作序列能够复现相同状态序列。
- 保存后重新打开可以从同一状态继续演化。
- 恢复后执行下一时间步的结果与未中断运行完全一致，包括随机选择和因果节点 ID。
- 新运行时不存在通过预制事件摘要反向创造世界对象的路径。
- 指标、事件和解释都可以删除后重新由状态与观察数据推导，不造成事实丢失。
- 传统空间对象至少完成一次形成、变化或消亡过程，并且不再只是静态样本卡片。
- 迁移矩阵中归属于步骤 2 的旧生成、时间线、指标、空间、存档和页面模块均完成处理。
- 在合理性能预算内，玩家可以观察到至少一个完整的状态形成、变化和后续影响过程。

## 6. 步骤 3：自由探索的观察台

### 用户获得的产品

玩家不再面对平铺的全部真相，而是自主选择观察对象、观测方式和研究方向，并通过信号、变化和关联逐步形成理解。

### 实现重点

- 分离宇宙事实、可观察信号、内部认知和玩家推测。
- 建立按对象、尺度、时间和现象组织的自由观察入口。
- 让未知状态、证据强度和解释冲突成为正式产品状态。
- 提供关注、标记、笔记和观察历史，帮助玩家延续自己的问题。
- 使用异常和变化提供可选线索，但不生成强制任务和默认探索终点。
- 重构主导航与页面层级，围绕当前宇宙、观察、实验、历史分支和存档组织，星系、文明等对象类型不再是永久一级栏目。
- 将旧 Overview 替换为当前可知状态、近期变化、未知区域和玩家关注内容，不显示未经观察的底层数量。
- 删除旧 Logs 的“重要事件、稀有发现、潜在终局”生成页，替换为研究记录簿。
- 将 Laws 页面拆分为玩家设定的公开公理、已发现规律、内部实体观点和未知规则，禁止直接平铺全部底层法则。
- 将 `ObservationConsole` 改为消费观测信号而不是完整事实，并为观察方式保留由宇宙宪法动态提供的接口。
- 将 SpaceExplorer 和 CivilizationPanel 降为按宇宙实际对象条件出现的上下文视图，没有对应对象时不显示空白入口。

### 可运行产物

一款没有固定任务路线，玩家可以围绕任意可观察现象展开调查并逐步获得证据的探索产品。

### 完成标准

- 玩家可以忽略生命、文明或任何预设主题，仍然完成一次有证据积累的探索过程。
- 底层事实不会在未经观察时自动作为答案展示。
- 同一事实可以被不同观察方式部分揭示，并保留证据来源。
- 玩家能够保存关注对象、观察记录和自己的推测。
- 产品至少提供多条互不依赖的观察路径，不存在唯一正确游玩顺序。
- 任何一级导航都不以生命、文明、神话、星系或奇迹必然存在为前提。
- 旧 Overview、Logs 和九页后台式导航不再进入新产品主流程。
- 玩家看到的法则、指标和对象详情都经过认知边界，不得通过旧选择器绕过观察系统读取底层事实。
- 迁移矩阵中归属于步骤 3 的信息架构、观察、日志和条件视图条目均完成处理。

## 7. 步骤 4：可分支的宇宙实验室

### 用户获得的产品

玩家可以从任意有效状态创建实验分支，改变一个或多个显式条件，并比较不同宇宙为何走向不同结果。

### 实现重点

- 建立不可变历史、分支身份和共同祖先模型。
- 分离 `ExperimentInput` 与 `InUniverseIntervention`，二者使用统一可追踪协议，但保持不同权限、可观察性、历史语义和界面入口。
- 建立宇宙定义 ID、分支 ID、检查点 ID、状态哈希和历史哈希；状态相同的分支不得因此丢失不同因果历史。
- 任何会影响宇宙的观察行为必须作为显式因果输入，纯界面浏览不得进入历史。
- 提供分支间的状态差异、因果路径差异和关键分歧点比较。
- 禁止静默改写既有历史，所有回到过去的修改都创建新分支。
- 支持重放完整输入序列，验证实验结果的确定性。
- 删除旧 MiraclePanel 的固定奇迹、清空干预和覆盖式重生成交互，替换为界外实验与宇宙内干预两套明确入口。
- 将旧 Seed 法则对比替换为共同祖先分支比较；跨宇宙比较作为独立工具保留，并明确不存在共同历史。
- 定义创世条件包和可继续历史分支包，接收者继续操作时创建自己的新分支。

### 可运行产物

一款可以提出“如果改变这个条件会怎样”，并通过真实分支演化获得答案的宇宙实验产品。

### 完成标准

- 任意有效状态都可以创建可独立继续运行的分支。
- 分支保留共同历史，并从首个不同输入开始产生差异。
- 比较界面能够指出关键结果差异对应的原因和规则。
- 玩家干预成为正式因果节点，并展示直接结果及其继续传播的影响。
- 原宇宙不因创建或运行分支而发生改变。
- 相同完整原因和输入顺序产生相同分支结果，同一结果不要求来自相同原因。
- 两个分支即使状态哈希相同，只要历史哈希不同就仍是不同分支。
- 玩家不能通过“清除干预”无痕回写当前历史，只能返回原分支或创建新分支。
- 创世条件分享与历史分支分享在格式、界面和恢复语义上明确区分。
- 迁移矩阵中归属于步骤 4 的比较、干预、分享、分支和图书馆条目均完成处理。

## 8. 步骤 5：多本体宇宙沙盒

### 用户获得的产品

玩家可以创建运行基础显著不同的宇宙，而这些宇宙都能持续演化、被观察、被实验并保持因果闭合。

### 实现重点

- 将宇宙宪法拆分为本体、作用、约束、优先级、时间、空间拓扑、认知、可观察量、事件分类、干预能力和边界等可组合规则模块。
- 建立统一的条件判断、效果执行、冲突裁决和代价结算机制。
- 允许不同因果机制共享同一状态转换和因果追踪基础设施。
- 建立规则包校验，阻止缺少定义、互相矛盾或无法执行的宇宙宪法进入模拟。
- 提供若干运行基础显著不同的参考规则包，用于证明引擎没有绑定单一物理世界观。
- 将旧十种模板迁移为宪法预设或旧存档映射，固定六领域权重不再构成新规则包接口。
- 允许规则包定义自己的状态指标、时间模型、空间结构、观察通道、事件类别和可用干预，不再强制固定七指标、固定纪元、三级空间和五种观察叠层。
- 将固定奇迹清单、奇迹点和强制反噬从产品级硬编码中删除；类似机制只有在对应宇宙宪法明确允许时才出现。
- 提供至少一个没有传统星系—恒星—行星结构且没有生命、文明专属指标的参考宇宙。

### 可运行产物

一款能够选择、组合并运行多类宇宙宪法，且所有结果都能统一追因的多本体宇宙沙盒。

### 完成标准

- 不同规则包通过同一模拟入口运行，不依赖各自独立的硬编码产品流程。
- 规则冲突能够按照宇宙宪法稳定裁决并记录原因。
- 非物理结果与物理结果遵守相同的因果闭包要求。
- 每个参考宇宙都能完成创建、演化、观察、干预、分支和保存闭环。
- 新增合规规则包不需要修改既有规则包的内部实现。
- 不存在传统空间、生命或文明结构的参考宇宙不会出现无意义的固定页面、指标、纪元或干预选项。
- 观察方式、指标、事件分类和可用干预都能随宇宙宪法变化，界面不存在硬编码旧领域的必选入口。
- 旧模板、固定法则领域、固定空间树和固定奇迹只能在旧存档只读兼容路径中存在。
- 迁移矩阵中归属于步骤 5 的模板、法则、指标、空间、观察和干预能力条目均完成处理。

## 9. 步骤 6：具有自主实体的世界

### 用户获得的产品

当某个宇宙允许自主实体存在时，这些实体能够依据自己的感知、记忆、认知、意图和约束行动，而不是等待事件模板替它们安排历史。

### 实现重点

- 建立通用实体状态、有限感知、记忆、信念、意图和行动协议。
- 将实体认知与宇宙事实分离，允许误解、未知和认知变化。
- 让实体行动通过规则校验和状态转换进入因果网络。
- 支持实体关系、长期记忆和行动后果持续影响未来选择。
- 保持实体系统可选，不要求所有宇宙都产生生命或自主主体。
- 将生命从即时阈值结果改为可形成、变化和消亡的状态过程，不再预先生成文明种子、未来能力或终局倾向。
- 禁止实体诞生时预赋完整路径、未来历史和最终命运，历史只能由后续状态转换与行动形成。
- 分离宇宙事实、实体信念、社会叙述、神话档案和真实自主实体，同名对象也不得因此自动视为同一事实。
- 将固定文明页面替换为条件出现的自主实体视图，支持非生命主体和不存在自主实体的宇宙。
- 删除文明数量、文明候选转化率、固定历史条数、终局类型覆盖数和路径多样性等旧产品配额门禁。

### 可运行产物

一款既能运行无主体宇宙，也能在规则允许时自然产生自主行为、认知冲突和长期历史的世界模拟产品。

### 完成标准

- 每次自主行动都能追溯到感知、认知、意图、约束和适用规则。
- 实体可以基于不完整或错误认知采取符合自身逻辑的行动。
- 世界能够在没有玩家干预时持续形成新的因果历史。
- 玩家可以观察实体行为的证据，但不会自动获得实体不可见的全部真相。
- 自主实体不存在时，宇宙仍能正常演化和提供探索价值。
- 生命、文明或其他自主主体的出现与消亡都能追溯到状态、规则和实际行动，不依赖预制候选种子。
- 实体可以拥有错误信仰或虚构神话，系统不会把其叙述自动提升为宇宙事实。
- 自主实体的未来在当前时刻没有被预先物化为完整路径、历史或终局记录。
- 没有自主实体的宇宙不会生成占位文明、神话页面或对应数量指标。
- 迁移矩阵中归属于步骤 6 的生命、生物圈、文明、神话和自主实体条目均完成处理。

## 10. 步骤 7：可创作的长期宇宙

### 用户获得的产品

玩家可以长期保存、继续、分支、复现和整理一个宇宙，并把真实形成的世界规则与历史导出为可靠的创作基础。

### 实现重点

- 保存完整宇宙状态、因果网络、观察记录、分支关系、玩家标记和版本信息。
- 建立长期存档迁移、完整性校验和可恢复性验证。
- 提供宇宙宪法、状态档案、因果历史、分支对比和结构化编年史导出。
- 叙事化内容只能引用模拟事实或明确标注的内部视角。
- 分享内容必须能够恢复原宇宙，或者明确声明为不可运行的静态创作快照。
- 将本地图书馆替换为分支与档案管理器，分别管理宇宙定义、运行分支、检查点、观察记录和创作档案。
- 浏览器状态存储使用版本化 IndexedDB 适配器，`localStorage` 只保留轻量偏好与迁移标记。
- 提供三种明确格式：创世条件包、可继续历史分支包和不可运行静态创作档案，并为每种格式提供可下载文件与完整性校验。
- A1 存档只作为旧基线只读导入格式；打开后不得冒充新运行分支，转换为新宇宙时必须创建新身份并声明历史不等价。
- 旧规则版本只能受控迁移、明确拒绝或作为静态快照打开，禁止用当前规则重新生成并冒充原宇宙。
- 删除单一分享码作为宇宙、分支与存档共同身份的设计，所有导入导出保留宇宙定义 ID、分支 ID、检查点 ID、状态哈希和历史哈希。
- 将旧 JSON 文本框导出降为调试能力，正式产品使用类型明确的文件导入导出流程。

### 可运行产物

一款可以长期培育宇宙，并将其作为小说、电影、游戏或其他作品世界观基础使用的完整产品。

### 完成标准

- 完整存档导出后能够恢复宇宙并继续确定性演化。
- 分支、观察记录和玩家标记不会在迁移与恢复过程中丢失。
- 导出的关键结论能够回链到对应事实和因果节点。
- 叙事化输出不会产生与模拟状态冲突的未标注事实。
- 创作者无需阅读内部数据结构，也能理解世界规则、关键历史和重要分歧。
- 从历史分支包恢复后继续运行，其下一状态与原分支未中断运行完全一致。
- 创世条件包只创建共同起点，接收者的第一次不同因果输入会产生新的分支身份。
- 静态创作档案在界面和文件元数据中明确标记为不可运行，不能被错误恢复成活动宇宙。
- A1、未知规则版本和损坏存档都有明确且经过测试的迁移、只读或拒绝结果。
- 迁移矩阵中归属于步骤 7 的存档、分享、版本、导入导出和创作条目均完成处理。

## 11. 实施优先级与范围纪律

步骤 1 是所有后续工作的强制前置条件。在因果闭包成立前，不增加新的对象规模、事件池、视觉复杂度或生成文案。

步骤 2 和步骤 3 负责证明产品从事件浏览器转变为演化与探索产品。在这两个步骤完成前，不把创作者导出、内容扩展和高级自主实体作为主要开发方向。

步骤 4 至步骤 7 只能复用已经通过验收的因果、状态、观察和分支模型，不得另建绕过核心规则的快捷生成链路。

任何新需求都必须回答三个问题：它改变了哪个宇宙状态，它为什么能够发生，以及玩家如何观察或验证它。如果无法回答，则不进入实现范围。

## 12. 旧模块最终关闭审计

步骤 7 完成前，必须按照 [旧产品模块迁移矩阵](legacy-module-migration.md) 执行最终审计：

- 每个旧模块都有唯一且明确的最终处理方式与负责步骤。
- 标记为保留或保留并增强的模块已经符合新产品事实、状态、观察和分支边界。
- 标记为降级的模块已经退出事实来源、主流程和永久一级导航。
- 标记为替换的模块已经由新实现接管所有有效职责，旧实现不再被主流程调用。
- 标记为删除的行为在新运行时中不可达，只能存在于明确隔离的历史只读路径中。
- 所有旧形状测试已经迁移、降级或删除，不再强制固定事件、对象、文明、奇迹和页面结构。
- 当前产品定义、实施路线、架构说明、质量门禁、非功能要求、实现和测试相互一致。

只要矩阵中仍存在未处理条目、隐式兼容入口或无法证明退役的旧模块，产品重构就不得宣告完成。
