# 产品重构实施路线

本文件描述项目从当前“结构化结果与事件浏览器”迁移到目标产品的实施顺序。当前产品定义以 [product-definition.md](product-definition.md) 为唯一事实来源，原阶段 0 至阶段 8 文档只作为既有实现和历史决策的证据，不再决定后续产品方向。

[旧版完整里程碑](archive/legacy-phases/milestones_old.md) 保存此前的完整里程碑原文，不在本路线中修改。

## 1. 当前基线

当前版本已经具备确定性 Seed、结构化法则、时间线、空间对象、文明与神话样本、干预、本地存档、分享和二维观察台等工程能力。

这些能力构成可复用的工程基线，但当前核心体验仍以生成完成后的结果浏览为主，尚未形成持续演化、自主探索和因果推理闭环。因此，原阶段 0 至阶段 8 的“完成”只代表对应工程规格完成，不代表目标产品已经成立。

## 2. 所有步骤共同遵守的交付规则

以下步骤必须按顺序交付，每一步都必须产生一款能够独立启动、操作、保存结果并通过自动化检查的产品版本。

- 不允许设置只交付底层模型、纯重构或者纯文档的实现阶段。
- 每一步都必须包含一个用户可以从界面完整体验的纵向闭环。
- 每一步完成后，上一版本的有效存档必须得到迁移、兼容或者明确拒绝，不能静默损坏。
- 每一步都必须保持确定性，同一显式输入和操作序列必须复现同一结果。
- 每一步都必须通过因果闭包检查，不允许新增无法追溯来源的结果。
- 每一步都必须通过 [quality-gates.md](quality-gates.md) 定义的质量门禁、构建、单元测试和用户界面测试，并遵守 [non-functional-requirements.md](non-functional-requirements.md) 的非功能要求。
- 如果一个阶段无法通过浏览器演示其新增价值，则该阶段不得标记完成。

统一运行方式：

```text
npm install
npm run dev
```

统一停止方式是在启动开发服务器的可见终端中按 `Ctrl + C`。

统一验收方式：

```text
npm run check
```

## 3. 实施步骤总览

| 步骤 | 可运行产品版本 | 核心价值 | 状态 |
| --- | --- | --- | --- |
| 基线 | 结构化宇宙浏览器 | 保留现有工程能力和数据兼容边界 | 已存在 |
| 步骤 1 | 可追因的宇宙生成器 | 任意生成结果都能追溯原因 | 未开始 |
| 步骤 2 | 持续演化的宇宙 | 宇宙由状态变化产生历史 | 未开始 |
| 步骤 3 | 自由探索的观察台 | 玩家围绕自己的问题获取证据 | 未开始 |
| 步骤 4 | 可分支的宇宙实验室 | 玩家通过实验比较偶然与必然 | 未开始 |
| 步骤 5 | 多本体宇宙沙盒 | 不同宇宙宪法由统一机制运行 | 未开始 |
| 步骤 6 | 具有自主实体的世界 | 实体依据认知、意图和约束行动 | 未开始 |
| 步骤 7 | 可创作的长期宇宙 | 完整历史可保存、复现、解释和导出 | 未开始 |

## 4. 步骤 1：可追因的宇宙生成器

### 用户获得的产品

玩家仍可生成和浏览当前类型的宇宙，但点击任意结果、指标、状态或事件时，都可以沿因果关系逐层追溯，直到宇宙公理、初始状态或显式玩家输入。

### 实现重点

- 建立统一的因果节点、因果边、规则引用和来源契约。
- 把宇宙公理、初始状态、外部输入、状态变化、事件和观察结果纳入同一因果模型。
- 为现有生成流程补齐直接原因、适用规则、前置状态和确定性选择记录。
- 建立因果闭包校验器，拒绝孤立结果、丢失根因、无规则支撑的变化和未经宇宙宪法许可的因果循环。
- 将现有“解释文案”降级为因果事实的展示结果，不能继续充当事实来源。
- 在界面提供“为什么发生”和“产生了什么后果”两种查询方向。

### 可运行产物

一款能够生成宇宙、浏览现有内容并交互式查看完整因果链的网页产品。

### 完成标准

- 所有用户可见的派生结果都拥有直接原因和规则来源。
- 任意结果都能递归追溯到合法根因，不存在孤立节点。
- 允许的因果闭环能够追溯到高阶公理和一致性约束。
- 随机选择同时保留候选规则、前置状态和确定性抽样记录。
- 同一输入能够复现完全一致的结果与因果图。
- 原有主要浏览、分享和本地存档流程仍可使用。

## 5. 步骤 2：持续演化的宇宙

### 用户获得的产品

玩家启动的不再是一份已经生成完毕的宇宙报告，而是一个可以暂停、推进和继续演化的宇宙。历史事件由实际状态变化产生，玩家能够观察对象和规律随时间发生变化。

### 实现重点

- 建立确定性的模拟时钟、状态快照和状态转换协议。
- 将预制完整时间线迁移为按时间步或条件触发的状态演化。
- 让事件成为状态变化的可读投影，而不是未来结果的事实来源。
- 支持暂停、单步推进、连续运行、速度调整和从存档继续。
- 为长时间跨度建立分层精度和计算预算，避免无意义的微观全量模拟。

### 可运行产物

一款可以从初始状态启动、由玩家控制时间并持续产生可追因变化的宇宙观察产品。

### 完成标准

- 宇宙启动时不再以预制事件列表作为未来历史来源，界面也不提前平铺尚未演化到的结果。
- 每个事件都由真实状态差异生成，并进入因果网络。
- 相同初始输入和操作序列能够复现相同状态序列。
- 保存后重新打开可以从同一状态继续演化。
- 在合理性能预算内，玩家可以观察到至少一个完整的状态形成、变化和后续影响过程。

## 6. 步骤 3：自由探索的观察台

### 用户获得的产品

玩家不再面对平铺的全部真相，而是自主选择观察对象、观测方式和研究方向，并通过信号、变化和关联逐步形成理解。

### 实现重点

- 分离宇宙事实、可观察信号、内部认知和玩家推测。
- 建立按对象、尺度、时间和现象组织的自由观察入口。
- 让未知状态、证据强度和解释冲突成为正式产品状态。
- 提供关注、标记、笔记和观察历史，帮助玩家延续自己的问题。
- 使用异常和变化提供可选线索，但不生成强制任务和默认探索终点。

### 可运行产物

一款没有固定任务路线，玩家可以围绕任意可观察现象展开调查并逐步获得证据的探索产品。

### 完成标准

- 玩家可以忽略生命、文明或任何预设主题，仍然完成一次有证据积累的探索过程。
- 底层事实不会在未经观察时自动作为答案展示。
- 同一事实可以被不同观察方式部分揭示，并保留证据来源。
- 玩家能够保存关注对象、观察记录和自己的推测。
- 产品至少提供多条互不依赖的观察路径，不存在唯一正确游玩顺序。

## 7. 步骤 4：可分支的宇宙实验室

### 用户获得的产品

玩家可以从任意有效状态创建实验分支，改变一个或多个显式条件，并比较不同宇宙为何走向不同结果。

### 实现重点

- 建立不可变历史、分支身份和共同祖先模型。
- 将实验条件和宇宙内干预统一记录为显式因果输入。
- 提供分支间的状态差异、因果路径差异和关键分歧点比较。
- 禁止静默改写既有历史，所有回到过去的修改都创建新分支。
- 支持重放完整输入序列，验证实验结果的确定性。

### 可运行产物

一款可以提出“如果改变这个条件会怎样”，并通过真实分支演化获得答案的宇宙实验产品。

### 完成标准

- 任意有效状态都可以创建可独立继续运行的分支。
- 分支保留共同历史，并从首个不同输入开始产生差异。
- 比较界面能够指出关键结果差异对应的原因和规则。
- 玩家干预成为正式因果节点，并展示直接结果及其继续传播的影响。
- 原宇宙不因创建或运行分支而发生改变。

## 8. 步骤 5：多本体宇宙沙盒

### 用户获得的产品

玩家可以创建运行基础显著不同的宇宙，而这些宇宙都能持续演化、被观察、被实验并保持因果闭合。

### 实现重点

- 将宇宙宪法拆分为本体、作用、约束、优先级、时间、认知和边界等可组合规则模块。
- 建立统一的条件判断、效果执行、冲突裁决和代价结算机制。
- 允许不同因果机制共享同一状态转换和因果追踪基础设施。
- 建立规则包校验，阻止缺少定义、互相矛盾或无法执行的宇宙宪法进入模拟。
- 提供若干运行基础显著不同的参考规则包，用于证明引擎没有绑定单一物理世界观。

### 可运行产物

一款能够选择、组合并运行多类宇宙宪法，且所有结果都能统一追因的多本体宇宙沙盒。

### 完成标准

- 不同规则包通过同一模拟入口运行，不依赖各自独立的硬编码产品流程。
- 规则冲突能够按照宇宙宪法稳定裁决并记录原因。
- 非物理结果与物理结果遵守相同的因果闭包要求。
- 每个参考宇宙都能完成创建、演化、观察、干预、分支和保存闭环。
- 新增合规规则包不需要修改既有规则包的内部实现。

## 9. 步骤 6：具有自主实体的世界

### 用户获得的产品

当某个宇宙允许自主实体存在时，这些实体能够依据自己的感知、记忆、认知、意图和约束行动，而不是等待事件模板替它们安排历史。

### 实现重点

- 建立通用实体状态、有限感知、记忆、信念、意图和行动协议。
- 将实体认知与宇宙事实分离，允许误解、未知和认知变化。
- 让实体行动通过规则校验和状态转换进入因果网络。
- 支持实体关系、长期记忆和行动后果持续影响未来选择。
- 保持实体系统可选，不要求所有宇宙都产生生命或自主主体。

### 可运行产物

一款既能运行无主体宇宙，也能在规则允许时自然产生自主行为、认知冲突和长期历史的世界模拟产品。

### 完成标准

- 每次自主行动都能追溯到感知、认知、意图、约束和适用规则。
- 实体可以基于不完整或错误认知采取符合自身逻辑的行动。
- 世界能够在没有玩家干预时持续形成新的因果历史。
- 玩家可以观察实体行为的证据，但不会自动获得实体不可见的全部真相。
- 自主实体不存在时，宇宙仍能正常演化和提供探索价值。

## 10. 步骤 7：可创作的长期宇宙

### 用户获得的产品

玩家可以长期保存、继续、分支、复现和整理一个宇宙，并把真实形成的世界规则与历史导出为可靠的创作基础。

### 实现重点

- 保存完整宇宙状态、因果网络、观察记录、分支关系、玩家标记和版本信息。
- 建立长期存档迁移、完整性校验和可恢复性验证。
- 提供宇宙宪法、状态档案、因果历史、分支对比和结构化编年史导出。
- 叙事化内容只能引用模拟事实或明确标注的内部视角。
- 分享内容必须能够恢复原宇宙，或者明确声明为不可运行的静态创作快照。

### 可运行产物

一款可以长期培育宇宙，并将其作为小说、电影、游戏或其他作品世界观基础使用的完整产品。

### 完成标准

- 完整存档导出后能够恢复宇宙并继续确定性演化。
- 分支、观察记录和玩家标记不会在迁移与恢复过程中丢失。
- 导出的关键结论能够回链到对应事实和因果节点。
- 叙事化输出不会产生与模拟状态冲突的未标注事实。
- 创作者无需阅读内部数据结构，也能理解世界规则、关键历史和重要分歧。

## 11. 实施优先级与范围纪律

步骤 1 是所有后续工作的强制前置条件。在因果闭包成立前，不增加新的对象规模、事件池、视觉复杂度或生成文案。

步骤 2 和步骤 3 负责证明产品从事件浏览器转变为演化与探索产品。在这两个步骤完成前，不把创作者导出、内容扩展和高级自主实体作为主要开发方向。

步骤 4 至步骤 7 只能复用已经通过验收的因果、状态、观察和分支模型，不得另建绕过核心规则的快捷生成链路。

任何新需求都必须回答三个问题：它改变了哪个宇宙状态，它为什么能够发生，以及玩家如何观察或验证它。如果无法回答，则不进入实现范围。
