# 宇宙创世沙盒项目演进里程碑规格

版本：0.1  
日期：2026-06-30  
定位：兴趣向、造物主视角、硬科幻与神话幻想混合的宇宙生成沙盒

## 1. 项目愿景

本项目不是严格复刻现实宇宙的学术模拟器，而是一个“宇宙生成神话引擎”。

用户输入或随机生成一个 seed，系统基于该 seed 生成一个自洽宇宙。这个宇宙可以符合现实物理，也可以存在魔法、神明、灵魂、奇迹、轮回、时间回流、命名术、信仰成神等超自然法则。

核心体验是：

> 用户像造物主一样设定或抽取初始法则，然后观察一个宇宙如何从虚无中演化出星辰、生命、文明、神话、灾难和终局。

## 2. 产品原则

### 2.1 确定性

同一个 seed 与同一组规则版本，必须生成完全一致的宇宙结果。

要求：

- 所有随机过程必须来自统一的可复现 PRNG。
- 禁止在模拟核心中直接使用不可控随机源。
- 每个模块使用派生随机流，例如：
  - `cosmology`
  - `laws.magic`
  - `galaxy.1024`
  - `planet.1024.3.life`
  - `civilization.1024.3.7`
- 模拟规则升级后需要记录规则版本，避免旧 seed 结果被静默改变。

### 2.2 自洽优先于真实

现实物理是可选法则集之一，不是唯一标准。

要求：

- 每个宇宙必须有明确底层法则。
- 魔法、神明、奇迹也必须被规则化，而不是任意发生。
- 法则之间需要有兼容、冲突、代价或边界。

### 2.3 先好玩，再加深

优先让用户在早期版本中感受到：

- 我创造了一个独特宇宙。
- 这个宇宙有自己的命运。
- 我能观察、命名、记录、干预它。
- 同一个 seed 可以被分享和复现。

### 2.4 分层模拟

不要试图完整模拟每个粒子、每颗恒星、每个生命个体。

采用分层近似：

- 宇宙级：法则、时间、物质、魔力、神性。
- 星系级：结构、恒星形成、资源、灾难。
- 恒星系级：恒星、行星、宜居性、魔法环境。
- 生命级：生命出现、复杂化、智能、灵魂或意识形态。
- 文明级：技术、魔法、信仰、扩张、毁灭、飞升。
- 神话级：神明诞生、奇迹、神战、法则破裂、宇宙终局。

## 3. 总体里程碑

### 阶段 0：概念规格与原型边界

目标：把项目从灵感变成可执行的产品与技术边界。

交付物：

- 项目愿景文档。
- 核心玩法说明。
- seed 确定性原则。
- 宇宙法则分类。
- MVP 范围。
- 暂不做清单。

验收标准：

- 能用一页文档解释项目是什么、不是是什么。
- 能说明第一版用户打开应用后会做什么。
- 能说明为什么它不是严肃物理模拟器，而是自洽宇宙沙盒。

暂不做：

- 不做 3D 宇宙。
- 不做复杂生命演化树。
- 不做多人在线。
- 不做插件系统。
- 不做 AI 自动长篇小说生成。

当前状态：阶段 0 已落地为正式规格，见 [docs/phase-0.md](phase-0.md)。

### 阶段 1：可复现宇宙卡片 MVP

目标：用户输入或随机生成 seed 后，得到一个完整、可复现、有风味的宇宙摘要。

当前状态：阶段 1 已实现为网页应用，见 [docs/phase-1.md](phase-1.md)。

用户故事：

- 作为用户，我可以点击“创世”生成一个新宇宙。
- 作为用户，我可以输入 seed 重现同一个宇宙。
- 作为用户，我可以看到这个宇宙的名称、类型、底层法则、演化概况和命运倾向。
- 作为用户，我可以复制 seed 与宇宙摘要分享给别人。

核心功能：

1. Seed 系统
   - 随机生成 seed。
   - 手动输入 seed。
   - 显示当前 seed。
   - 一键复制 seed。

2. 宇宙法则生成
   - 生成宇宙名称。
   - 生成宇宙类型。
   - 生成物理法则摘要。
   - 生成超自然法则摘要。
   - 生成生命与意识法则摘要。
   - 生成神性系统摘要。

3. 宇宙摘要面板
   - 宇宙年龄。
   - 宇宙稳定度。
   - 生命潜力。
   - 文明潜力。
   - 魔法强度。
   - 神性活跃度。
   - 因果稳定度。

4. 初始时间线
   - 至少生成 8 至 12 个宇宙大事件。
   - 事件覆盖创世、星辰、生命、文明、神话或终局。

示例输出：

```text
宇宙名：银烬庭
Seed：LUX-7F3A-91C2
类型：高魔法、多神、熵不稳定宇宙
魔法来源：恒星死亡时释放的灵质
神性来源：文明集体记忆
主要风险：奇迹过度使用会撕裂因果
```

数据规格：

```ts
type UniverseSummary = {
  seed: string
  rulesetVersion: string
  name: string
  archetype: UniverseArchetype
  metrics: UniverseMetrics
  laws: UniverseLaws
  timeline: TimelineEvent[]
}
```

验收标准：

- 同一个 seed 刷新页面后结果完全一致。
- 随机生成 50 个 seed，不出现空白宇宙或明显字段缺失。
- 宇宙摘要能在 2 秒内生成。
- 用户不需要理解复杂参数也能看懂这个宇宙的特点。

暂不做：

- 不做 3D 可视化。
- 不做详细星系地图。
- 不做玩家干预。
- 不做存档列表。

### 阶段 2：宇宙法则引擎

目标：把“物理、魔法、神性、生命、意识”从文案变成结构化规则系统。

当前状态：阶段 2 已实现为宇宙法则引擎，见 [docs/phase-2.md](phase-2.md)。

用户故事：

- 作为用户，我能看到一个宇宙的法则为什么会导致某些结果。
- 作为用户，我能比较两个 seed 的法则差异。
- 作为用户，我能选择“现实偏向”“高魔法”“多神”“混沌”等宇宙倾向。

核心功能：

1. 法则分类
   - 物理法则。
   - 魔法法则。
   - 生命法则。
   - 意识与灵魂法则。
   - 神性法则。
   - 因果与时间法则。

2. 法则强度
   - 每类法则有数值强度或枚举状态。
   - 法则之间存在兼容性或冲突。

3. 宇宙类型模板
   - 硬科学宇宙。
   - 低魔宇宙。
   - 高魔宇宙。
   - 神话宇宙。
   - 多神战争宇宙。
   - 梦境宇宙。
   - 轮回宇宙。
   - 机械神宇宙。
   - 因果裂缝宇宙。
   - 混沌法则宇宙。

4. 法则解释器
   - 为关键结果生成简短解释。
   - 例如：“该宇宙生命潜力较低，因为重元素生成率低且因果稳定度不足。”

数据规格：

```ts
type UniverseLaws = {
  physics: PhysicsLaws
  magic: MagicLaws
  life: LifeLaws
  consciousness: ConsciousnessLaws
  divinity: DivinityLaws
  causality: CausalityLaws
}

type LawRating = {
  value: number
  label: string
  explanation: string
}
```

验收标准：

- 每个宇宙至少有 12 条结构化法则。
- 法则能影响指标，而不是只作为描述文字。
- 选择不同宇宙模板会明显改变生成结果。
- 同 seed + 同模板结果完全一致。

暂不做：

- 不允许玩家逐条编辑所有法则。
- 不做复杂公式编辑器。
- 不做模组系统。

### 阶段 3：宇宙时间线与纪元系统

目标：让宇宙不只是一个静态卡片，而是有历史、有阶段、有命运推进。

当前状态：阶段 3 已实现为宇宙时间线与纪元系统，见 [docs/phase-3.md](phase-3.md)。

用户故事：

- 作为用户，我可以从创世一路看见宇宙走向终局。
- 作为用户，我可以查看每个纪元发生了什么。
- 作为用户，我能感受到某些早期法则如何影响后期生命、文明和神话。

核心功能：

1. 时间轴
   - 支持宇宙从 `t=0` 到远未来。
   - 以纪元而非连续秒数为主要展示单位。
   - 支持跳转到关键事件。

2. 纪元模型
   - 创世纪元。
   - 星火纪元。
   - 元素纪元。
   - 生命纪元。
   - 文明纪元。
   - 神话纪元。
   - 飞升或衰亡纪元。
   - 终局纪元。

3. 事件系统
   - 事件有时间、地点、类型、影响。
   - 事件影响后续概率。
   - 事件可以来自自然、魔法、神性、文明或造物主干预。

4. 事件详情
   - 显示事件原因。
   - 显示事件结果。
   - 显示影响指标。

数据规格：

```ts
type TimelineEvent = {
  id: string
  age: number
  ageLabel: string
  era: EraId
  type: EventType
  title: string
  description: string
  causes: string[]
  effects: EventEffect[]
  importance: number
  location: string
  sourceIds: string[]
  triggeredByEventIds: string[]
  causalNotes: string[]
}
```

验收标准：

- 每个宇宙生成至少 30 个事件。
- 时间线能按纪元筛选。
- 关键事件之间存在因果解释。
- 至少 5 类事件会影响后续生成结果。

暂不做：

- 不做可视化分支时间线。
- 不做玩家回滚历史。
- 不做复杂文明个体叙事。

---

### 阶段 4：星系、恒星、行星与生命样本

目标：让宇宙有可探索的局部对象，而不是只有宏观历史。

当前状态：阶段 4 准备开发，前置契约见 [docs/phase-4.md](phase-4.md)。阶段 3 已输出 `timelineImpact`，阶段 4 应直接消费该摘要、结构化法则和确定性随机流。

用户故事：

- 作为用户，我可以进入一个宇宙中的代表性星系。
- 作为用户，我可以查看恒星系与行星。
- 作为用户，我可以发现某些行星上出现生命、魔法生态或文明。

核心功能：

1. 星系样本生成
   - 生成一批代表性星系，而非完整宇宙所有星系。
   - 星系有类型、质量、年龄、金属丰度、魔力通量、神性残留。

2. 恒星系样本生成
   - 生成代表性恒星系。
   - 恒星属性包括质量、寿命、稳定性、光度、异常属性。

3. 行星生成
   - 行星类型：岩石、气态、冰冻、海洋、荒漠、浮空、梦境、灵质、机械等。
   - 指标：温度、水、稳定轨道、大气、魔法浓度、灵魂容纳性。

4. 生命样本
   - 无生命。
   - 原始生命。
   - 复杂生命。
   - 智慧生命。
   - 魔法生命。
   - 灵体生命。
   - 机械生命。

数据规格：

```ts
type Galaxy = {
  id: string
  name: string
  type: GalaxyType
  mass: number
  metallicity: number
  magicFlux: number
  divineResidue: number
  starSystems: StarSystem[]
}

type Planet = {
  id: string
  name: string
  type: PlanetType
  habitability: number
  magicSaturation: number
  biosphere?: Biosphere
}
```

验收标准：

- 每个宇宙至少生成 12 个代表性星系。
- 每个星系至少可展开 3 至 8 个代表性恒星系。
- 行星生命结果受宇宙法则与局部环境共同影响。
- 用户能从宇宙摘要一路点击到某颗行星详情。

暂不做：

- 不生成数十亿真实对象。
- 不做真实 N 体引力模拟。
- 不做行星表面地图。

### 阶段 5：文明演化与神话生成

目标：让生命不止停留在“存在/不存在”，而是能发展出文明、信仰、科技、魔法体系和神话历史。

用户故事：

- 作为用户，我可以看到某个文明如何诞生、扩张、衰落或飞升。
- 作为用户，我可以看到文明是否依赖科技、魔法、神明或集体意识。
- 作为用户，我可以看到神明是外部造物者、自然神，还是由文明信仰生成。

核心功能：

1. 文明模型
   - 文明名称。
   - 起源行星。
   - 物种类型。
   - 技术等级。
   - 魔法等级。
   - 信仰结构。
   - 扩张倾向。
   - 灭绝风险。

2. 文明路径
   - 原始部落。
   - 城邦。
   - 行星文明。
   - 星系文明。
   - 魔法帝国。
   - 神权文明。
   - 集体意识。
   - 飞升文明。
   - 失落文明。

3. 神话系统
   - 无神。
   - 造物主神。
   - 自然神。
   - 信仰神。
   - 恒星神。
   - 黑洞神。
   - 死亡或梦境神。
   - 被文明制造的机械神。

4. 文明事件
   - 第一次火种或语言。
   - 第一次魔法。
   - 第一次天文观测。
   - 第一次接触神明。
   - 世界大战。
   - 星海远航。
   - 飞升仪式。
   - 文明灭绝。

数据规格：

```ts
type Civilization = {
  id: string
  name: string
  originPlanetId: string
  speciesType: SpeciesType
  technologyLevel: number
  magicLevel: number
  faithIntensity: number
  expansionDrive: number
  stability: number
  fate: CivilizationFate
}
```

验收标准：

- 生命行星中按规则概率生成文明。
- 文明路径与宇宙法则一致。
- 至少生成 5 类文明终局。
- 每个重要文明有 8 至 15 条历史事件。

暂不做：

- 不做完整 4X 游戏。
- 不做文明微观经济系统。
- 不做外交和战争沙盘细节。

### 阶段 6：造物主干预与奇迹系统

目标：让用户真正获得“造物主感”，但干预必须有代价、记录和后果。

用户故事：

- 作为用户，我可以对宇宙施加有限干预。
- 作为用户，我能看到每次干预如何改变后续历史。
- 作为用户，我需要在拯救、加速、惩罚、祝福、扭曲法则之间做选择。

核心功能：

1. 造物主模式
   - 观察者模式：只看，不干预。
   - 调参造物模式：创世前调整初始法则。
   - 奇迹模式：创世后有限干预。

2. 奇迹类型
   - 祝福行星。
   - 稳定恒星。
   - 注入生命。
   - 赐予魔法。
   - 降下灾难。
   - 复活文明。
   - 封印神明。
   - 修复因果裂缝。
   - 改写局部法则。

3. 干预代价
   - 消耗奇迹点。
   - 降低因果稳定度。
   - 增加法则裂缝。
   - 引发神性反噬。
   - 改变未来事件概率。

4. 干预日志
   - 记录用户何时干预。
   - 记录干预对象。
   - 记录直接结果。
   - 记录长远后果。

数据规格：

```ts
type Miracle = {
  id: string
  type: MiracleType
  targetId: string
  cost: MiracleCost
  immediateEffects: EventEffect[]
  longTermRisks: string[]
}

type InterventionLog = {
  id: string
  age: number
  miracleId: string
  resultEventIds: string[]
}
```

验收标准：

- 至少支持 6 种奇迹。
- 每次奇迹会改变至少一个后续指标或事件概率。
- 奇迹过度使用会触发负面后果。
- 干预后的宇宙仍可通过 seed + 干预日志复现。

暂不做：

- 不做无限自由文本施法。
- 不做完全任意编辑历史。
- 不做多人共同干预。

### 阶段 7：可视化宇宙观察台

目标：把宇宙从文字与表格升级为可观察、可缩放、可沉浸浏览的界面。

用户故事：

- 作为用户，我可以看到宇宙结构的视觉呈现。
- 作为用户，我可以从宏观宇宙缩放到星系、恒星系、行星。
- 作为用户，我可以通过颜色、光效或标记识别生命、魔法、文明、神性异常。

核心功能：

1. 宇宙视图
   - 星系点云。
   - 星系团结构。
   - 宇宙背景视觉。
   - 魔法或神性异常视觉层。

2. 缩放层级
   - 宇宙。
   - 星系。
   - 恒星系。
   - 行星。
   - 文明概览。

3. 时间播放
   - 播放宇宙演化。
   - 暂停。
   - 拖动时间。
   - 跳转关键事件。

4. 信息叠层
   - 生命热点。
   - 文明热点。
   - 魔法风暴。
   - 神性遗迹。
   - 因果裂缝。

验收标准：

- 用户能通过视觉界面选择一个星系并查看详情。
- 时间轴变化会影响视觉状态。
- 可视化在普通桌面浏览器中保持流畅。
- 移动端至少能查看摘要和时间线。

暂不做：

- 不做真实天文级渲染。
- 不做行星表面 3D 漫游。
- 不做 VR。

### 阶段 8：存档、分享与宇宙图书馆

目标：让用户保存、收藏、比较和分享自己发现的宇宙。

用户故事：

- 作为用户，我可以保存喜欢的宇宙。
- 作为用户，我可以给宇宙、星系、行星、文明重新命名。
- 作为用户，我可以分享一个宇宙链接给别人。
- 作为用户，我可以比较多个宇宙的差异。

核心功能：

1. 本地存档
   - 保存 seed。
   - 保存规则版本。
   - 保存模板。
   - 保存干预日志。
   - 保存用户命名。

2. 分享链接
   - URL 包含 seed。
   - URL 包含规则版本。
   - 可选包含干预日志摘要。

3. 宇宙图书馆
   - 收藏列表。
   - 标签。
   - 搜索。
   - 按生命潜力、文明数量、魔法强度排序。

4. 宇宙比较
   - 指标对比。
   - 法则对比。
   - 时间线对比。

验收标准：

- 用户能保存至少 20 个宇宙到本地。
- 刷新页面后存档不丢失。
- 分享链接打开后能复现同一宇宙。
- 用户重命名不会破坏 seed 复现。

暂不做：

- 不做云账号。
- 不做公开社区。
- 不做排行榜。

### 阶段 9：创作者工具与内容扩展

目标：支持未来扩展更多宇宙类型、法则、事件、文明路径和文案素材。

用户故事：

- 作为设计者，我可以添加新的宇宙模板。
- 作为设计者，我可以添加新的事件类型。
- 作为设计者，我可以调整某类宇宙的概率权重。
- 作为用户，我能持续发现新的宇宙风味。

核心功能：

1. 内容配置化
   - 宇宙模板配置。
   - 法则配置。
   - 事件池配置。
   - 名称生成配置。
   - 文明路径配置。

2. 权重系统
   - 不同模板使用不同权重。
   - 法则影响事件池权重。
   - 局部环境影响生命与文明权重。

3. 内容校验
   - 检查缺失字段。
   - 检查非法权重。
   - 检查事件引用不存在的法则。

4. 调试面板
   - 显示 PRNG 分支。
   - 显示关键概率。
   - 显示生成链路。

验收标准：

- 新增一个宇宙模板不需要改核心模拟代码。
- 新增一种事件能被时间线系统消费。
- 内容配置错误能给出明确报错。
- 调试面板能解释一个文明为何出现或灭绝。

暂不做：

- 不开放用户自制模组。
- 不做脚本语言。
- 不做在线市场。

可选方向：

1. 更深生命演化

   - 大灭绝。
   - 共生。
   - 魔法生态。
   - 灵魂生态。

   - 贸易。
   - 战争。
   - 宗教传播。

   - 神明代理战争。

   - 热寂。

   - 大坍缩。
   - 神性吞噬。
   - 轮回重启。

## 4. MVP 推荐范围

建议第一个可玩的版本只包含阶段 1 至阶段 3 的核心能力。

MVP 必须有：

- Seed 输入与随机生成。
- 宇宙摘要。
- 宇宙法则。
- 宇宙指标。
- 纪元时间线。
- 分享 seed。

MVP 可以有但不强求：

- 少量星系或行星样本。
- 简单干预。
- 本地存档。

MVP 不做：

- 3D。
- 复杂文明模拟。
- 奇迹系统。
- 创作者工具。
- 云端社区。

MVP 验收目标：

- 用户在 30 秒内生成第一个宇宙。
- 用户在 3 分钟内愿意再生成一个 seed。
- 用户能用一句话描述这个宇宙的独特之处。
- 用户能复制 seed 给别人复现同一个宇宙。

## 7. 风险与约束

### 7.1 范围膨胀

风险：宇宙、生命、文明、神明、魔法都很诱人，容易无限扩张。

约束：

- 每个阶段必须有可验收结果。
- 新系统必须服务核心体验。
- 不在早期做底层无关的大型模拟。

### 7.2 随机但无意义

风险：内容看似随机丰富，但缺少因果和记忆。

约束：

- 所有重要结果必须能解释来源。
- 时间线事件必须影响后续概率。
- 指标变化必须有法则或事件依据。

### 7.3 文案重复

风险：宇宙生成器容易很快看腻。

约束：

- 文案与结构数据分离。
- 同一事件类型需要多模板表达。
- 宇宙类型要改变事件池和指标，而不仅是换名字。

### 7.4 版本边界

风险：规则升级导致旧 seed 结果被误认为仍可复现。

约束：

- 所有生成结果绑定 `rulesetVersion`。
- 分享链接包含版本。
- 当前应用只保证当前规则版本内复现。
- 非当前规则短码只提示不受支持，不提供旧规则运行时兼容。

## 8. 近期可执行任务

阶段 0 至阶段 3 已经建立可复现创世、结构化法则、指标影响追踪、seed 法则对比、30 条以上纪元时间线、事件因果解释、纪元筛选、时间线影响摘要、内容池分层和 CI 检查。下一步进入阶段 4，按以下顺序推进：

1. 定义代表性星系、恒星系、行星和生命样本的数据结构。
2. 从结构化法则和 `timelineImpact` 中提取局部对象生成影响来源。
3. 每个宇宙至少生成 12 个代表性星系。
4. 每个星系至少可展开 3 至 8 个代表性恒星系。
5. 让行星生命结果受宇宙法则、时间线事件和局部环境共同影响。
6. 在页面中提供从宇宙摘要或时间线进入局部对象详情的路径。
7. 增加阶段 4 自动化验收测试，继续验证 seed 复现性。

## 9. 一句话定义

这是一个 seed 驱动的宇宙创世沙盒：用户播下一组法则，观察物理、魔法、生命、文明与神明如何在同一个确定性宇宙中生长、冲突并走向命运。
