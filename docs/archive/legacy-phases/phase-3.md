# 阶段 3：宇宙时间线与纪元系统

版本：1.0  
日期：2026-07-07  
状态：已完成

## 1. 阶段目标

阶段 3 的目标是把阶段 1 和阶段 2 中的“关键事件列表”升级为可筛选、可解释、会影响后续生成倾向的纪元时间线系统。

阶段 3 完成后，系统应能回答：

1. 一个宇宙从创世到终局至少经历了哪些 30 个以上关键事件。
2. 每个事件属于哪个纪元、哪类事件、发生在什么时间段。
3. 关键事件由哪些法则、指标或前序事件触发。
4. 事件会如何影响后续事件概率或指标倾向。
5. 用户如何按纪元筛选并查看事件详情。

## 2. 范围

### 2.1 必须实现

- 继续保留阶段 1 与阶段 2 的确定性生成能力。
- 每个宇宙至少生成 30 条时间线事件。
- 时间线覆盖创世、星辰、元素、生命、文明、神话、飞升或衰亡、终局等纪元。
- 时间线能按纪元筛选。
- 每个关键事件包含时间、纪元、类型、标题、描述、原因、后果、重要度和影响来源。
- 关键事件之间存在可追踪的因果解释。
- 至少 5 类事件会影响后续事件概率或指标倾向。
- 页面提供纪元筛选、事件列表和事件详情查看。
- 为阶段 3 新增自动化验收测试。

### 2.2 明确不做

- 不做可视化分支时间线。
- 不做玩家回滚历史。
- 不做复杂文明个体叙事。
- 不做星系、恒星、行星的局部对象展开。
- 不做造物主干预或奇迹系统。

## 3. 数据结构

阶段 3 在阶段 2 的 `TimelineEvent` 基础上扩展时间线事件数据。

### 3.1 时间线事件

```ts
type TimelineEvent = {
  id: string
  age: number
  ageLabel: string
  era: EraId
  type: EventType
  title: string
  description: string
  causes: string[]
  effects: EventEffect[]
  importance: number
  location: string
  sourceIds: string[]
  triggeredByEventIds: string[]
  causalNotes: string[]
}
```

要求：

- `id` 必须稳定，排序后仍可复现。
- `era` 必须属于阶段 3 的纪元集合。
- `type` 必须能表达至少 5 类会影响后续生成的事件类型。
- `location` 用于说明事件发生在宇宙尺度、星云、生命海、文明圈、神话层或终局边界等位置。
- `sourceIds` 必须追踪到结构化法则、法则关系、指标或前序事件。
- `triggeredByEventIds` 用于记录直接影响该事件的前序事件。
- `causalNotes` 必须用可读文本解释前序法则、指标或事件为什么会推动当前事件。

### 3.2 事件影响

```ts
type EventEffect = {
  metric: MetricId | "timeline" | "laws"
  delta: number
  description: string
  influence: "metric" | "probability" | "law-pressure"
  affectsFuture: boolean
}
```

要求：

- 至少 5 类事件的 `effects` 中必须存在 `affectsFuture: true`。
- `influence` 为 `probability` 时，表示该事件会调整后续事件类型或纪元权重。
- `influence` 为 `metric` 时，表示该事件会改变后续指标倾向。
- `influence` 为 `law-pressure` 时，表示该事件会放大或缓解法则冲突。

### 3.3 纪元筛选

```ts
type EraId =
  | "creation"
  | "stars"
  | "elements"
  | "life"
  | "civilization"
  | "myth"
  | "ascension"
  | "ending"
```

要求：

- 页面必须提供“全部”和按纪元筛选两类查看方式。
- 筛选只改变展示范围，不改变底层生成结果。
- 分享同一 seed 后，筛选前后的事件数据必须一致。

### 3.4 时间线影响摘要

阶段 3 额外输出 `TimelineImpactSummary`，用于把事件影响沉淀为阶段 4 可消费的局部对象生成上下文。

```ts
type TimelineImpactSummary = {
  eventCount: number
  futureAffectingEventCount: number
  metricDeltas: Record<MetricId, number>
  pressureDeltas: Record<"timeline" | "laws", number>
  localBiases: LocalGenerationBias[]
  eraProfiles: TimelineEraProfile[]
  keySourceEventIds: string[]
  summary: string
}
```

要求：

- `metricDeltas` 记录时间线事件对宇宙指标倾向的累计影响。
- `localBiases` 至少包含星系密度、恒星稳定、行星宜居、生命出现、文明种子、魔法异常、神性遗迹和因果风险。
- 每个 `localBias` 必须包含可追踪的 `sourceEventIds`。
- 阶段 4 只能消费该摘要和结构化法则，不能重新扫描 UI 文案来推断生成倾向。

## 4. 生成链路

阶段 3 的生成顺序固定为：

1. 标准化 seed。
2. 解析模板。
3. 绑定 `rulesetVersion`。
4. 创建根 PRNG。
5. 生成结构化法则与法则关系。
6. 根据法则与关系计算指标。
7. 初始化时间线生成状态，包括事件权重、纪元权重和指标倾向。
8. 生成每个基础纪元的锚点事件。
9. 按当前权重生成补充事件，直到事件数量不少于 30 条。
10. 每生成一个会影响未来的事件，就更新后续事件类型权重或指标倾向。
11. 为关键事件写入前序事件引用和因果解释。
12. 排序并稳定化事件 ID。
13. 汇总时间线影响，生成 `timelineImpact`。
14. 生成解释、观察日志和分享信息。

## 5. 纪元模型

阶段 3 使用以下纪元：

- 创世纪元：规则、边界、初始常数和因果方向建立。
- 星辰纪元：恒星、星云、黑洞和宇宙大尺度结构形成。
- 元素纪元：重元素、灵质、暗物质、法则残响和生命材料积累。
- 生命纪元：生命萌芽、灭绝、复苏、生态路径和意识前体出现。
- 文明纪元：语言、技术、观测、远航、秩序和文明网络形成。
- 神话纪元：信仰、神性、梦境、奇迹和神话事件进入历史。
- 飞升或衰亡纪元：文明飞升、神性吞噬、法则重写或系统性衰败。
- 终局纪元：热寂、轮回、坍缩、裂缝、神战余波或渐进飞升终局。

## 6. 事件类型影响

至少以下 5 类事件必须影响后续生成：

- `stars`：调整生命、元素或稳定相关事件出现概率。
- `elements`：调整生命潜力、文明潜力或魔法强度倾向。
- `life`：调整文明、神话或灭绝复苏相关事件概率。
- `civilization`：调整神话、飞升或终局事件概率。
- `myth`：调整神性、因果、奇迹和终局相关事件概率。
- `anomaly`：调整因果裂缝、法则压力和终局风险。
- `ascension`：调整终局类型和文明飞升倾向。

## 7. 页面要求

阶段 3 的页面在阶段 2 基础上扩展：

1. 时间线区显示 30 条以上事件。
2. 时间线区提供纪元筛选控件，支持“全部”和每个纪元。
3. 筛选后的列表显示当前筛选数量。
4. 事件详情显示时间、纪元、类型、发生位置、原因、后果、影响来源和因果解释。
5. 事件列表中应能看出事件是否影响后续生成。
6. 页面仍保持首屏即应用，不引入营销页。

## 8. 测试与验收

必须提供自动化测试覆盖：

- 每个宇宙至少生成 30 条时间线事件。
- 时间线至少覆盖 8 个纪元中的 6 个，并且必须包含创世和终局。
- 时间线能按纪元筛选，筛选结果只包含目标纪元事件。
- 关键事件之间存在因果解释。
- 至少 5 类事件包含会影响后续生成的效果。
- 至少一部分事件引用前序事件作为触发来源。
- 同一 `seed + templateId + rulesetVersion` 的时间线结果完全一致。
- 阶段 1 与阶段 2 既有测试继续通过。
- `src/sim/**` 中不直接使用 `Math.random()`。

建议命令：

```text
npm test
npm run build
```
