# 阶段 0：概念规格与原型边界

版本：1.0  
日期：2026-06-30  
状态：已锁定，可作为阶段 1 开工依据

## 1. 阶段目标

阶段 0 的目标不是写代码，而是把“宇宙创世沙盒”从灵感整理成可执行的产品与技术边界。

阶段 0 完成后，团队应当能够明确回答：

- 这个项目是什么，核心体验是什么。
- 这个项目不是什么，早期哪些诱人的方向暂时不做。
- 第一版用户打开应用后会看见什么、操作什么、得到什么。
- 为什么它不是严肃物理模拟器，而是一个强调自洽、可复现和叙事张力的宇宙生成沙盒。
- 阶段 1 应该实现哪些数据结构、生成规则和页面区域。

## 2. 一页产品定义

Universe Genesis Sandbox 是一个 seed 驱动的宇宙创世沙盒。

用户输入或随机生成一个 seed，系统基于确定性随机过程生成一个自洽宇宙。这个宇宙可以接近现实物理，也可以包含魔法、神明、灵魂、轮回、因果裂缝、文明飞升等超自然法则。

核心体验：

1. 用户播下一个 seed。
2. 系统生成一组底层宇宙法则。
3. 法则推动宇宙指标、纪元事件、生命潜力、文明趋势和神话风险出现。
4. 用户观察这个宇宙如何从虚无走向命运。
5. 同一个 seed 与同一个规则版本可以被分享并完全复现。

一句话承诺：

> 播下一组法则，观察物理、魔法、生命、文明与神明如何在同一个确定性宇宙中生长、冲突并走向命运。

## 3. 它是什么

- 一个兴趣向、造物主视角的宇宙生成沙盒。
- 一个硬科幻与神话幻想混合的生成引擎。
- 一个重视 seed 可复现性的规则系统。
- 一个用结构化结果支撑叙事风味的观察工具。
- 一个先从宇宙摘要、法则、指标、时间线开始，逐步扩展到星系、文明、神明和干预系统的长期项目。

## 4. 它不是什么

- 不是严肃天体物理或宇宙学模拟器。
- 不是粒子级、恒星级或生命个体级的真实模拟。
- 不是 4X 游戏、文明经营游戏或即时策略游戏。
- 不是自动生成长篇小说的 AI 写作器。
- 不是开放式任意文本魔法系统。
- 不是多人在线宇宙社区。

这个项目可以借用科学概念，但不以现实准确性作为最高目标。最高目标是：规则自洽、结果可复现、用户能感受到“这是一个有命运的宇宙”。

## 5. 产品原则

### 5.1 确定性优先

同一个 `seed + rulesetVersion + templateId` 必须生成完全一致的宇宙结果。

硬性约束：

- 模拟核心禁止直接使用 `Math.random()` 或系统时间作为生成输入。
- 所有随机过程必须来自统一的确定性 PRNG。
- 每个模块必须使用可命名的派生随机流。
- 规则升级必须改变或记录 `rulesetVersion`，不能静默改变旧 seed 的结果。

推荐随机流命名：

```text
root
cosmology
laws.physics
laws.magic
laws.life
laws.consciousness
laws.divinity
laws.causality
metrics
timeline.era.creation
timeline.era.life
timeline.era.civilization
names.universe
explain.summary
```

### 5.2 自洽优先于真实

现实物理只是可选法则集之一。魔法、神明、灵魂、奇迹、轮回、时间回流也可以存在，但必须被规则化。

每个宇宙至少要回答：

- 物质与能量如何存在。
- 生命为何可能或为何艰难。
- 意识、灵魂或信息是否具有独立地位。
- 魔法或超自然力量是否存在，代价是什么。
- 神性来自哪里，是外部造物者、自然现象、文明信仰，还是宇宙结构本身。
- 因果与时间是否稳定，是否存在裂缝、回流、轮回或命运锁定。

### 5.3 先好玩，再加深

阶段 1 的首要目标是让用户在 30 秒内理解并生成一个有风味的宇宙。

早期体验判断标准：

- 用户能立刻看到一个独特宇宙。
- 用户能理解这个宇宙为什么独特。
- 用户愿意再换一个 seed 看看。
- 用户能复制 seed 分享给别人。

### 5.4 分层近似

项目采用分层近似，不追求微观全量模拟。

层级顺序：

1. 宇宙级：法则、时间、物质、魔力、神性、因果。
2. 星系级：结构、资源、灾难、异常。
3. 恒星系级：恒星、行星、宜居性、局部魔法环境。
4. 生命级：生命出现、复杂化、智能、意识或灵魂形态。
5. 文明级：技术、魔法、信仰、扩张、毁灭、飞升。
6. 神话级：神明诞生、奇迹、神战、法则破裂、宇宙终局。

阶段 1 只实现宇宙级与时间线摘要，不展开局部对象模拟。

## 6. 第一版最小可用版本范围

第一版最小可用版本只包含阶段 1 至阶段 3 的核心能力压缩版：seed、宇宙摘要、法则、指标、纪元时间线、分享 seed。

### 6.1 必须有

- Seed 输入。
- 随机 seed 生成。
- 当前 seed 展示。
- 复制 seed 或分享文本。
- 宇宙名称。
- 宇宙模板或类型。
- 宇宙核心摘要。
- 宇宙指标。
- 结构化宇宙法则。
- 8 至 12 条关键纪元事件。
- 每个关键结果的简短解释。
- 同一 seed 刷新后结果完全一致。

### 6.2 可以有但不强求

- 少量代表性星系或行星样本。
- 简单收藏到本地。
- 更丰富的分享文本。
- 时间线事件筛选。

这些内容不得阻塞最小可用版本。

### 6.3 明确不做

- 3D 宇宙可视化。
- 星系地图。
- 行星表面地图。
- 复杂生命演化树。
- 复杂文明模拟。
- 奇迹或造物主干预系统。
- 多人在线。
- 云端账号。
- 插件系统。
- 创作者工具。
- AI 自动长篇小说生成。

## 7. 第一版用户路径

用户首次打开应用后应当进入可用工具界面，而不是营销落地页。

默认流程：

1. 页面自动生成或预填一个 seed。
2. 用户点击“创世”。
3. 页面展示宇宙名称、类型、核心摘要与指标。
4. 页面展示宇宙法则：物理、魔法、生命、意识、神性、因果。
5. 页面展示纪元时间线。
6. 用户可以修改 seed 或随机生成新 seed。
7. 用户可以复制 seed 或分享摘要。

用户不需要理解复杂参数，也不需要先配置大量选项。

## 9. 宇宙法则分类

阶段 1 的法则系统至少包含六类。

### 9.1 物理法则

关注物质、能量、恒星、空间结构和基础稳定性。

示例维度：

- 物理稳定度。
- 重元素丰度。
- 恒星形成效率。
- 空间曲率异常。
- 真空衰变风险。

### 9.2 魔法法则

关注超自然能量是否存在、如何流动、代价是什么。

示例维度：

- 魔法强度。
- 魔力来源。
- 施法代价。
- 魔法污染。
- 与物理法则冲突程度。

### 9.3 生命法则

关注生命出现与复杂化的可能性。

示例维度：

- 宜居窗口宽度。
- 生命自发概率。
- 复杂生命门槛。
- 灭绝压力。
- 非碳基生命可能性。

### 9.4 意识法则

关注意识、灵魂、信息和梦境是否具有独立规则地位。

示例维度：

- 意识独立性。
- 灵魂存在性。
- 集体意识倾向。
- 梦境现实渗透。
- 记忆改变现实的能力。

### 9.5 神性法则

关注神明、信仰、奇迹和超越实体。

示例维度：

- 神性活跃度。
- 神明来源。
- 信仰反馈强度。
- 奇迹可用性。
- 神战风险。

### 9.6 因果与时间法则

关注因果是否稳定、时间是否线性、命运是否可被改变。

示例维度：

- 因果稳定度。
- 时间线弹性。
- 轮回可能性。
- 预言准确度。
- 因果裂缝风险。

## 10. 第一批宇宙模板

阶段 1 先准备 10 个模板。模板只影响权重和风味，不破坏 seed 可复现性。

1. `hard_science`：硬科学宇宙，魔法和神性极低，生命依赖稳定物理窗口。
2. `low_magic`：低魔宇宙，超自然存在但稀有且代价高。
3. `high_magic`：高魔宇宙，魔力参与物质、生命和文明演化。
4. `mythic`：神话宇宙，神性活跃，纪元事件常由神性触发。
5. `polytheistic_war`：多神战争宇宙，神性冲突会影响文明命运。
6. `dream_realm`：梦境宇宙，意识与梦境能改变现实边界。
7. `reincarnation_cycle`：轮回宇宙，时间和灵魂有循环结构。
8. `mechanical_divinity`：机械神宇宙，神性来自计算、秩序或机器意志。
9. `causal_fracture`：因果裂缝宇宙，事件后果不稳定但叙事张力强。
10. `chaotic_laws`：混沌法则宇宙，多类法则高度冲突，稳定性低。

## 11. 核心数据结构草案

阶段 0 不实现代码，但锁定阶段 1 的主要数据轮廓。

```ts
type UniverseSummary = {
  seed: string
  rulesetVersion: string
  templateId: UniverseTemplateId
  name: string
  archetype: string
  tagline: string
  description: string
  metrics: UniverseMetrics
  laws: UniverseLaws
  timeline: TimelineEvent[]
  explanations: Explanation[]
}

type UniverseMetrics = {
  age: LawRating
  stability: LawRating
  lifePotential: LawRating
  civilizationPotential: LawRating
  magicIntensity: LawRating
  divineActivity: LawRating
  causalityIntegrity: LawRating
}

type UniverseLaws = {
  physics: PhysicsLaws
  magic: MagicLaws
  life: LifeLaws
  consciousness: ConsciousnessLaws
  divinity: DivinityLaws
  causality: CausalityLaws
}

type LawRating = {
  value: number
  label: string
  explanation: string
}

type TimelineEvent = {
  id: string
  age: number
  ageLabel: string
  era: EraId
  type: EventType
  title: string
  description: string
  causes: string[]
  effects: EventEffect[]
  importance: number
}

type Explanation = {
  id: string
  targetId: string
  text: string
}
```

阶段 1 实现时可以细化字段，但不得删除以下核心要求：

- `seed` 必须可见。
- `rulesetVersion` 必须进入生成结果。
- `templateId` 必须进入生成结果。
- `laws` 必须是结构化对象，而不是单段描述文本。
- `timeline` 必须是结构化数组，而不是纯文本段落。
- 关键指标和事件必须能解释来源。

## 13. 内容质量约束

生成结果不能只是随机拼词。每个宇宙都必须有因果感。

最低质量要求：

- 名称不能为空。
- 类型不能为空。
- 每类法则都必须有值、标签和解释。
- 指标必须受法则或模板影响。
- 时间线必须覆盖创世、星辰、生命、文明、神话或终局中的至少四类。
- 重要事件必须有原因和后果。
- 同一模板下不同 seed 应有可感知差异。
- 不同模板下同一 seed 应有明显风味差异。

## 14. 版本边界策略

`rulesetVersion` 是 seed 可复现性的边界。只要生成规则、模板权重、事件池、名称池或解释规则发生会影响结果的变化，就必须显式变更规则版本。

阶段 1 采用以下约束：

- `rulesetVersion` 必须是显式字符串，不能隐含在代码版本里。
- 分享结果必须能够恢复 `seed`、`templateId` 和 `rulesetVersion`，但默认分享文案不要求显式展示全部字段。
- 同一 `seed + templateId + rulesetVersion` 的结果必须完全一致。
- 影响生成结果的内容变更必须记录在变更说明中。
- 当前应用只保证当前规则版本内复现；非当前规则短码必须提示不受支持，不提供旧规则运行时兼容。

推荐初始格式：

```text
ugs-ruleset@0.1.0
```

推荐分享策略：

- 面向用户和社区讨论时，优先展示宇宙名称、短 seed、模板风味和一句话摘要。
- 面向系统复现时，通过分享码或链接参数携带完整复现信息。
- 分享码应短、可读、可复制，不让用户手动理解 `rulesetVersion`。

示例分享文案：

```text
银烬庭
Seed: LUX-7F3A-91C2
分享码: UGS01-HM-LUX7F3A91C2
```

示例链接参数：

```text
?s=LUX7F3A91C2&t=HM&v=UGS01
```

其中 `UGS01` 映射到 `ugs-ruleset@0.1.0`，`HM` 映射到 `high_magic`，`LUX7F3A91C2` 是标准化后的 seed。
