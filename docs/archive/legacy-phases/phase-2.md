# 阶段 2：宇宙法则引擎

版本：1.0  
日期：2026-07-07  
状态：已完成

## 1. 阶段目标

阶段 2 的目标是把阶段 1 中的“法则摘要”升级为可计算、可解释、可比较的结构化法则系统。

阶段 2 完成后，系统应能回答：

1. 一个宇宙有哪些具体法则条目。
2. 每条法则属于哪个领域。
3. 法则强度如何影响指标和事件。
4. 法则之间是否兼容或冲突。
5. 两个 seed 的法则差异在哪里。

## 2. 范围

### 2.1 必须实现

- 继续保留阶段 1 的确定性生成能力。
- 每个宇宙至少生成 12 条结构化法则。
- 每类法则至少包含 2 条结构化法则。
- 每条结构化法则包含 ID、名称、领域、数值、标签、影响目标、解释。
- 生成法则兼容性或冲突关系。
- 法则直接参与指标计算，而不是只作为展示文本。
- 模板权重影响结构化法则、指标和解释。
- 提供法则解释器，说明关键指标为何出现。
- 提供两个宇宙之间的法则差异对比。
- 为阶段 2 新增自动化验收测试。

### 2.2 明确不做

- 不允许用户逐条编辑所有法则。
- 不做复杂公式编辑器。
- 不做模组系统。
- 不做阶段 3 的完整时间线系统扩展。
- 不做星系、行星、文明个体模拟。

## 3. 数据结构

阶段 2 在阶段 1 的 `UniverseSummary` 基础上新增法则引擎数据。

### 3.1 结构化法则

```ts
type StructuredLaw = {
  id: string
  domain: LawDomainId
  name: string
  value: number
  label: string
  effectTargets: MetricId[]
  polarity: "support" | "pressure" | "volatile"
  explanation: string
}
```

要求：

- `id` 必须稳定。
- `domain` 必须属于六类法则之一。
- `value` 范围为 0 至 100。
- `effectTargets` 至少包含一个指标。
- `explanation` 必须说明该法则如何影响结果。

### 3.2 法则关系

```ts
type LawInteraction = {
  id: string
  sourceLawId: string
  targetLawId: string
  kind: "synergy" | "conflict" | "constraint"
  impact: number
  explanation: string
}
```

要求：

- 每个宇宙至少生成 3 条法则关系。
- 冲突关系会降低稳定度或因果完整度。
- 协同关系会提高生命、文明、魔法或神性相关指标。

### 3.3 指标影响

```ts
type MetricInfluence = {
  sourceId: string
  sourceLabel: string
  targetMetric: MetricId
  delta: number
  explanation: string
}
```

要求：

- 每个关键指标至少有 1 条影响来源。
- 指标解释不能只引用模板名称，必须引用法则或法则关系。
- 影响来源需要可追踪到结构化法则或法则关系。

### 3.4 法则差异对比

```ts
type LawComparison = {
  leftSeed: string
  rightSeed: string
  templateId: UniverseTemplateId
  domainDiffs: DomainLawDiff[]
  summary: string
}
```

要求：

- 同模板下比较两个 seed。
- 至少输出每个法则领域的强度差异。
- 标出差异最大的领域。
- 比较结果必须确定性复现。

## 4. 生成链路

阶段 2 的生成顺序固定为：

1. 标准化 seed。
2. 解析模板。
3. 绑定 `rulesetVersion`。
4. 创建根 PRNG。
5. 生成每个领域的法则域强度。
6. 生成每个领域至少 2 条结构化法则。
7. 生成法则关系。
8. 根据结构化法则和法则关系计算指标影响。
9. 生成指标。
10. 生成名称、摘要、时间线、解释和分享信息。

## 5. 法则分类

阶段 2 保留六类法则：

- 物理法则。
- 魔法法则。
- 生命法则。
- 意识法则。
- 神性法则。
- 因果与时间法则。

每类至少包含 2 条结构化法则，总数至少 12 条。

## 6. 模板影响

模板必须影响：

- 法则领域强度。
- 结构化法则条目的数值。
- 法则关系倾向。
- 指标影响来源。
- 解释文本中的因果重点。

阶段 2 不要求模板数量增加，但模板差异必须在法则系统中可观测。

## 7. 页面要求

阶段 2 的页面在阶段 1 基础上扩展：

1. 法则区展示每类法则下的结构化法则条目。
2. 法则区展示法则关系。
3. 指标区展示主要影响来源。
4. 增加 seed 对比输入，用于比较当前 seed 与另一个 seed 的法则差异。
5. 对比结果展示各领域差异和最大差异领域。

## 8. 测试与验收

必须提供自动化测试覆盖：

- 每个宇宙至少有 12 条结构化法则。
- 每类法则至少有 2 条结构化法则。
- 每条结构化法则至少影响 1 个指标。
- 每个宇宙至少有 3 条法则关系。
- 每个关键指标至少有 1 条法则或关系影响来源。
- 法则影响指标值，而不是只作为展示文本。
- 不同模板会明显改变法则结果。
- 同一 `seed + templateId + rulesetVersion` 结果完全一致。
- 两个 seed 的法则差异对比结果确定且非空。
- `src/sim/**` 中不直接使用 `Math.random()`。
- 阶段 1 既有测试继续通过。

建议命令：

```text
npm test
npm run build
```

## 9. 阶段 2 完成定义

阶段 2 视为完成，需要同时满足：

- 本文档存在且与实现一致。
- 结构化法则总数和分类数量满足验收标准。
- 指标计算能追踪到法则影响来源。
- 页面展示结构化法则、法则关系和法则差异对比。
- 自动化测试通过。
- 构建通过。
- 不启动后端服务。

## 10. 后续移交给阶段 3

阶段 2 完成后，阶段 3 可以在以下基础上继续：

- 让时间线事件更深入消费结构化法则。
- 让关键事件之间形成更明确的因果链。
- 让事件结果反向影响后续生成概率。
- 增加纪元筛选和事件详情解释。
