# 质量门禁

## 1. 唯一完整检查

```text
npm run check
```

完整检查依次执行代码规范、应用与测试类型检查、AST 架构与单元测试、分层覆盖率、浏览器组件交互、生产构建和构建体积预算。

发布级检查还必须执行真实浏览器端到端测试：

```text
npm run check:release
```

## 2. 自动化门禁

| 门禁 | 要求 |
| --- | --- |
| ESLint | TypeScript 推荐规则、React Hooks 规则、禁止显式 `any` 和未使用变量 |
| 架构边界 | 使用 TypeScript AST 解析真实导入、导出、别名调用和浏览器状态访问，不使用文本正则作为主要依据 |
| 文件规模 | 核心编排文件设置行数上限，超过上限必须先拆分职责 |
| 确定性 | 固定 Seed、模板、规则版本和干预输入必须完全复现 |
| 规则哈希 | 生成规则内容变化必须同步规则版本和测试基线 |
| 覆盖率 | 模拟核心和 UI 分别生成报告，模拟核心语句与行不低于 90%，UI 的 App、组件与 view-model 分别设置独立阈值 |
| 组件交互 | 主导航和奇迹操作必须通过真实 DOM 交互测试 |
| 无障碍 | 核心页面通过键盘聚焦测试和 axe 自动化扫描 |
| 真实浏览器 | Playwright 覆盖 Chromium、Firefox、WebKit、移动视口、分享恢复、错误恢复、剪贴板和真实 axe 扫描 |
| 构建 | TypeScript 与 Vite 生产构建必须成功 |
| 体积 | JavaScript 压缩体积不超过 110 KiB，CSS 不超过 10 KiB |

## 3. 测试分层

- `tests/phase-*.test.ts` 验证阶段领域行为和确定性契约。
- `tests/architecture.test.ts` 验证依赖方向与文件规模。
- `tests/ui/*.test.tsx` 验证浏览器组件交互、键盘与无障碍行为。
- `tests/e2e/*.spec.ts` 验证真实浏览器、移动视口和跨浏览器行为。
- 负向测试必须覆盖无效 Seed、未知模板、非法奇迹、重复干预 ID、错误目标和损坏分享载荷。

## 4. 变更纪律

- 仅修改 UI 或文档时不得改变规则内容哈希。
- 修改 `src/sim/**` 时必须判断是否改变确定性输出。
- 行为变化必须更新 `RULESET_VERSION`、`RULESET_SHORT_CODE`、内容哈希、阶段文档和分享策略说明。
- 不允许通过降低覆盖率、性能或文件规模阈值来绕过失败，应先修复实现或补充合理证据。

## 5. 本地真实浏览器验证

根据仓库进程管理要求，开发者应在可见终端中手动启动预览服务：

```text
npm run build
npm run preview -- --port 4173
```

在另一个终端执行：

```text
npm run test:e2e
```

测试结束后在预览服务终端按 `Ctrl + C` 停止服务。
