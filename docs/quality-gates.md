## 1. 检查命令

日常完整检查只有一个入口：

```text
npm run check
```

它依次执行：

```text
eslint
应用与测试 TypeScript 类型检查
模拟核心单元测试与覆盖率
UI 组件测试与覆盖率
额外 UI 测试
生产构建
构建体积检查
```

发布级检查还必须执行真实浏览器端到端测试：

```text
npm run check:release
```

本地需要手动运行真实浏览器时，在一个可见终端执行：

```text
npm run build
npm run preview -- --port 4173
```

在另一个终端执行：

```text
npm run test:e2e
```

测试结束后必须在预览服务终端按 `Ctrl + C` 停止服务，并删除本次生成且不需要保留的 `coverage/`、`dist/`、`playwright-report/` 与 `test-results/`。

## 3. 步骤 1 强制门禁

步骤 1 只有同时满足下列条件才能进入严格评审：

| 门禁 | 必须证明的事实 |
| --- | --- |
| 可见结果覆盖 | 每一个旧界面可见派生结果都有唯一可查询的因果节点或明确的一对多历史节点，不存在事实孤岛 |
| 合法根 | 无原因节点只能是宇宙公理、独立初始状态或有序显式外部输入，根列表与节点标记一致；模板配置和创世初态必须分别存在 |
| 直接原因与规则 | 每个非根节点至少有一个直接原因和一个有效法则／公理引用 |
| 根可达性 | 每个派生节点都能递归到合法根，不能用循环引用掩盖缺失原因 |
| 合法闭环 | 只允许由指定高阶公理、精确成员、精确内部边、全部一致性约束和外部合法根路径共同授权的闭环；其余循环全部拒绝 |
| 双向一致 | 边端点存在，节点原因／后果索引与边集合完全一致，双向查询返回相同关系 |
| 精确随机证据 | 算法版本、流 ID、完整命名空间、一基抽样序号、类型化操作参数、候选顺序、权重、选中值和用途稳定；受控工厂重新执行完整生成并逐项核对调用转录 |
| 随机结果绑定 | 每个随机引用绑定结果节点、结果主题、节点类型、类型化领域定位器、领域值指纹、原始作用域和字段／集合／负事实语义；生产认证必须解析真实领域结果，同步修改结构副本不得重新获得认证 |
| 输入根真实性 | 因果图保存规范化、有序且类型明确的显式输入清单；输入根与清单一一对应，生成身份同时绑定随机追踪，额外、缺失、重排和值错配全部拒绝 |
| 确定性 | 同一 Seed、模板、规则版本和有序干预输入产生完全一致的旧字段和因果图 |
| 同果异因 | 显示值或状态相同不得合并不同原因节点或不同历史 |
| 展示降级 | 名称、解释和观察文案只能是派生投影，不能成为事实根或反向驱动模拟 |
| 双生成入口 | `generateUniverse` 只承担非枚举惰性兼容边界；当前产品必须使用 `generateCausalUniverse`，在返回前强制物化、校验并冻结因果图 |
| 运行时不可变 | `causalGraph` 非枚举且不可配置，图、节点、边、随机决策、闭环授权和嵌套集合全部运行时冻结 |
| 生产拒绝 | 认证写入能力只存在于私有闭包；基础图必须通过完整领域重放、随机转录、独立结果绑定和闭包断言；投影只能从已认证父图派生为非生产视图 |
| 认证结果不可污染 | 生产图的公共成功校验结果与问题数组不可修改；非认证图每次返回独立问题结果，调用方修改不得影响后续校验或序列化 |
| 按需展示投影 | 普通观察与法则页面渲染不得批量追加因果图；只有点击具体原因入口时追加一个投影节点并聚焦查询 |
| 跨宇宙比较 | 每项 Seed 法则差异与总结先建立组合证据，绑定左右生成身份、左右图来源节点、两侧评分、差值和固定顺序裁决；左右入口分别展示本侧依据且不得合并随机追踪 |
| 用户闭环 | 玩家可以从一级入口选择真实结果，分别查看原因和后果，并沿文字路径继续查询 |
| 旧页面就地追因 | 实际渲染的结果标记必须拥有同主题生产入口或按需状态值投影；事件效果与观察关联不得借用宽实体入口；返回后保留筛选、分页、选中对象、零／非零滚动位置和唯一焦点令牌 |
| 无障碍 | 因果入口、结果列表、方向标签和路径操作支持键盘，组件与真实浏览器 axe 无严重违规 |
| 兼容性 | 旧创世、主要浏览、UGS070/I1 分享和 A1 图书馆仍可运行，且因果图不改变旧领域字段 |
| 旧测试换轨 | 每条旧形状断言已分类并登记退出步骤，步骤 1 不再把固定数量解释为永久要求 |

因果校验器每一个公开问题码都必须有直接负向测试，至少包括：

- 重复节点与重复边。
- 悬空边端点。
- 非法根、根有原因和根列表不一致。
- 无直接原因、无规则引用与未知规则引用。
- 双向邻接不一致。
- 无法到达合法根。
- 未授权循环、缺少成员或边的授权、缺少高阶公理、缺少一致性约束和没有外部根路径的伪授权循环。
- 无效随机流或抽样区间。
- 决策 ID、操作、样本序号、候选集标识或选中值与随机追踪不一致。
- 类型化范围、概率、候选顺序或权重与兼容表示不一致。
- 随机引用与结果绑定表不一致。
- 决策实际所属随机流与引用声明的流不一致。
- 显式输入清单缺失、额外、重排、类型或值错配，以及清单生成身份与随机追踪不一致。

产品定义允许由高阶公理和一致性约束明确许可的因果闭环，因此步骤 1 必须同时保留“有效授权通过”和“无授权／无效授权拒绝”两组测试，不再允许用评审说明代替可执行契约。

步骤 1 当前具有独立性能契约，非覆盖率门禁在两次预热后检查十二个多模板样本的 P50 与 P90 均小于 1500 毫秒，覆盖率环境代表性场景预算为 3000 毫秒；该门禁已进入 `check:release`。

## 4. 步骤 2 强制门禁

步骤 2 进入独立评审前必须同时证明：

- 默认启动与主导航只进入运行中宇宙，旧完整生成器仅在明确标识的隔离兼容视图中按需装配。
- `UniverseDefinition`、`UniverseState`、时钟、随机流、转换、差异、事件、存档和存储适配器拥有独立版本与职责。
- 相同定义、状态、随机流和有序输入产生完全一致的状态哈希、转换、事件、因果节点与随机转录。
- 保存恢复后的下一步与未中断路径完全一致，状态变化后的存档也能稳定 JSON 往返。
- 每个事件只由已提交 `StateDiff` 投影，并能追溯到前置状态、规则、输入和精确随机决定。
- 传统空间对象保持稳定身份，完成形成、变化及可观察后续影响，不预制生命、文明或未来终局。
- 历史游标只读，不改变当前状态哈希、输入日志、随机流或模拟时钟。
- 连续运行、重复命令和存档任务被串行化；单次连续调度达到 100 步预算后自动暂停，不跳步或改变确定性序列。
- 运行存档对无效 JSON、无效形状、未知版本、校验和、身份、状态与历史错误使用稳定错误码直接拒绝；缺失输入、随机流和转换历史均有直接负向测试。
- 运行数据只经版本化 IndexedDB 适配器保存，`localStorage` 不写入状态、历史、因果或随机流。
- 步骤 1 因果查询和旧静态能力在隔离兼容视图中通过回归测试，旧分享查询不冒充等价运行历史。
- 四浏览器真实闭环、键盘、窄屏、axe、生产构建、体积和独立性能测试全部通过。

步骤 2 独立非覆盖率性能门禁在两次预热后测量十二个多模板样本，分别检查单步 P50／P90、一百步 P50／P90、快照 P90 和恢复 P90，并单独验证 1、2、4、8 倍速度在当前时间步提交完成后 100 毫秒内停止下一步调度，预算以 `step-2-freeze.md` 第 6.3 节为准。

## 5. 步骤 3 强制门禁

步骤 3 进入独立评审前必须同时证明：

- 新主流程区分宇宙事实、可观察信号、内部认知边界和玩家推测，页面不得绕过观察系统平铺完整运行事实。
- 相同状态、逻辑时刻、观察方式和显式参数产生完全一致的信号身份、可见值、证据强度和来源引用。
- 同步重算证据身份、信号身份和外层校验和后，伪造结论、不存在来源对象和不存在因果节点仍必须被可信运行历史语义校验拒绝。
- 同一历史中的 `beforeStateId`、其他 `afterStateId` 和当前状态身份不得互换；可信复验必须证明状态身份就是被观察逻辑时刻对应的状态。
- 未观察、证据不足、无结果、确认不存在、支持和冲突状态在契约中可以区分，支持与反对证据不会互相覆盖。
- 玩家可以从多条互不依赖的观察方式开始研究，不以生命、文明、神明、异常或任何固定对象作为前置条件或完成标准。
- 关注、标签、笔记、推测、观察信号和观察历史使用版本化研究存档保存恢复，跨宇宙身份、损坏引用、未知版本和校验和错误明确拒绝。
- 研究写入完整提交或保持先前有效版本；IndexedDB 事务中止和适配器失败不得更新页面成功状态。
- 纯观察、浏览、关注、笔记和推测不改变运行状态身份、逻辑时钟、随机流、输入日志、转换历史或因果网络。
- 新主导航围绕当前宇宙、观察、研究记录和已发生历史组织，旧九页导航、旧概览、旧日志和固定对象入口只存在于隔离兼容视图。
- 真实浏览器覆盖完整普通路径、纯键盘路径、360 像素导航、axe 和错误反馈，步骤 1、2 永久不变量与兼容路径通过回归。
- 观察、研究存档、研究存储和工作台控制器达到各自独立覆盖率阈值，性能预算按 `step-3-freeze.md` 第 6.3 节执行。

步骤 3 独立性能测试在两次预热后使用至少十二个代表性样本，检查单次观察 P50 小于 50 毫秒、P90 小于 100 毫秒，一百条观察历史重建 P50 小于 500 毫秒、P90 小于 1000 毫秒，研究保存 P90 小于 500 毫秒，恢复与完整性校验 P90 小于 1000 毫秒。

步骤 4 增加以下门禁：

- 分支核心验证任意有效时刻分叉、父分支不可变、状态收敛不合并、确定性重放、恢复后下一状态和共同祖先比较。
- 损坏与错配测试覆盖分支版本、宇宙身份、状态哈希、历史哈希、检查点、谱系、输入顺序、重放、存档和两类分享包。
- 架构门禁禁止 `MiraclePanel`、旧单一分享、旧 Seed 对比、A1 和 `UniverseSummary` 进入新分支主流程。
- 分支存储直接验证 IndexedDB 往返、二十条容量、不可用、打开阻塞、读取失败、事务中止和旧值保持。
- 研究记录绑定正式分支历史身份，合法步骤 3 根记录受控迁移，跨分支记录明确拒绝。
- 四浏览器 E2E 覆盖创建、实验、干预、比较、保存、刷新、恢复、继续和研究记录分支隔离。
- 分支创建、百步重放、比较、保存和恢复按 `step-4-freeze.md` 的 P50／P90 预算执行。

## 8. 变更纪律

- 修改 `src/sim/**` 时必须判断是否改变确定性输出、因果图或公共契约。
- 生成规则、随机算法、因果契约或输出字段发生变化时，必须同步评估规则版本、规则短码、内容哈希、因果契约版本和兼容策略。
- 仅修改 UI、测试或文档不得改变模拟核心内容哈希；领域结果与因果证据契约必须分别评估对应版本。
- 删除或替换旧行为前，必须先提供新契约测试、迁移测试或明确拒绝测试。
- 归档文档只用于历史说明，不能作为当前完成证据。
- 测试成功不等于步骤完成；还必须核对产品闭环、迁移矩阵、退役登记、当前架构和非功能要求。

## 9. 严格架构评审

每个新产品步骤完成实现与完整检查后，必须启动一个此前未参与该步骤实现的全新独立 Agent，只依据该步骤冻结的验收基线进行评审。

评审必须：

- 按易用性、可扩展性、可维护性、边界清晰度、阶段功能完整度、文档明确度、测试有效性、数据正确性和运行可靠性分别评分。
- 将问题分为 P0、P1、P2 和建议项。
- 为每项问题提供涉及位置、触发或复现方式、实际影响、判断依据和整改建议。
- P0 和 P1 必须引用冻结验收要求或必须保持的核心不变量；无法给出现实触发路径、具体影响或验证证据的问题只能归类为建议项。
- 评审不得增加冻结基线中不存在的需求，也不得评价后续步骤。
- 把报告保存到 `docs/reviews/step-N-review-XX.md`。

有效 P0 和 P1 必须关闭，P2 必须修复、以证据驳回、降级或登记技术债并给出明确处置。每轮整改后使用另一个全新 Agent 有限复查尚未关闭问题、本轮修改及直接回归，不得复用上一轮评审者或重新开放式评审。建议项不阻塞步骤完成，但不得把实际 P2 改名规避处置。
