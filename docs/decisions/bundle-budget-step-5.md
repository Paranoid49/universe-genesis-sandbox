# 步骤 5 JavaScript 体积预算决策

决策日期：2026-07-15

状态：生效

## 背景

步骤 4 完整构建为 JavaScript gzip 112627 字节，已经超过 110 KiB 日常保护线，但仍低于 115 KiB 发布硬上限。

步骤 5 必须增加版本化宇宙宪法、十一类模块、声明式规则执行与执行记录、三个参考预设、动态观察指标、动态干预能力、跨宪法身份校验以及对应创世界面，同时继续保留步骤 1 至步骤 4 已冻结的因果、运行、观察、分支、存档和隔离兼容能力。

完成当前职责拆分后，新产品主链只依赖 `src/sim/current.ts`，旧模板运行入口已拆到明确命名的兼容适配器，固定观察常量已经退出当前公共入口，生产依赖没有增加。当前完整生产构建为 JavaScript gzip 121124 字节、CSS gzip 6500 字节左右。

继续维持 115 KiB 硬上限需要删除冻结范围内的有效能力、取消旧版隔离兼容入口，或者弱化宪法执行与因果证据，均不符合当前验收基线。

## 决策

- JavaScript gzip 日常保护线调整为 125 KiB。
- JavaScript gzip 发布硬上限调整为 130 KiB。
- CSS gzip 日常保护线继续保持 8 KiB，发布硬上限继续保持 10 KiB。
- 体积检查继续累加全部生产 JavaScript 与 CSS 资产，不通过延迟加载、拆包或忽略兼容资产隐藏总体积。
- 步骤 6 冻结前必须重新评估预算，不能把本次余量自动视为自主实体系统的可用配额。

## 判断依据

- 当前增量来自步骤 5 冻结要求中的执行契约、校验、动态能力和参考规则数据，不是新增框架、分析工具或装饰性依赖。
- 新主流程与旧模板兼容已经通过独立入口隔离，避免旧固定类型继续进入多本体运行依赖图。
- 125 KiB 保护线相对当前构建保留约 6.8 KiB 整改余量，130 KiB 硬上限继续阻止无审计扩张。
- 现有 Preact React-compat 决策继续有效，没有证据表明切换界面运行时能够在不扩大风险的情况下释放所需体积。

## 后续约束

- 新增规则包不得复制执行器、观察器、分支器或存档器实现。
- 参考预设数据不得通过预设 ID 专用代码路径扩大生产体积。
- 第二轮整改后的完整发布门禁实测 JavaScript gzip 为 123319 字节，CSS gzip 为 6492 字节，均通过当前保护线；步骤 5 关闭前仍须以最终有限复查工作树重新确认。
- 步骤 6 如再次超过保护线，必须先拆分职责、删除重复实现或形成新的独立决策，不能直接修改本决策数字。
