# 步骤 1 JavaScript 体积预算决策

决策日期：2026-07-14

状态：生效

## 背景

原 JavaScript gzip 日常保护线为 105 KiB，硬上限为 110 KiB。首轮评审整改后的构建为 107471 字节，只剩 49 字节保护线余量。

第二轮严格复审要求补充集合边界因果节点、输入事实一致性校验、随机决定反向覆盖和候选语义校验。这些属于步骤 1 最高优先级因果真实性契约，不能为了保留旧体积数字而删除或降级。

整改后的构建为 108733 字节，超过原保护线 1213 字节，但仍低于原 110 KiB 硬上限 3907 字节。

## 决策

- JavaScript gzip 日常保护线调整为 110 KiB。
- JavaScript gzip 发布硬上限调整为 115 KiB。
- CSS gzip 日常保护线继续保持 8 KiB，发布硬上限继续保持 10 KiB。
- 体积检查继续统计全部生产 JavaScript 与 CSS 资产的 gzip 总和，不通过拆包隐藏总体积增长。
- 每个后续产品步骤开始前重新评估保护线，不允许自动随实现增长。

## 理由

本次增长来自可执行因果真实性校验与对应错误证据，而不是新增内容池、重复 UI 框架或装饰性依赖。

110 KiB 保护线为当前构建保留约 3.8 KiB 余量，仍能在日常开发阶段阻止无审计扩张；115 KiB 硬上限只用于区分可整改的保护线超限与禁止发布的绝对超限。

## 后续约束

- 步骤 2 开始前必须评估运行时状态、存档适配器和控制界面的增量预算。
- 新增长期依赖、重复校验实现或大型内容池前必须提供体积影响证据。
- 如果构建超过 110 KiB 保护线，必须先优化、拆除旧主流程或形成新的正式决策，不能直接修改检查脚本数字。
